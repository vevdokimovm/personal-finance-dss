# FINPILOT — Нагрузочное тестирование (P0.4)

Сценарий `locust` для проверки пропускной способности и поведения под конкуренцией. Гоняет
реалистичную смесь гостевых запросов по демо-данным (`health`, `dashboard`, быстрая
`recommendation`, тяжёлый `planning/calculate` с Monte Carlo).

## Запуск

```sh
# 1. Поднять приложение (локально или прод-стек)
SECRET_KEY=... DATABASE_URL=... \
  uvicorn app.main:app --host 127.0.0.1 --port 8000

# 2. Преднагрузить демо (данные гостя общие — user_id=None)
curl -X POST "http://127.0.0.1:8000/api/demo/load?case=anna"

# 3. Прогнать locust (headless)
locust -f loadtest/locustfile.py --host http://127.0.0.1:8000 \
  --headless -u 60 -r 20 -t 30s --csv /tmp/finpilot_load
```

Метрики — в `/tmp/finpilot_load_stats.csv` (RPS, медиана, p95/p99 по эндпоинтам) и
`_failures.csv`. Нагрузку наращивать флагом `-u` (число пользователей).

## Baseline в песочнице (индикативно, НЕ прод)

> Среда: **1 воркер uvicorn, слабый общий CPU, SQLite**. Абсолютные числа — ориентир, не
> прод-показатель. На VPS (gunicorn multi-worker + PostgreSQL) пропускная способность будет выше —
> перегнать **этот же сценарий** там для боевых чисел. Это и есть назначение артефакта.

**60 конкурентных пользователей, 30с (чистый прогон, 3333 запроса, RPS ~110):**

| Эндпоинт | RPS | Медиана | p95 | p99 | Ошибки |
|---|---:|---:|---:|---:|---|
| `GET /dashboard` | 33 | 140 мс | 330 мс | 380 мс | 0 |
| `GET /health` | 10 | 88 мс | 270 мс | — | 0 |
| `POST /api/planning/calculate` | 23 | **360 мс** | 690 мс | 860 мс | 0 |
| `POST /api/recommendation` | — | — | — | — | 429 (rate limit, см. находки) |
| **Aggregated** | **~110** | 130 мс | 490 мс | 690 мс | — |

При повышении до ~150 пользователей (короткий пробник) хвостовые задержки растут
(`calculate` p95 → ~1.1 с, `dashboard` p95 → ~0.8 с): сервер насыщается, но **не падает**, ошибок
на compute-путях нет. `/health` зелёный после всех прогонов.

## Находки

1. **Бутылочное горло — `POST /api/planning/calculate`** (Monte Carlo, 1000 итераций): медиана
   360 мс, хвост до ~1.2 с. Самый дорогой путь; первым упрётся в CPU при росте нагрузки. Прямой
   кандидат на кэш (**P1.2**) — расчёт детерминирован (`seed=42`), кэшируется как рекомендации.
2. **[Hardening] `calculate` НЕ покрыт rate-лимитером.** `RateLimitMiddleware` защищает дешёвые
   `recommendation`/auth/export (30 req/60с на `IP+путь`), но **самый дорогой эндпоинт оставлен без
   защиты** от абьюза. Рекомендация: добавить `/api/planning/calculate` в `_protected`
   (`app/middleware.py`) — отдельной задачей (это вне скоупа замера P0.4).
3. **Стабильность под нагрузкой:** на 60–150 пользователях сервер не падал, деградации доступности
   и утечек не видно, отказы строго на rate-limited пути (ожидаемо, лимитер работает).
