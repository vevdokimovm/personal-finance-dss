# Как писать требования (EARS + реестр)

> Компактный гайд: как формулировать требования так, чтобы их можно было реализовать и
> проверить. Развёрнутая методология — в SOP проекта; здесь рабочий минимум. Требования к
> интерфейсу — в `ui_requirements_guide.md` (та же система, своя специфика).

---

## 0. Типы требований и доменные префиксы

Прежде чем писать требование — пойми, какого оно типа. От этого зависит, что в нём проверять.

**Функциональные (FR) — ЧТО система делает** (наблюдаемое поведение). Большинство требований.
Пишутся по EARS (§2). Пример: «When пользователь нажимает «Удалить», система ДОЛЖНА запросить
подтверждение».

**Нефункциональные (NFR) — НАСКОЛЬКО ХОРОШО система это делает.** Главное правило: NFR обязан
быть **измеримым**. «Быстро», «надёжно», «безопасно» — это не требования. Категории:
- **Производительность (PERF):** «p99 latency эндпоинта рекомендации ДОЛЖЕН быть < 200 мс при
  N параллельных запросах».
- **Надёжность (REL):** «При недоступности внешнего источника система ДОЛЖНА деградировать
  graceful (последние известные данные), а не падать».
- **Безопасность (SEC):** «Админ-эндпоинты ДОЛЖНЫ требовать ADMIN_API_KEY».
- **Масштабируемость, совместимость, доступность (a11y — см. UI-гайд).**
- **Юридические (LEGAL):** «Система ДОЛЖНА предоставлять пользователю механизм удаления его
  персональных данных» (152-ФЗ).

**UI/UX** — отдельный класс, по `ui_requirements_guide.md` (состояния, токены, a11y, адаптив).

### Доменные префиксы ID (единая схема)

Каждое требование получает ID `<ПРЕФИКС>-NN`. Префикс = домен:

| Префикс | Домен |
|---|---|
| `AUTH-` | Аутентификация/авторизация |
| `SEC-` | Безопасность |
| `FX-` | Курсы валют / финансовые данные |
| `PERF-` | Производительность |
| `REL-` | Надёжность / отказоустойчивость |
| `DATA-` | Данные, хранение, миграции |
| `LEGAL-` | Юридические (152-ФЗ и пр.) |
| `UX-` | Поведение/опыт (логика взаимодействия) |
| `UI-` | Интерфейс (визуал, состояния — см. UI-гайд) |

Новый домен — добавляй префикс сюда, чтобы схема оставалась единой.

## 1. Нормативный язык (RFC 2119)

| Слово | Смысл |
|---|---|
| **ДОЛЖЕН / MUST** | Обязательно. Невыполнение = дефект, блокирующий приёмку. |
| **СЛЕДУЕТ / SHOULD** | Настоятельная рекомендация. Отклонение — при явном обосновании. |
| **МОЖЕТ / MAY** | Опционально. |
| **НЕ ДОЛЖЕН / MUST NOT** | Прямой запрет. |

Формулировка без этих слов («хорошо бы», «желательно сделать удобно») — это не требование,
а пожелание. Требование проверяемо.

## 2. Шаблон EARS

EARS (Easy Approach to Requirements Syntax) — пять форм. Выбираешь по ситуации:

1. **Ubiquitous** (всегда):
   *Система ДОЛЖНА <реакция>.*
   > Система ДОЛЖНА хранить суммы с точностью до копейки.

2. **Event-driven** (When — на событие):
   *When <триггер>, система ДОЛЖНА <реакция>.*
   > When пользователь нажимает «Удалить», система ДОЛЖНА запросить подтверждение.

3. **State-driven** (While — в состоянии):
   *While <состояние>, система ДОЛЖНА <реакция>.*
   > While пользователь не авторизован, система ДОЛЖНА показывать только демо-данные.

4. **Unwanted behaviour** (If — нежелательное):
   *If <нежелательное условие>, система ДОЛЖНА <реакция>.*
   > If внешний источник курсов недоступен, система ДОЛЖНА сохранить последние известные курсы.

5. **Optional feature** (Where — для опции):
   *Where <функция включена>, система ДОЛЖНА <реакция>.*
   > Where задан ADMIN_API_KEY, система ДОЛЖНА требовать его на админ-эндпоинтах.

## 3. Карточка требования

У каждого требования:

```markdown
**ID:** REQ-XX (доменный префикс: AUTH-, SEC-, UX-, FX-, …)
**Формулировка:** (по шаблону EARS, с MUST/SHALL)
**Приоритет:** MoSCoW — Must / Should / Could / Won't
**Критерий приёмки:** как проверить, что выполнено (конкретно, наблюдаемо)
**Источник:** откуда (стандарт WCAG/ISO/ГОСТ, закон 152-ФЗ, персона, решение)
**Связь:** другие требования / тест-кейсы
```

Пример:

```markdown
**ID:** SEC-07
**Формулировка:** While окружение = production, если ADMIN_API_KEY не задан, система ДОЛЖНА
прервать старт (fail-loud).
**Приоритет:** Must
**Критерий приёмки:** запуск в production без ADMIN_API_KEY завершается ошибкой при старте;
тест validate_production_security возвращает проблему.
**Источник:** решение по защите админ-эндпоинтов.
**Связь:** SEC-01 (JWT_SECRET), тест test_security_hardening.
```

## 4. Реестр

Все требования сводятся в таблицу-реестр для трассировки:

| ID | Формулировка (кратко) | Приоритет | Критерий приёмки | Тест |
|----|----------------------|-----------|------------------|------|
| SEC-07 | fail-loud без ADMIN_API_KEY в prod | Must | старт падает | test_security_hardening |
| FX-03 | сохранять курсы при недоступности ЦБ | Must | source=fallback | test_cbr_fx |

Трассировка (требование → код → тест) — это то, что отличает «мы что-то сделали» от
«мы выполнили требование и можем это доказать».

---

## 5. Признаки плохого требования

- Не проверяемо («сделать удобно», «быстро работать» без метрики).
- Содержит решение вместо потребности («использовать Redis» вместо «кэшировать с TTL»).
- Несколько требований в одном предложении (разбить).
- Нет критерия приёмки (как ревьюер примет работу?).
