# FINPILOT — Инфраструктура проекта

Карта того, как устроен проект: из чего собран, на чём работает, как запускается,
и как соответствующие вещи называются в продакшене. Для быстрого въезда — себе, новому
человеку или AI в новой сессии.

---

## 1. Что это за система

**FINPILOT** — веб-приложение, реализующее **СППР** (систему поддержки принятия решений)
для персональных финансов на российском рынке. Пользователь вносит доходы, расходы,
обязательства и цели; система считает финансовые показатели, перебирает варианты
распределения свободных денег и выдаёт обоснованную рекомендацию.

**Тип архитектуры — монолит.** Один разворачиваемый сервис: и API, и веб-страницы, и
бизнес-логика в одном приложении. (Противоположность — *микросервисы*, где это
раздельные сервисы. Для одиночного проекта монолит — правильный выбор: проще писать,
тестировать и деплоить.)

**Стиль рендера — SSR (server-side rendering).** HTML собирается на сервере шаблонами
Jinja2 и отдаётся готовым, а vanilla JS на странице дорисовывает интерактив и ходит в
API. (Противоположность — SPA, где сервер отдаёт пустую страницу, а весь UI рисует JS-
фреймворк вроде React.)

---

## 2. Архитектура приложения (слои)

Запрос проходит сверху вниз. Разделение на слои — чтобы логика не смешивалась с
вебом и базой (это называется *separation of concerns*).

1. **API / роутер** (FastAPI) — принимает HTTP-запросы, валидирует вход, отдаёт ответ.
   В продакшене это «контроллеры» или «эндпоинты».
2. **Services / бизнес-логика** — расчёт показателей, перебор альтернатив, прогноз,
   рекомендация. Здесь живёт математическая модель. Не знает про HTTP и про базу.
3. **Models / ORM** (SQLAlchemy) — описание таблиц как Python-классов. Слой доступа к
   данным; в продакшене — *data access layer* / *repository*.
4. **База данных** — PostgreSQL (прод) или SQLite (локально).

**ORM (Object-Relational Mapping)** — прослойка, которая превращает строки таблиц в
объекты Python и обратно, чтобы не писать SQL руками. SQLAlchemy — самый
распространённый ORM в Python.

## 3. Технологический стек

### Backend
- **Python 3.12** — язык.
- **FastAPI** — веб-фреймворк (роутинг, валидация, авто-документация API). В продакшене
  работает как *ASGI-приложение* (асинхронный стандарт интерфейса между сервером и
  приложением).
- **Uvicorn** — *ASGI-сервер*, который запускает приложение и слушает порт. Для
  продакшена обычно за ним ставят полноценный сервер/прокси (см. деплой).
- **Pydantic v2** — валидация и схемы данных (что приходит в API и уходит из него).
- **pydantic-settings** — конфиг из переменных окружения и `.env`.

### Данные
- **SQLAlchemy 2.0** — ORM.
- **Alembic** — *миграции схемы*: версионируемые изменения структуры БД (добавить
  колонку, таблицу). Аналог git, но для схемы базы. Каждое изменение — отдельный файл-
  миграция; накат командой `alembic upgrade head`.
- **PostgreSQL** — основная СУБД в продакшене.
- **SQLite** — лёгкая файловая БД для локальной разработки.

### Frontend
- **Jinja2** — шаблонизатор: HTML с подстановками на стороне сервера.
- **Vanilla JS** — чистый JavaScript без фреймворка, плюс HTML/CSS.

### Инфраструктура и качество (подробности — в разделах ниже)
- **pytest** + **Playwright** — тесты (юнит/интеграционные + браузерные E2E).
- **flake8 / pylint / mypy / black** — линтеры и проверка типов.
- **pre-commit** — хуки, прогоняющие проверки перед коммитом.
- **Sentry** — мониторинг ошибок в рантайме.
- **Docker** — контейнеризация для деплоя.

## 4. Локальное окружение и запуск

**Окружение:** macOS (Intel), IDE — **PyCharm**, версионирование — Git + GitHub.
Приложение поднимается на `http://127.0.0.1:8000`.

**Виртуальное окружение (venv)** — изолированная папка с Python и зависимостями именно
этого проекта, чтобы не смешивать с системными и другими проектами. В продакшене ту же
роль играет образ контейнера.

### Запуск сервера (с нуля)

Из папки нужной версии проекта (там, где лежат `run.py` и `requirements.txt`):

```
cd ~/Downloads/<папка-версии>          # напр. finpilot_v4_16_15_intl
python3.12 -m venv .venv               # создать виртуальное окружение
source .venv/bin/activate              # активировать (в начале строки появится (.venv))
pip install --upgrade pip
pip install -r requirements.txt        # поставить зависимости
```

Конфиг — файл `.env` в корне. Если есть `.env.example`, скопируй его:
```
cp .env.example .env
```
Минимальный `.env` для локали (SQLite):
```
DATABASE_URL=sqlite:///./sppr.db
DEBUG=true
```

Запуск:
```
python run.py
```
Открыть в браузере: **http://127.0.0.1:8000**

### Нюансы

- **Миграции.** Если при старте приложение само создаёт таблицы (`init_db`) — больше
  ничего не нужно. Если версия требует Alembic — накатить схему один раз:
  `alembic upgrade head`.
- **Порт 8000 занят** — запусти на другом:
  `python -m uvicorn app.main:app --host 127.0.0.1 --port 8001 --reload`
- **Повторный запуск** — окружение создаётся один раз; потом достаточно
  `source .venv/bin/activate && python run.py`.
- **Сброс локальной базы** — удалить файл `sppr.db` и запустить заново (SQLite пересоздастся).
- `--reload` (он включён в `run.py`) — автоперезапуск при изменении кода. Только для
  разработки, в продакшене выключается.

│   └── static/           CSS, JS, изображения
├── alembic/ (migrations) миграции схемы БД

## 7. Тестирование

**Зачем строго.** Проект большой и связный — без тестов легко пропустить регрессию, в
том числе математическую (поехавшая формула не упадёт с ошибкой, просто будет считать
неверно). Правило: **TDD** — сначала тест, потом реализация, прогон до green.

- **Юнит/интеграционные тесты** — `pytest`. Проверяют функции, API, бизнес-логику.
- **E2E (end-to-end) тесты** — `Playwright`, браузер Chromium. Проверяют реальные
  пользовательские сценарии в браузере (клики, формы), а не только серверный HTML.
- **Coverage gate 90%** — *покрытие тестами* (доля кода, исполняемого тестами) не должна
  падать ниже 90%, иначе сборка считается непройденной.
- **Матрица SQLite + PostgreSQL** — тесты гоняются на обоих движках. Причина: SQLite
  молча прощает нарушения внешних ключей (FK), которые PostgreSQL ловит. Что зелено на
  SQLite, может падать на PostgreSQL — поэтому проверяются оба.
- **WebKit/Safari — критическое слепое пятно.** Исторически половина багов проявлялась
  только в WebKit. Требует явной проверки глазами в Safari.

## 8. CI/CD и качество кода

**CI (Continuous Integration)** — автоматический прогон проверок (тесты, линтеры) на
каждый push/PR, чтобы поломка ловилась сразу. Настроено на GitHub (GitHub Actions).

**Линтеры** — инструменты, проверяющие код без запуска:
- **flake8** — стиль и явные ошибки.
- **pylint** — более глубокий статический анализ.
- **mypy** — проверка типов (type hints на сигнатурах функций).
- **black** — автоформаттер. ⚠️ Массовый `black .` запрещён — он переформатирует всё и
  ломает diff/merge между двумя параллельными копиями проекта.
- **pre-commit** — хуки, прогоняющие часть проверок локально перед каждым коммитом.

Принцип: **линтеры информативны, тесты блокируют.** Линтер может ругаться — это сигнал;
упавший тест — стоп.

**CD (Continuous Delivery/Deployment)** — автоматизация выкатки. На текущем этапе деплой
ручной/полуавтоматический (см. ниже).

## 9. Мониторинг и надёжность

- **Sentry** — сбор ошибок в рантайме: когда у пользователя что-то падает, трейс
  прилетает в Sentry с контекстом. Подключён в *lifespan* приложения (хук старта/остановки).
- **Health-эндпоинты** — служебные URL, по которым внешний мониторинг проверяет, что
  сервис жив (*health check*).
- **Структурное логирование** — логи в машиночитаемом виде (структурированные поля, а не
  свободный текст), чтобы их можно было фильтровать и анализировать.
- **Бэкапы БД** — регулярные резервные копии данных.

## 10. Деплой и контейнеризация

**Docker** — упаковка приложения вместе с окружением в *образ* (image), который
одинаково запускается где угодно. Запущенный образ — *контейнер*. Это убирает проблему
«у меня работает, на сервере нет».

- Сборка и запуск: `docker compose up --build` (без `-v`, чтобы не сносить тома с данными).
- **Reverse proxy** — в продакшене перед приложением ставят прокси (nginx/Traefik):
  принимает внешние запросы, терминирует HTTPS, отдаёт статику, проксирует остальное в
  Uvicorn.
- **GitHub Container Registry (ghcr.io)** — место для хранения собранных Docker-образов
  (часть GitHub Packages). Опционально, понадобится на этапе продакшен-деплоя.

## 11. Сетевые особенности РФ

Инфраструктурные ограничения, которые влияют на код и деплой:

- **Docker Hub заблокирован для РФ.** Зеркала: `dh-mirror.gitverse.ru`,
  `dockerhub.timeweb.cloud`.
- **cbr.ru отдаёт 403 датацентровым IP** (DDoS-защита, не баг). Для курсов/ставки ЦБ в
  продакшене используется зеркало `cbr-xml-daily.ru`; при недоступности — фолбэк, расчёт
  не блокируется.
- **GitHub TLS-таймауты из РФ.** В deploy-скриптах нужна retry-логика; при ручном push —
  просто повторить.

## 12. Соответствие и приватность

- **152-ФЗ** (О персональных данных) — согласие на обработку ПДн как отдельный документ;
  обязательный чекбокс согласия при регистрации.
- **Шифрование ПДн в покое** (*encryption at rest*) — чувствительные данные шифруются в
  базе; плюс аудит доступа.
- **Защита админ-эндпоинтов** — служебные маршруты (аналитика, cron-триггеры) закрыты
  ключом `X-Admin-Key`; в продакшене ключ обязателен (*fail-loud* на старте, если не задан).

## 13. Мини-глоссарий продакшен-терминов

| Термин | Простыми словами |
|---|---|
| Монолит | Один сервис со всем внутри (наш случай) |
| SSR | HTML собирается на сервере |
| ORM | Таблицы как Python-объекты вместо ручного SQL |
| Миграция | Версионируемое изменение схемы БД |
| ASGI | Стандарт связки асинхронного сервера и приложения |
| Reverse proxy | Прокси перед приложением (HTTPS, статика, маршрутизация) |
| CI/CD | Автопрогон проверок и автовыкатка |
| Линтер | Проверка кода без запуска (стиль, ошибки, типы) |
| Coverage | Доля кода, покрытая тестами |
| E2E | Тест полного сценария в реальном браузере |
| Health check | Служебный URL «жив ли сервис» |
| Encryption at rest | Шифрование данных в хранилище |
| Image / контейнер | Упакованное приложение / его запущенный экземпляр |
| Fail-loud | Падать сразу и явно при неверной конфигурации |

---

**Одной строкой:** монолит на FastAPI + SQLAlchemy/Alembic + Jinja2, SQLite локально и
PostgreSQL в проде, тесты на pytest+Playwright с матрицей по двум БД, качество через
линтеры и CI, мониторинг в Sentry, деплой в Docker с поправкой на РФ-блокировки.
