# FINPILOT — Вахтенный журнал (единый источник «где мы сейчас»)

> **Что это и почему так названо.** Вахтенный журнал — судовой документ, который вахтенный ведёт
> **непрерывно во время вахты** и передаёт сменщику, чтобы тот мгновенно принял управление. Здесь
> так же: два аккаунта Claude несут «вахту» по очереди (у одного кончаются токены — продолжает
> другой), а этот журнал — то, что передаётся между ними. Прочитав его, принимающий Claude понимает
> точку, в которой стоим, и продолжает **без повторов и без дрейфа от плана** — как будто Василий
> просто написал «продолжай» в том же чате. (Метафора в тему бренда FINPILOT — штурман у штурвала.)

> ## [!] КРИТИЧНО — журнал ведётся НЕПРЕРЫВНО, параллельно работе
>
> Этот журнал обновляется **НЕ по просьбе Василия и НЕ только в конце сессии**, а **сразу по ходу
> работы**: сделал подшаг — обнови §0/§4; принял решение или нашёл грабли — впиши в §5. Причина
> жёсткая: **токены могут кончиться в середине батча.** Если журнал писать только «в конце», этого
> конца может не наступить — Василий получит устаревшее состояние, и смысл журнала обнуляется.
> **В ЛЮБОЙ момент журнал обязан отражать реальную текущую точку — так, будто сессия может
> оборваться на следующем сообщении.** Само название к этому и обязывает: вахтенный журнал по
> определению пишут в момент события, а не «потом».

> **Как пользоваться (Василий).** Первым сообщением новому Claude: этот файл (или его текст) +
> последний архив кода → пишешь «продолжай». Всё.
>
> **Где живёт.** В репозитории, `docs/WATCHLOG.md`. Едет в архиве вместе с кодом, поэтому версия в
> §0 физически не расходится с `app/config.py`. Имя без версии — перезаписывается каждый батч.

## §1. Протокол вахты (как вести этот журнал)

**Заступая на вахту (начало сессии), принимающий Claude:**
1. Прочитай этот файл целиком + `FINPILOT_Claude_sandbox_runbook.md` (грабли песочницы — НЕ
   диагностируй заново: dash-shell, вывод в файл, `setsid` для долгого, chromium `--no-sandbox`).
2. Распакуй последний архив; сверь версию в `app/config.py` с §0.
3. Подними venv (`python3 -m venv .venv` + `pip install -r requirements.txt -r requirements-dev.txt`),
   при работе с E2E — `playwright install chromium`.
4. Отчитайся Василию коротко (3-4 строки): версия, что закрыто в P0, следующий шаг. Вопрос — только
   если §0 расходится с архивом.
5. НЕ перепроверяй и НЕ переделывай помеченное `[x]` в §2/§3. НЕ уходи с порядка роадмапа без команды.

**Неся вахту — обновляй журнал ПО ХОДУ, не откладывая на конец (см. callout вверху):**
- Поменялась точка работы / закончил подшаг → сразу обнови §0 (resume point) и §4 (нити в работе).
- Принял значимое решение, нашёл грабли, выяснил факт → сразу впиши в §5.
- Закрыл батч (bump версии) → допиши строку в §3 `vX.Y.Z — что — чем проверено`, обнови статусы §2.
- **Отдай журнал Василию ОТДЕЛЬНЫМ файлом** через `present_files` в конце батча — не только внутри
  архива. Василию так удобнее передавать его в другой аккаунт. Всегда два экземпляра: один в
  архиве (`docs/WATCHLOG.md`), один отдельным `.md` в чат.
- Держи журнал **тонким**: не дублируй стабильные доки (§7) — ссылайся. Устаревшее перезаписывай,
  не накапливай.

> Правило вахтенного: запись делается в момент события, а не «потом». Оборвёт на полуслове — сменщик
> поднимает управление по тому, что уже записано в §0/§4.

## §5. Решения и находки этой ветки (контекст, чтобы не переоткрывать)

- **Математическая модель — канон `docs/Математическая_модель_v3_0_0.md`, сверено с кодом:**

  параметризован), MC-интервал 80%.** Если в контексте всплывает спека с **21 альтернативой / шаг 20%
  / flow-based `Lt = Rt/(E+ΣP)` / 95%** — это **УСТАРЕВШАЯ** редакция (v2.0.2), НЕ источник истины,
  не переписывать под неё. Источник истины — код. (Закрыто в v4.16.10.)
- **Фокус — только продакшн.** Бакалаврская ВКР сдана; академический слой (ГОСТы, защита) больше не
  релевантен. Цель — рабочий публичный продукт уровня Т-Банк/Альфа/Сбер к концу лета 2026.
- **Главный рычаг к «рабочему продукту» — деплой на VPS** (руки Василия по `docs/DEPLOY.md`).
  Остаток P0 (живой SMTP/ЦБ, прод-секреты) разблокируется сервером. В песочнице деплой не поднять.
- **Два аккаунта мёржатся** → массовый реформат (`black .`) запрещён, иначе конфликты слияния.
- **Кэш дорогих расчётов — на уровне чистой функции расчёта, а не роута**, если в роуте есть
  сайд-эффекты, которые обязаны выполняться всегда (логирование, аналитика). Ключ строится по
  **эффективным входам функции** (после резолва дефолтов/prefs), иначе кэш «слипает» логически
  разные запросы и отдаёт чужой результат. (P1.2, v4.16.16.)
- **PostgreSQL ПОДНИМАЕТСЯ В ПЕСОЧНИЦЕ — не спихивать PG-проверку на разработчика.**
  Драйвер `psycopg[binary]` в requirements, `apt` доступен (archive.ubuntu.com в allowlist),
  пользователь — root. Процедура (один раз за сессию):
  1. `apt-get update -qq && DEBIAN_FRONTEND=noninteractive apt-get install -y -qq postgresql postgresql-contrib`
  2. `pg_ctlcluster 16 main start` (кластер `main` создаётся пакетом; порт 5432).
  3. Роль/БД: `su postgres -c "psql -c \"CREATE USER finpilot WITH PASSWORD 'finpilot' SUPERUSER;\""`
     + `... CREATE DATABASE finpilot_test OWNER finpilot;`
  4. Прогон: `DATABASE_URL="postgresql+psycopg://finpilot:finpilot@localhost:5432/finpilot_test"
     SECRET_KEY=x ./.venv/bin/python -m pytest ...` (формат psycopg3 — `postgresql+psycopg://`).
     `conftest.py` уважает внешний `DATABASE_URL` и гоняет тот же набор на PG.
  Это и есть обязательная PG-половина матрицы (§6.3) — критична, т.к. SQLite молча игнорирует FK.
  На v4.16.18 вся матрица (558) прошла на PG 16.14 в песочнице.

## §6. Жёсткие правила (чек-лист «не дрейфовать»)

1. **TDD** — тест → красный → реализация → зелёный. Реально прогонять, не теоретизировать.
2. **`app/core/` не трогать** без явного согласования (исключение — `spending_advice.py`, не входит в `U(a)`).
3. **Схемные изменения проверять на PostgreSQL** (SQLite молчит про FK — корень BUG-019/023).
4. **SemVer строго:** фича=minor, фикс=patch. Каждый батч — bump `APP_VERSION` + запись в `CHANGELOG.md`.
5. **Один архив на батч**, inner-папка `finpilot_vX_Y_Z_intl`. Файлы — через `present_files`.
6. **Без массового реформата** (см. §5).
7. **Роадмап (P0-P3) — единственный источник приоритетов, строго по порядку.** Не переключаться и не
   изобретать параллельную нумерацию без команды Василия.
8. **Без эмодзи** нигде. Русская проза + английский код. По-братски, прямо, без канцелярита и лишних извинений.
9. **Не предлагать обход** (новый чат и т.п.) до нахождения корня проблемы. `returncode -1` с пустым
   выводом — почти всегда способ вызова (runbook §2), НЕ смерть песочницы.
10. **Перед «реализовать с нуля» — проверь, не сделано ли уже** (§4).
11. **НЕ объявляй задачу невозможной и НЕ спихивай на Василия, не исчерпав фактическую
    проверку.** «Нельзя в песочнице» — это гипотеза, а не факт, пока не проверено командой.
    Что реально поднимается здесь и НЕ является «долгом разработчика»: **PostgreSQL** (§5 —
    apt + pg_ctlcluster), **браузер для E2E** (`playwright install chromium` работает —
    chromium гоняется в песочнице), **apt-пакеты** (allowlist). Сначала попробуй в песочнице,
    потом делай вывод. Перекладывание работы как «долг на стороне Василия» без попытки —
    запрещённый анти-паттерн (разбор: `FINPILOT_postgres_incident_postmortem.md`, INC-PG-SANDBOX).
    Исключение — то, что реально вне песочницы по инфра-причинам (`cbr.ru` 403 из дата-центра,
    деплой на VPS, живой SMTP): это не «невозможность из лени», а внешние ограничения. Граница
    простая: локальный инструмент → подними и проверь; внешняя сеть/инфра → честно отметь и
    укажи, что разблокируется на стороне Василия.
