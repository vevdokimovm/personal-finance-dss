# FINPILOT — Вахтенный журнал (быстрый снимок «где мы сейчас»)

> **Что это и зачем.** Снимок-контекст для мгновенного входа в работу. Василий ведёт продукт на
> **4 аккаунтах Claude (V, J, M, S)** и в нескольких чатах внутри проекта; токены кончаются —
> вахту принимает другой аккаунт/чат. Этот журнал даёт принимающему быструю сводку: **где стоим,
> что сделано, что дальше, на какие грабли не наступать** — чтобы поднять картину с ОДНОГО архива,
> не читая 20k строк кода. (Метафора в тему бренда — штурман у штурвала.)
>
> **Принцип тонкости.** Журнал НЕ дублирует историю. Полная история версий — в `CHANGELOG.md`
> (каждый патч) и `docs/RELEASES.md` (эпохи). Направление и задачи — в `docs/ROADMAP.md`. Здесь
> только выжимка + ссылки.
>
> ## [!] ЖЁСТКОЕ ПРАВИЛО: §3 хранит РОВНО 10 последних версий, не больше
> §3 — это окно из **10 самых свежих версий, и только**. Добавляя новую версию сверху, СРАЗУ удаляй
> самую старую снизу, чтобы их всегда оставалось 10. Без этого журнал быстро разрастается и теряет
> смысл «быстрого снимка». Удалённое не пропадает — вся история живёт в `CHANGELOG.md` и
> `docs/RELEASES.md`. Это правило касается ИМЕННО этого файла: здесь старое именно УДАЛЯЕТСЯ.

> ## [!] Журнал ведётся НЕПРЕРЫВНО, по ходу работы
> Обновляется СРАЗУ по ходу (сделал подшаг → §0; нашёл грабли/факт → §4), а не «в конце сессии» —
> токены могут кончиться в середине батча. В любой момент журнал обязан отражать реальную текущую
> точку.

## §1. Протокол вахты + аккаунты

**Аккаунты:** Василий работает на 4 аккаунтах Claude — **V, J, M, S** — несущих вахту по очереди
через этот журнал (плюс несколько чатов внутри одного проекта). Текущую вахту описывает **S**.

самодостаточен для любого.

**Заступая на вахту:**
1. Прочитай этот файл + `docs/ROADMAP.md` (направление) + `docs/sandbox_runbook.md` (грабли
   песочницы — НЕ диагностируй заново).

3. Подними venv (`python3 -m venv .venv` + `pip install -r requirements.txt -r requirements-dev.txt`);
   для E2E — `playwright install chromium`.
4. Отчитайся Василию коротко (3-4 строки): версия, текущая веха, следующий шаг.
5. НЕ переделывай закрытое, НЕ уходи с порядка ROADMAP без команды.

**Неся вахту:** обновляй §0/§4 по ходу; разборы оформляй САМ по триггеру (инцидент/расследование/
pitfall — маппинг в §5); в конце батча отдавай журнал ОТДЕЛЬНЫМ файлом через `present_files` (имя

## §4. Заметки и находки (не переоткрывать)

**Текущая фаза планирования (что согласовано с Василием):**
- Карта версий 4-9 подтверждена; правило «вехи ≠ SemVer» — осознанное решение Василия.
- Фронт-стек React+TS зафиксирован с обоснованием — `docs/adr_001_frontend_stack.md`.
- Техдолг (mypy/flake8, utcnow, ExpenseRecord) РАЗБЛОКИРОВАН: работаем с конечной версией
  последовательно, параллельных правок нет → конфликтов мёржа не будет.
- Принцип заглушек вехи 4: MVP/каркас для того, что в роадмапе и точно нужно фронту; спекулятивное
  не выдумываем (YAGNI).

**Мат-модель — канон `docs/math_model_v3_0_0.md`** (источник истины — код): 66 альтернатив, шаг 10%,
stock-based `Lt = B_liq/E` (месяцы), L_min=0 (фильтр выключен), MC-интервал 80%, инварианты Rt≥0 и
ПДН≤0.40. Если всплывёт спека с 21 альтернативой / шаг 20% / flow-based Lt / 95% — это УСТАРЕВШАЯ
v2.0.2, не источник истины.

**Время — единый контракт (после BUG-026, v4.25.3):** всё «сейчас» в коде идёт через
`app/utils/time.py::utcnow()` (naive UTC). Прямой `datetime.now()` (локаль сервера) в `app/` —
запрещён, он давал 3ч-сдвиг на MSK-VPS. Исключение — намеренные aware-вызовы `datetime.now(timezone.utc)`
в `cbr_rate`/`cbr_fx`/`security` (там нужен aware UTC). Колонки моделей — naive UTC (вариант A, не aware).
Костыль `obligation.replace(tzinfo=None)` оставлен намеренно — гасит aware-вход из Pydantic, это НЕ про
«сейчас». Полная tz-корректность (зона юзера → локальный календарь для дни-недели/внутримесячных
паттернов) — отдельный заход в слое паттернов, там же заведётся поле `tz` у User (сейчас его нет).
Вариант B (миграция timestamp-колонок на `timezone=True`) НЕ нужен для корректности приложения —
только для интеропа БД у вехи деплоя; шов готов (убрать `.replace(tzinfo=None)` в helper). Полное
обоснование развилки A/B (почему B — не требование корректности) — `docs/adr_002_datetime_storage.md`.

**JWT-ревокация (v4.26.0, раздел 4.4) — устройство и контракт.** Логика отзыва централизована в
`app/database/revocation.py`: точечный рубеж — блок-лист `is_token_revoked`/`revoke_token`/`purge_expired`;
массовый — предикат `tokens_invalidated_for(user, iat)` + `bump_tokens_valid_since`; конвертер
`epoch_to_naive_utc` нормализует JWT-таймстампы (int/float/aware-datetime) в naive-UTC. `dependencies`
и `routes_auth` таймстампы НЕ трогают — только зовут модуль (`revoke_token` принимает сырой `exp`).
Рубеж `tokens_valid_since` режется до секунды (iat — секундная гранулярность), сравнение строгий `<` —
немедленный re-login после logout-all проходит. Легаси-токены без `jti`/`iat` остаются валидными
(обратная совместимость, покрыто тестом).

**Инвалидация сессий при смене/сбросе пароля (v4.27.0, раздел 4.4) — устройство и дизайн-выбор.**
Подключена к тому же массовому рубежу `tokens_valid_since`. `change_password`: `bump_tokens_valid_since`
(валит другие устройства) + `_revoke_request_token` (точечно гасит текущий токен по `jti` — закрывает
sub-секундное окно округления рубежа: токен той же секунды иначе пережил бы строгий `<`) + чистка
auth-cookie + событие `password_changed`. `reset_password`: только `bump` (юзер не залогинен; текущего
токена/cookie нет; вход новым паролем проходит сразу — округление рубежа не задевает свежий токен).
Изменения ТОЛЬКО в `routes_auth.py`, миграций нет (механизм из 0027 переиспользован). **Дизайн-выбор
Option A — намеренный, НЕ переоткрывать:** для финпродукта (банковская планка) смена креденшелов валит
ВСЕ сессии, включая текущую — сильнее и ожидаемее «оставить текущую» (Google/GitHub). Option B
(сохранить текущую сессию) сознательно отклонён: требует ре-выпуска токена + нового cookie + чтобы
фронт подхватил новый токен, а фронт — отдельная веха (пока Jinja+vanilla JS). Инвалидация привязана к
УСПЕХУ смены (verify + update прошли); неверный текущий пароль → 400, рубеж не сдвинут. Карта линии для
слияния с другими вахтами — `docs/merge_manifest_v4_27_0.md`.

**Консолидация трёх параллельных линий v4.26.0 (правило §5 #15).** JWT-ревокацию делали параллельно
три вахты (M, J, S). Эта версия — их слияние по лучшим практикам, не переписывание: ядро `revocation.py`
(инкапсуляция + предикат + конвертер) и тест-набор — из линии M (самый полный, 17 кейсов, форж токена с
прошлым iat для кросс-сессионного logout-all); DRY-хелпер `_revoke_request_token` в routes — из линии J;
обвязка релиза (CHANGELOG/ROADMAP/WATCHLOG) — из S. PIT-004 (распознать сбой канала) восстановлен по
линии M, PIT-005 (на возобновлении верь архиву) — урок линии J. Модели/`security.py`/миграция 0027 у
всех трёх совпали — мёрджить нечего.

**Известные баги к разбору в вехе 4** (давние заметки Василия): (1) ставка ЦБ всегда ставит дефолт —
реального fetch нет (вероятно cbr.ru 403 → зеркало `cbr-xml-daily.ru`); (2) письма «отправлены», но
на почте пусто (SMTP no-op в dev маскирует). Диагностика до корня, не угадывать.

**Открытые вопросы для вехи 7** (тестирование): график прогноза — у Василия ощущение что баг (всегда
прямая линия от константы-зарплаты); нужен тест-кейс где НЕ прямая. Совет при `Rt<0` — исследовать
(реальный кейс Василия с кредиткой). Шаг 5% вместо 10% — замерить скорость.

**PostgreSQL поднимается в песочнице** (не спихивать на разработчика): `apt install postgresql` БЕЗ
`apt-get update` (nodesource 403; Ubuntu-репы рабочие) → `pg_ctlcluster 16 main start` → роль/БД через
`su postgres -c psql` → `DATABASE_URL=postgresql+psycopg://finpilot:finpilot@localhost:5432/finpilot_test`.
PG-долга по старым нет (миграции 0018-0022 проверены на PG 16); миграции **0023–0027 ждут пакетного
PG-прогона перед мёржем** (проект копит и гонит цепочку пачкой — все они аддитивны: новые таблицы +
nullable-колонки, на SQLite зелены). 0027 (revoked_tokens + users.tokens_valid_since) — максимально
аддитивна, встаёт в ту же очередь.

**Грабли песочницы** (детали — `docs/sandbox_runbook.md`): полный одновременный прогон всех тестов
рубится лимитом инструмента (~30с → returncode -1) → гонять пофайловыми группами; итоговая строка
pytest не флашится в файл → использовать `--junitxml` + парсить XML + exit code; фоновые процессы не
флашат лог надёжно; многострочные bash с `\` ломаются; браузеры firefox/webkit недоступны (Azure CDN),
chromium есть.

## §5. Жёсткие правила (чек-лист «не дрейфовать»)

1. **TDD** — тест → красный → реализация → зелёный, реально прогонять.
2. **`app/core/` не трогать** без согласования (исключение — `spending_advice.py`).
3. **Схемные изменения — на PostgreSQL** (SQLite молчит про FK).
4. **SemVer внутри вехи:** фича=MINOR, фикс=PATCH. Bump `APP_VERSION` + `CHANGELOG.md` каждый батч.
   Вехи (v4-v9) — крупные этапы, не классический SemVer (см. §2).
5. **Без массового реформата** кроме координированных окон (сейчас окно открыто — техдолг закрываем).
6. **ROADMAP — источник приоритетов, строго по порядку.** Не изобретать параллельную нумерацию.
7. **Без эмодзи.** Русская проза + английский код. По-братски, прямо, без канцелярита.
8. **НЕ объявляй невозможным, не проверив.** PostgreSQL/chromium/apt поднимаются здесь. Реально
   внешнее (cbr.ru 403, живой SMTP, деплой) — честно отметить.
9. **Перед «с нуля» — проверь, не сделано ли уже** (бэк часто опережает план).
10. **Вывод тестов — через `--junitxml`** (не `tail` по логу): итоговая строка не флашится надёжно.
    Парсить XML + exit code. (§4, `docs/pitfalls.md`.) **Что гонять и когда (узкий цикл vs полный) —
    `docs/test_run_optimization.md`; механика полного fast — `docs/sandbox_runbook.md` §7.**
11. **НИКОГДА не писать имена в гите / релизах / публичных артефактах** — официально безымянно.
12. **Тестирование мат-модели — СТРОГО вне проекта** (Opus 4.8, без памяти/инструкций → не biased).
13. **E2E добавляются с unit** для изменений фронта/API; CRUD-флоу обязаны иметь E2E.
14. **Разборы — АВТОМАТИЧЕСКИ по триггеру:** дефект в коде → инцидент (`docs/incidents_summary.md`);
    искали, но дефекта нет → расследование (`docs/investigations_summary.md`); повторная ошибка
    воркфлоу → pitfall (`docs/pitfalls.md`); найденный баг → карточка по `docs/bug_report_template.md`.
15. **Слияние разошедшихся линий → консолидация, не переписывание истории.** Полная процедура,
    гейты и антипаттерны — **`docs/merge_and_fork_guide.md`**. Профилактика: синкать аккаунты чаще,
    координировать номера версий (версия закреплена за задачей, не за аккаунтом).
