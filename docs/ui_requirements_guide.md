# Как писать требования к UI/UX

> Дополнение к `requirements_guide.md` для интерфейсной части. Общие правила (нормативный
> язык, реестр, признаки плохого требования) — оттуда; здесь — что специфично для UI: состояния,
> доступность, токены, адаптив, интеракции. Полная действующая спецификация интерфейса —
> `docs/ui_ux_design_standard.md` (домены TOK / IA / ST, нумерация `[TOK-NN]`, `[IA-NN]`,
> `[ST-NN]`). Этот файл — про то, **как формулировать новое UI-требование**, чтобы оно легло
> в ту же систему.

---

## 1. Нормативный язык (как в requirements_guide)

`ДОЛЖЕН` (MUST) — обязательно · `СЛЕДУЕТ` (SHOULD) — настоятельно, с обоснованием отклонений ·
`МОЖЕТ` (MAY) — допустимо. Без «удобно», «красиво», «современно» — это не требования, а вкус.

---

## 2. Что обязательно проверить в UI-требовании

UI-требование неполно, пока не закрыты пять измерений. Это чеклист «думай об этом всегда»:

1. **Состояния (ST).** Для любой области данных — **полный набор**: default, loading, empty,
   error, и где применимо — hover/focus/active/disabled. Самая частая дыра — описан только
   «успешный» вид, а empty/error/loading забыты.
2. **Доступность (a11y).** WCAG 2.2 AA: видимый фокус (focus ring), контраст текста, навигация
   с клавиатуры (tab order), корректные роли/ярлыки для скринридера, уважение
   `prefers-reduced-motion`. Для FINPILOT критично — в аудитории есть персона 55+.
3. **Токены, не сырые значения.** Цвета/типографика/отступы/радиусы/тени/движение — только через
   семантические токены (`[TOK-01..08]`), не хардкод `#hex` и не произвольные кегли/пиксели.
4. **Адаптив.** Поведение на контрольных точках (`[TOK-08]` breakpoints): что происходит на
   узком экране, что сворачивается/переносится. Мобайл — крупнейший незакрытый сегмент аудитории.
5. **Интеракция и аффорданс.** Интерактивное выглядит интерактивным; у каждого действия есть
   видимый отклик; экран не заканчивается тупиком (`[IA-04]` — всегда есть следующий шаг).

## 3. Шаблон карточки UI-требования

```markdown
### [UI-XX] Краткое название (экран / компонент)

**Контекст:** <какой экран/компонент, для какой задачи пользователя>

**Требование (нормативно):**
Система ДОЛЖНА <наблюдаемое поведение интерфейса>.

**Состояния:**
- default: …
- loading: <индикация процесса, защита от «вечного спиннера», ST-02>
- empty: <что это и что добавить — actionable empty state, ST-03>
- error: <что случилось, почему, что делать — человеческим языком, ST-04>
- (hover / focus / active / disabled — если применимо)

**Доступность:** <focus ring, контраст, клавиатура, скринридер, reduced-motion>

**Токены:** <какие семантические токены используются; никаких сырых значений>

**Адаптив:** <поведение на breakpoints — что меняется на узком экране>

**Критерий приёмки:** <как проверить, что выполнено — наблюдаемо и однозначно>
```

---

## 4. Признаки плохого UI-требования

- Описан только успешный вид — **нет empty / error / loading**.
- «Красиво», «удобно», «современно» вместо наблюдаемого поведения.
- Сырые значения (`#3A7AFE`, `14px`, `margin: 12px`) вместо токенов.
- Нет упоминания клавиатуры/фокуса/контраста — a11y «как-нибудь потом».
- Нет поведения на узком экране — «потом адаптируем».
- Критерий приёмки непроверяем («должно выглядеть хорошо»).

---

## 5. Связь с реестром и кодом

- Новое UI-требование получает ID `[UI-XX]` и попадает в общий реестр требований
  (`requirements_guide.md`, §4).
- Если требование меняет дизайн-систему (новый токен, новый паттерн) — это решение уровня
  `docs/ui_ux_design_standard.md`; туда и вносится, а не дублируется по экранам.
- Реализация (`frontend/`) должна соответствовать токенам и состояниям из требования —
  расхождение «макет ↔ код» лечится так же, как «модель ↔ код»: правим код или обновляем
  требование.

---

## 6. Источник истины

Полная спецификация интерфейса с конкретными правилами по доменам — в
`docs/ui_ux_design_standard.md`:
- **TOK** — дизайн-токены (цвет, типографика, 8-pt grid, радиусы/тени, движение, focus ring, breakpoints).
- **IA** — информационная архитектура и навигация (одна задача на экран, ≤2 шага к действию, аффорданс, progressive disclosure).
- **ST** — состояния интерфейса (полный набор, loading/empty/error по правилам).

Этот гайд не заменяет стандарт — он помогает писать новые требования в его системе.
