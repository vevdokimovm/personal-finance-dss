# FINPILOT — Версионирование и релизы на GitHub

Окончательная методичка, выверенная практикой. Цель: заливать версии **одной командой**,
без ручной возни, без дублей, с правильным форматом и описаниями.

- Репозиторий: `vevdokimovm/personal-finance-dss`
- Клон: `~/Downloads/personal-finance-dss`
- Архивы версий: `~/Downloads/finpilot_vX_Y_Z_intl.zip`
- Главный скрипт: **`finpilot_publish.sh`** (отдельный файл)

---

## 0. TL;DR

Накопились новые версии в `~/Downloads` (архивы `finpilot_v*_intl.zip`)? Запусти:
```
zsh ~/Downloads/finpilot_publish.sh
```
Скрипт сам найдёт незалитые версии, нарастит `main` коммитами, поставит теги, запушит и
оформит Releases с правильными описаниями и архивами. Готовое пропустит. Всё.

---

## 1. Понятия

- **Тег** — неизменная метка на коммите (`v4.16.20`). Живёт в git.
- **Релиз (GitHub Release)** — страница поверх тега: имя, описание, прикреплённые файлы.
- **SemVer** — `MAJOR.MINOR.PATCH`: PATCH — фикс, MINOR — фича, MAJOR — несовместимость.
- **CHANGELOG.md** — основной источник описаний релизов.
- **Ассет** — файл при релизе; у нас сборка `finpilot_vX_Y_Z_intl.zip` (+ авто Source code).
- **forward-build** — наращивание `main` коммитами из архивов версий, когда кода ещё нет
  в истории (теги тогда указывают на правильный код, а не на старый коммит).

---

## 2. Главный инструмент: `finpilot_publish.sh`

Единый скрипт «запустил и забыл». Идемпотентный — гоняй сколько угодно.

**Что делает за прогон:**
1. **Разведка** — определяет, где `main` (HEAD), какие версии уже на гите, какие архивы
   лежат в `~/Downloads`. Если рабочее дерево клона грязное (есть незакоммиченное кроме
   `.DS_Store`) — останавливается, не тронув твою работу.
2. **forward-build** — для архивов новее HEAD, которых нет в тегах: распаковка → замена
   дерева → коммит → тег. Затем `push` main + теги.
3. **Releases** — для новых версий и тегов без релиза: создаёт Release в правильном
   формате, прикрепляет архив-ассет, ставит Latest на самую свежую. Уже оформленные
   пропускает.

**Сценарии:**

| Команда | Действие |
|---|---|
| `zsh ~/Downloads/finpilot_publish.sh` | Залить всё незалитое + оформить релизы. Обычный режим. |
| `zsh ~/Downloads/finpilot_publish.sh --verify-all` | Пройти по ВСЕМ тегам и пере-применить формат+ассеты (если надо причесать существующие). |

**Источник описаний (по приоритету):**
1. `CHANGELOG.md` (текущий) — полная секция.
2. `git show vX:CHANGELOG.md` — секция из коммита тега (спасает, если секцию потом
   потеряли из файла — реальный случай с 4.16.12).
3. `docs/WATCHLOG.md` (текущий или из коммита) — краткое описание, последний фолбэк.

**Требования:** `brew install gh && gh auth login` (разово).

---

## 3. Эталонный формат (соблюдать в CHANGELOG)

Скрипт формирует релиз из секции CHANGELOG, поэтому секцию пиши строго так:
```
## [ВЕРСИЯ] — ДАТА — Заголовок (TYPE)

Вводный абзац.

### Добавлено / Изменено / Исправлено
- …
```
Пример: `## [4.16.20] — 2026-06-25 — P2.4 слой 3-B: goal-aware советы (PATCH)`.

**Как это превращается в релиз:**
- **Имя (H1):** `FINPILOT vX.Y.Z — Заголовок` (без даты, без TYPE).
- **Тело:** начинается со строки `## [X.Y.Z] — ДАТА — Заголовок (TYPE)` (H2 с датой) и
  далее вся секция.
- **Ассеты:** `finpilot_vX_Y_Z_intl.zip` + авто `Source code (zip/tar.gz)` — три штуки.
- **Latest:** только у самой свежей версии.

## 4. Разведка (скрипт делает сам; вот те же команды руками)

Если нужно глянуть состояние самому — никогда не угадывай, проверь:
```
gh release list --limit 100                 # что уже залито, где Latest
gh release list --limit 100 | sort -t. -k1 -k2 -k3 -V
git describe --tags --abbrev=0              # на каком теге HEAD
git status --short && git branch --show-current
ls ~/Downloads/finpilot_v*_intl.zip        # какие архивы есть
grep -E '^## \[' CHANGELOG.md              # какие версии описаны
```

> Из песочницы Claude гит обычно недоступен (rate limit без токена, fetch заблокирован).
> Поэтому вывод этих команд даёт пользователь — это нормально и быстрее обходов.

## 5. Грабли и решения (всё из практики)

| Проблема | Симптом | Решение |
|---|---|---|
| Предположение о состоянии гита | План на неверный потолок версий, потеря времени | Разведка (раздел 4) до действий. Скрипт делает её сам. |
| Срезанная дата в описании | В теле нет строки `## [X] — дата` | Тело = вся секция ВКЛЮЧАЯ H2-заголовок. В скрипте учтено. |
| Лишний `(PATCH)` в имени | Имя релиза `… (PATCH)` | Имя без TYPE. Скрипт срезает `(PATCH/MINOR/MAJOR)`. |
| Нет архива-ассета | В Assets только 2 авто Source code | `gh release upload TAG zip --clobber`. Встроено. |
| Потерянная секция в CHANGELOG | «нет секции» для реальной версии (4.16.12) | Скрипт берёт из `git show vX:CHANGELOG.md`, затем из WATCHLOG. |
| Тег без релиза | Тег есть, релиза нет | Цели берём из ТЕГОВ; скрипт создаёт недостающий релиз. |
| TLS-таймаут из РФ | `TLS handshake timeout`, `EOF` на api.github.com | Ретраи (в скрипте) + идемпотентность; просто перезапустить. |
| Кириллица/тире в шелле | `invalid UTF-8`, кривой парс | Парсинг через `python3`, текст в gh через `--notes-file`. |
| zsh-массивы и `${(f)}` хрупки | Цикл склеил весь список версий в одну строку, unzip не нашёл архив | Цикл по версиям только через `while IFS= read -r ver; do … done < файл` — ведёт себя одинаково в bash и zsh. Не массивы, не `${(f)}`. |
| Зашумлённый PATH | `command not found` на find/rm/head | Абсолютные пути, `export PATH`, запуск файлом. |
| Копипаст в терминал | Сломанные многострочные функции | Только `zsh ~/Downloads/script.sh`, не вставлять в терминал. |
| forward-build снёс работу | Пропали незакоммиченные правки | Скрипт стопает, если дерево грязное (кроме `.DS_Store`). |
| Релиз без forward-build | Тег указал на старый коммит | Если код в архиве — forward-build (скрипт делает). |
| Парс `gh release list` по столбцам | Падает на именах с пробелами | `gh release list --json tagName --jq '.[].tagName'`. |
| `gh release view` + heredoc-код | Питон получал пустой stdin (всё «NORELEASE») | JSON от gh писать в файл, питону передавать путь, не через pipe. |

## 6. Типовой цикл выпуска новой версии

1. Изменения + тесты (TDD: тест → red → impl → green).
2. Поднять `APP_VERSION` в `app/config.py`.
3. Дописать секцию в `CHANGELOG.md` (формат — раздел 3). Обновить `docs/WATCHLOG.md`.
4. Собрать архив `finpilot_vX_Y_Z_intl.zip` в `~/Downloads`.
5. Закоммитить: `git add -A && git commit -m "vX.Y.Z: краткое описание"`.
6. `zsh ~/Downloads/finpilot_publish.sh` — тег, push, Release, ассет, Latest — автоматом.

> Шаг 5 не обязателен, если работаешь через архивы: тогда `publish.sh` сам сделает
> forward-build из архива. Но если код уже в `main` — просто закоммить и запусти скрипт.

## 7. Нестандартные случаи

- **Пре-релизы** (`4.0.0a1` и подобные) скрипт пропускает (не матчат `X_Y_Z`). Заливать
  вручную: `git tag vX … && gh release create …`.
- **Дырки ниже HEAD** (нет тега у версии меньше текущей) скрипт не трогает — нужен ручной
  разбор истории (rebase). На практике не возникает при последовательной работе.
- **Снести лишний тег:** `git tag -d vX && git push origin :refs/tags/vX`.
- **Поправить Latest:** `gh release edit vX.Y.Z --latest`.
- **Переоформить описание после правки CHANGELOG:** `--verify-all`.

---

## 8. Траблшутинг

1. `gh: command not found` → `brew install gh && gh auth login`.
2. `gh не авторизован` → `gh auth login`, проверка `gh auth status`.
3. `command not found` на утилитах → зашумлённый PATH; запускай файлом.
4. `TLS handshake timeout` / `EOF` → перезапусти скрипт (идемпотентный).
5. «Рабочее дерево не чистое» → закоммить/убери изменения, потом запусти снова.
6. Релиз без даты/ассета у существующей версии → `zsh finpilot_publish.sh --verify-all`.
7. Версия не подхватилась → проверь имя архива (`finpilot_vX_Y_Z_intl.zip`) и секцию в CHANGELOG.

формат, ассеты, Latest) скрипт делает сам, идемпотентно.
