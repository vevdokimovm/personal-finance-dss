# Реестр частых ошибок (pitfalls) FINPILOT

Повторяющиеся ошибки в рабочем процессе разработки: кривые команды, привычки, грабли воркфлоу.
Это **третий тип** разбора рядом с инцидентами и расследованиями, но самого лёгкого веса.

Чем отличается от соседей:

- **Инцидент** (`docs/incidents_summary.md`) — был дефект в продуктовом коде и/или влияние на
  пользователя/систему. Полный разбор — отдельным файлом (правило §12).
- **Расследование** (`docs/investigations_summary.md`) — дефекта в продуктовом коде НЕ оказалось
  (внешняя причина, артефакт теста, недопонимание); причина была неочевидна, её искали.
- **Pitfall (этот файл)** — повторяющаяся ошибка в воркфлоу самого Claude. Причина известна сразу,
  мистики нет, продукт не затронут. Это **не** отдельный файл-разбор, а ОДНА строка-урок:
  симптом → корень → как правильно → правило.

Граница с рунбуком: `docs/sandbox_runbook.md` описывает **свойства окружения** (как работать в
песочнице). Этот файл — про повторяющиеся **ошибки** (чего НЕ делать). По правилу §15 каждая запись
становится переиспользуемым правильным паттерном, чтобы не выводить его заново каждую сессию.

Заполняется АВТОМАТИЧЕСКИ по правилу §14 — как только ошибка воркфлоу повторилась, строка сюда, не
дожидаясь просьбы.

### PIT-001 — `grep` в цепочке `&&` обрывает выполнение; счётчик/итоги «не выводятся»

- **Симптом:** команда вида `... && grep -c X file && next` — `next` молча не выполняется; счётчик
  тестов или строка `TOTAL` покрытия «не выводится». Повторялось много раз.
- **Корень:** `grep` с НУЛЁМ совпадений возвращает код выхода `1` → оператор `&&` обрывает весь
  остаток цепочки. Усугубление: иногда греп идёт по файлу, который фоновый прогон ещё дописывает
  (гонка) — в тот момент совпадений нет; и кириллический вывод в пайпе мог дать битый UTF-8 и оборвать
  команду.
- **Как правильно:**
  1. не чейнить `grep` через `&&` — использовать `;` либо `grep ... || true`;
  2. дожидаться завершения по маркер-файлу (`EXIT=` в `/tmp/*.done`), а не грепать на ходу;
  3. счётчик/итоги брать из ГОТОВОГО файла: `tail -n3 /tmp/reg.txt` (там строка `N passed`) —
     не хрупким `grep -oE "[0-9]+ passed"`;
  4. вывод с кириллицей читать через `python3` или `tail`, не через `grep` внутри `&&`-цепочки.
- **Правило:** §15 — повторная ошибка кодифицируется один раз; правильная команда переиспользуется,
  а не выводится заново.

### PIT-002 — `DATABASE_URL=sqlite:///:memory:` ломает тесты с видимостью между коннекшенами

- **Симптом:** при ручном `export DATABASE_URL="sqlite:///:memory:"` падает группа scope/lifecycle-тестов
  (изоляция данных, видимость общих строк семьи, роспуск с возвратом строк) — хотя на дефолтной
  конфигурации те же тесты зелёные. Выглядит как регрессия или баг мёржа, провоцирует винить код.
- **Корень:** in-memory SQLite даёт ОТДЕЛЬНУЮ БД на каждое соединение — данные, записанные в одном
  запросе/коннекшене, не видны из другого. Тесты, которые пишут данные одним субъектом и читают другим
  (cross-connection scope), на `:memory:` ломаются по построению. `conftest.py` именно поэтому при
  отсутствии внешнего `DATABASE_URL` поднимает изолированный SQLite-**файл** (`tempfile`), а не `:memory:`.
  Ручной override `:memory:` перебивает этот намеренный выбор → ложные падения.
- **Как правильно:**
  1. не переопределять `DATABASE_URL` на `:memory:` для полного набора / scope-тестов — дать `conftest.py`
     поднять файловый SQLite самому (просто `unset DATABASE_URL` / не задавать);
  2. для PG-половины матрицы задавать `DATABASE_URL=postgresql+psycopg://...` — это валидный override,
     `conftest` его уважает и гоняет тот же набор на PG;
  3. падение именно группы scope/cross-request тестов при «ускоряющем» `:memory:` — сигнал «среда, не баг»:
     сверить дефолтную конфигурацию прежде чем винить код.
- **Правило:** §11 — «ложная тревога»: подозрение на дефект сначала проверяется дифференциально (та же
  среда на заведомо рабочем дереве); среда/конфиг исключаются раньше кода.

### PIT-003 — полный fast одной командой/в фоне тратит время; гонять синхронными группами

- **Симптом:** `pytest -m fast` целиком (~850 тестов) одной командой → `returncode -1`/`124`, XML
  пустой. Уход в `setsid`-фон с опросом маркера — фоновый процесс умирает между опрос-командами,
  `*.done` не появляется, время уходит на бесконечный опрос. Повторялось несколько прогонов подряд.
- **Корень:** (1) полный прогон дольше лимита инструмента (~3-4 мин) → обрыв до записи XML;
  (2) `setsid`-фон в этой песочнице не переживает завершение шелла предыдущего вызова bash_tool —
  процесс прибивается, маркер не пишется.
- **Как правильно:** гонять **синхронно группами по ~14-15 файлов** (`awk 'NR%5==N'`), каждую с
  `timeout 160 ... --junitxml=/tmp/pN.xml`, по 1-2 группы на вызов; итог суммировать из XML. Группа
  упёрлась в timeout → в ней тяжёлый файл (property/forecast/pdf), разбить её пополам. Полная
  процедура с командами — `docs/sandbox_runbook.md` §7.
- **Правило:** для полного fast — синхронные группы из рунбука §7, НЕ одна команда и НЕ `setsid`-фон.
