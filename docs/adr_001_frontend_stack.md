# ADR-001: Фронтенд-стек для вехи 6 — React + TypeScript

Дата: 2026-06-28 · Статус: Accepted · Вахта: J

> Зафиксировано по запросу Василия как репорт «на далёкое будущее»: чтобы решение о
> переходе с Jinja2 + vanilla JS на React + TypeScript и его обоснование не потерялись,
> и чтобы через год-два никто (включая нас самих на любом из 4 аккаунтов) не начал
> переделывать стек «потому что непонятно, почему выбрали так». Оформлено по
> `docs/adr_template.md`.

---

## Контекст

К моменту решения (v4.18.1) бэкенд FINPILOT — зрелый production-grade слой: ~10k строк,
FastAPI, чистый REST API, 776 fast-тестов, покрытие выше 90%, household-оверлей, A/B,
экспорт, spending-advice слои 1-3, импорт мультибанк. Подавляющее большинство возможностей
живёт **только в API** и не выведено на фронт.

Фронтенд при этом застрял: **Jinja2 (server-side rendering) + vanilla JS (без фреймворка) +
один файл CSS-токенов**. Сам по себе SSR — не «слабый стек для MVP» (на нём работают
серьёзные продукты), но:

- vanilla JS без фреймворка тяжело поддерживать для большого интерактивного дашборда;
- накоплен inline-хаос (`style="font-size:..."` рассыпан по шаблонам и `app.js` — задокументировано
  в `06_ТЗ_на_современный_UI.md`);
- ощущение «фронт на уровне мажора 2.x», тогда как бэк уже на 4.x.

**Силы, действующие на решение:**
- Требование Василия: надёжно, фундаментально, профессионально, прод-уровень масштаба
  Т-Банк / Альфа / Сбер.
- Бэк уже отдаёт чистый REST API → фронт может стать отдельным SPA-клиентом, бэк не трогаем.
- Жёсткое требование доступности WCAG AA (на бэке намучились с контрастом и `nested-interactive`).
- Финансовый дашборд = много таблиц, форм, графиков, состояний загрузки.
- Карьерный фактор: Василий — backend-разработчик, ищет первую работу; фронт-навык должен быть
  hireable (актив на рынке труда), фронт для него новый.
- Ограничение песочницы: фронт-тесты должны гоняться здесь (Vitest без браузера; Playwright только
  chromium — firefox/webkit заблокированы Azure CDN, подтверждено §11).

---

## Решение

Веха 6 строится на **React + TypeScript**, бэкенд остаётся как REST API. Полный стек (не одно
слово «React», а набор зрелых инструментов, каждый закрывает свою задачу — по аналогии с
бэком, где не просто «FastAPI», а FastAPI + SQLAlchemy + Alembic + Pydantic + pytest):

| Слой | Инструмент | Зачем |
|---|---|---|
| База | **React + TypeScript** | Индустриальный стандарт; TS даёт типобезопасность (для денег критично, как Decimal на бэке — ошибку типа компилятор ловит до прода). |
| Сборка/dev | **Vite** | Быстрый современный стандарт сборки и dev-сервера. |
| Серверное состояние | **TanStack Query** (ex-React Query) | Запросы к API, кэширование, синхронизация. Прямо под случай «готовый REST API, фронт его потребляет». |
| Таблицы | **TanStack Table** | Операции, история планов — сортировка/фильтры. Индустриальный стандарт для финансовых таблиц. |
| Формы + валидация | **React Hook Form + Zod** | Типобезопасные формы; Zod валидирует данные как Pydantic на бэке. |
| Компоненты + a11y | **Radix UI / shadcn/ui** | Доступность (клавиатура, фокус, ARIA) из коробки — закрывает WCAG AA-требование без изобретания заново. |
| Графики | **Recharts / visx** | Распределение плана, прогноз, спарклайны. |
| Формулы | **KaTeX** | Формулы с нижними индексами в LaTeX-стиле (давний незакрытый пункт). |

Тестирование фронта: **Vitest** (компоненты, без браузера — гоняется в песочнице полностью) +
**Playwright** на chromium (E2E здесь; все браузеры — на Mac/CI).

Реализация — через **Claude Design + дизайн-агенты Claude**, по эстетическому ТЗ
`06_ТЗ_на_современный_UI.md` (семантические токены, типографическая шкала, табличные цифры,
тёмная тема, motion 150-250мс, mobile ≥44px).

## Альтернативы

**Angular.** Энтерпрайз-уровень, банки используют (Тинькофф построил дизайн-систему Taiga UI
на Angular). Но: тяжелее на вход, строже, заточен под большие команды и жёсткую структуру. Для
одиночного разработчика-джуна — лишний вес. Отвергнут в пользу гибкости React.

**Vue / Svelte.** Хорошие, современные, приятный DX (Svelte проще React). Но: экосистема под
финтех меньше, рынок труда уже → для портфолио и hireable-фактора React сильнее. Отвергнуты.

**Next.js** (фреймворк поверх React). Мощный, многие прод-приложения на нём. Но: хочет быть
**и бэкендом тоже** (SSR, API-роуты) — а бэк уже есть на FastAPI → дублирование и лишняя
сложность. Чистый React + Vite под случай «бэк готов, нужен только клиент» — чище. Отвергнут.

**Остаться на Jinja + HTMX/Alpine.js.** Минимальная миграция, остаёшься на SSR с реактивностью.
Дёшево и быстро. Но: потолок интерактивности ниже SPA; не «резюме-стек»; не закрывает запрос
на прод-уровень масштаба большого финтеха. Отвергнут (рассматривался как более лёгкий путь, но
не отвечает критерию «фундаментально»).

## Последствия

**Плюсы:**
- Соответствует уровню крупных российских финтехов; React+TS не устаревает (на январь 2026 —
  самый используемый фронт-стек, TS де-факто стандарт для больших проектов).
- Богатейшая экосистема именно под финансовый дашборд (TanStack Table/Query, Recharts).
- Доступность WCAG AA — почти бесплатно через Radix.
- Бэк не трогаем — фронт отдельный SPA на готовом API.
- Hireable: прямой плюс карьере Василия, навык остаётся как актив.
- Тестируемо в песочнице (Vitest + Playwright chromium).

**Минусы / на что подписались:**
- Разделение фронт/бэк, build pipeline, отдельный SPA — больше движущихся частей, чем у SSR.
- Кривая обучения: фронт для Василия новый (он backend). Митигация: код пишет Claude, Василий
  разбирается по ходу; навык остаётся активом.
- E2E в песочнице только на chromium (firefox/webkit — на Mac/CI). Не новое ограничение —
  то же, что на бэке.

**Важная оговорка (зафиксировать, чтобы не было иллюзий):**
«Выглядит как Т-Банк» определяется **не выбором фреймворка**, а дисциплиной исполнения:
дизайн-система, токены, тестирование, доступность. React+TS даёт правильный фундамент и не
мешает достичь уровня — но сам по себе не гарантирует. Гарантирует то, КАК применим (опорный
документ — `06_ТЗ_на_современный_UI.md`). Фреймворк — фундамент, не готовый результат.

**Влияние:**
- Веха 6 = новый фронт-проект (React+TS+Vite) рядом с существующим бэком в репозитории.
- Появляется фронт-тест-инфраструктура (Vitest + Playwright) дополнительно к бэк-тестам.
- Jinja-шаблоны выводятся из активной разработки; могут остаться как fallback/лендинг, если
  понадобится.
- Все возможности бэка, висящие в API, получают визуализацию на фронте (главная цель вехи 6).
