# FINPILOT — Документация и стандарты проекта

> Точка входа в `docs/`. Здесь — рабочие стандарты, шаблоны и регламенты проекта: как
> писать требования, заводить баги, разбирать инциденты, фиксировать архитектурные решения.
> Документы держим **в репозитории рядом с кодом** сознательно — чтобы теория и практика не
> расходились и чтобы любой участник (человек или ИИ) работал по одним правилам.

---

## Как пользоваться

Нашёл задачу в левой колонке — открыл документ из правой. Если документа на твою ситуацию
нет — это повод его завести (по образцу существующих).

| Что делаешь | Документ |
|---|---|
| Пишешь требование (функциональное / нефункциональное) | `requirements_guide.md` (типы, EARS, RFC 2119, реестр) |
| Пишешь требование к UI/экрану/компоненту | `ui_requirements_guide.md` |
| Нужна полная спецификация UI (токены, состояния, a11y) | `ui_ux_design_standard.md` |
| Заводишь баг | `bug_report_template.md` (Severity/Priority, BUG-XXX) |
| Какой разбор выбрать (инцидент / расследование / баг / pitfall) | `report_types.md` (роутер-градация) |
| Разбираешь инцидент / «почему долго чинили» | `incident_postmortem_guide.md` (шаблон post-mortem) |
| Принимаешь важное техническое решение | `adr_template.md` (ADR — архитектурные решения) |
| Тестируешь / готовишь релиз | `QA.md` (уровни тестирования, чек перед релизом) |
| Деплоишь на прод | `DEPLOY.md` (VPS, Docker, TLS, cron, бэкапы, SMTP, cbr) |
| Контрибьютишь в репозиторий | `CONTRIBUTING.md` (ветки, коммиты, стиль, TDD) |
| Понять, ПОЧЕМУ так устроено (решения, философия) | `engineering_practices.md` |
| Понять, как устроена безопасность (что реально в коде) | `security.md` |
| Нужно определение термина | `GLOSSARY.md` |
| Нужны параметры мат-модели | `math_model_v3_0_0.md` (канон) |
| Понять, КАК модель работает по шагам в коде | `algorithm_stack.md` (9-шаговый конвейер ядра) |

## Карта документов по типам

**Требования и проектирование:**
- `requirements_guide.md` — функциональные требования (формат EARS, нормативный язык, реестр).
- `ui_requirements_guide.md` — требования к интерфейсу (состояния, интеракции, доступность, токены).
- `adr_template.md` — Architecture Decision Records: фиксация архитектурных решений и их причин.

**Качество и инциденты:**
- `bug_report_template.md` — карточка дефекта (две оси оценки, нумерация BUG-XXX).
- `report_types.md` — роутер: какой тип разбора писать (инцидент / расследование / баг / pitfall), критерии.
- `incident_postmortem_guide.md` — практический шаблон/метод инцидента (post-mortem, SEV, RCA).
- `investigation_report_guide.md` — практический шаблон расследования (дефекта в коде нет).
- `pitfalls.md` — реестр частых ошибок воркфлоу (строка-урок, третий лёгкий тип).
- `documentation_methodology.md` — общая теория: blameless, RCA-методы, runbook, ADR/RFC, SLO/error budget.
- `QA.md` — регламент тестирования и предрелизный чек.

**Модель и предметная область:**
- `math_model_v3_0_0.md` — каноническая спецификация математического ядра.
- `reference_profiles.md` — эталонные риск-профили.
- `GLOSSARY.md` — глоссарий терминов.

**Эксплуатация и процесс:**
- `DEPLOY.md` — деплой на VPS (Docker, nginx+TLS, systemd, бэкапы, SMTP, курсы ЦБ).
- `CONTRIBUTING.md` — правила работы с репозиторием (как делать).
- `engineering_practices.md` — инженерная философия и обоснование ключевых решений (почему так): CI/CD, линтеры, покрытие, версионирование.

## Принципы, общие для всех документов

1. **Документ должен порождать действие или знание.** Шаблон ради шаблона — мёртв.
2. **Согласованность с кодом.** Если документ описывает поведение системы — он должен
   соответствовать коду. Разошлись — либо чиним код, либо обновляем документ. Источник
   истины по математике — код и `math_model_v3_0_0.md`.
3. **Blameless при разборе инцидентов.** Ищем дыру в системе/процессе, не виноватого.
4. **Корень, а не симптом.** Любой разбор доходит до корневой причины и закрывает *класс*
   проблем, а не один случай.
5. **Единый язык.** Термины — из `GLOSSARY.md`; нормативные формулировки требований — по
   RFC 2119 (`requirements_guide.md`).
