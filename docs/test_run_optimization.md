# Оптимизация запуска тестов — что гонять и когда

> Зачем этот файл. Симптом, который повторяется из сессии в сессию: «опять гоняет кучу тестов /
> долго ждём / каждый раз одно и то же». Корень почти всегда один — **полный прогон (~900 тестов,
> ~9 минут) запускается там, где хватило бы одного тест-файла (секунды)**. Этот гайд — про **сужение
> объёма прогона под изменение**. Механика «как прогнать ВЕСЬ fast, чтобы он не умер по таймауту» —
> отдельно, в `docs/sandbox_runbook.md` §7; здесь — когда полный вообще нужен, а когда нет.

---

## 1. Главный принцип: масштаб прогона = масштаб изменения

Обратная связь должна быть **быстрой и узкой**. Полный suite — это финальный гейт перед упаковкой, а
не команда после каждой правки. 90% времени в цикле TDD достаточно одного-двух тест-файлов.

| Этап работы | Что гонять | Время |
|---|---|---|
| TDD red → green (пишу фичу) | **только новый/целевой тест-файл** | секунды |
| Тронул модуль `X` | тесты `X` + тесты прямых потребителей `X` | секунды–десятки сек |
| Закончил фичу, перед bump версии | **полный fast один раз** (§7 или фон) | ~9 мин |
| Перед упаковкой архива | полный fast (если не гонял после последней правки) + `flake8`+`mypy` | ~9 мин |
| Меняю фронт/API c E2E-флоу | соответствующие unit + `-m e2e` нужный файл | переменно |

Если правка не трогала код (доки, CHANGELOG, WATCHLOG) — **тесты гонять не нужно вообще**.

## 2. Быстрый узкий прогон (основной режим)

```sh
cd /home/claude/<проект>
# Один файл, машиночитаемый итог, без шума ворнингов:
.venv/bin/python -m pytest tests/test_<feature>.py -q -p no:warnings --junitxml=/tmp/t.xml
echo "exit=$?"
# Итог из XML (итоговая строка pytest в файл-редирект не флашится надёжно — PIT-001):
python3 -c "import xml.etree.ElementTree as ET; s=ET.parse('/tmp/t.xml').getroot(); s=s if s.tag=='testsuite' else s[0]; print('tests=%s fail=%s err=%s'%(s.get('tests'),s.get('failures'),s.get('errors')))"
```

Полезные сужения:
- Конкретный класс/тест: `pytest tests/test_x.py::TestGroup::test_case`.
- По подстроке имени: `pytest tests/test_x.py -k "logout and not legacy"`.
- Несколько связанных файлов разом: `pytest tests/test_auth.py tests/test_jwt_revocation.py`.
- Упал — повторить только упавшие: `pytest --lf` (last-failed).

Флаги, которые экономят время и нервы:
- `-p no:warnings` — гасит шум (в проекте есть `DeprecationWarning` из замороженных миграций — это
  не регресс, см. §4).
- `-x` — стоп на первом падении (быстрее увидеть проблему в TDD).
- `--junitxml` — точный итог независимо от буферизации stdout.
- **Не** добавлять `--cov` в узкий цикл: покрытие заметно замедляет прогон. Покрытие — отдельно и
  редко (перед закрытием раздела/релизом), не в каждом цикле.

## 3. Полный fast — только когда реально нужен

Полный прогон нужен на финале фичи и перед упаковкой — чтобы поймать регресс в смежном коде (особенно
после правок в `dependencies`/`crud`/`core`, которые задевают всё). Как прогнать его, чтобы он не
умер по таймауту инструмента — **синхронными группами**, процедура с командами в
`docs/sandbox_runbook.md` §7. Кратко: 5 групп по ~15 файлов, каждая `timeout 160 … --junitxml`, итог
суммировать из XML.

Про `setsid`-фон: runbook помечает его как **ненадёжный** в этой песочнице (фоновый процесс может
умереть между опрос-командами). Если решаешь им пользоваться — держи опрос частым и будь готов, что
маркер не появится; при первом таком зависании переключайся на группы §7, не теряй время.

E2E в обычный цикл **не попадают** — они за маркером `-m e2e` (исключены из дефолтного прогона через
`pytest.ini addopts`). Гонять их адресно и только когда менялся фронт/флоу.

## 4. Типовые пожиратели времени и как их убрать

- **Полный suite после каждой мелкой правки.** Самый частый. Лечение — §1: узкий файл в цикле, полный
  один раз на финале.
- **Пересоздание venv каждый раз.** `python3 -m venv + pip install` — это ~1-2 минуты. В пределах
  ОДНОЙ сессии venv переживает вызовы — создать один раз в начале и переиспользовать, не пересоздавать
  перед каждым прогоном. (Между сессиями песочница стирается — там пересоздание неизбежно; но не
  внутри сессии.) При упаковке `.venv` удаляется из архива — значит после упаковки для повторного
  прогона его придётся поднять снова: **сначала все прогоны, упаковка — последним шагом**.
- **Гонять полный, чтобы проверить один новый тест.** Новый тест-файл проверяется сам по себе за
  секунды (§2); полный — потом, чтобы убедиться в отсутствии регресса.
- **`--cov` в каждом прогоне.** Покрытие медленное; держать для контрольных точек, не для цикла.
- **`tail` по лог-файлу вместо XML.** Итоговая строка не флашится надёжно → кажется, что «ничего не
  вывело», запускаешь ещё раз. Всегда `--junitxml` + парс XML (PIT-001).
- **Повторный прогон из-за `DeprecationWarning`.** Ворнинги из замороженных миграций (`utcnow()` в
  старых `alembic/versions/*`) — не регресс и не повод перезапускать; гасятся `-p no:warnings`.

## 5. Чек-лист «не трачу время впустую»

1. Правка только в доках? → тесты не гоняю.
2. Пишу/чиню фичу? → гоняю её тест-файл (`-q -p no:warnings --junitxml`), читаю итог из XML.
3. Тронул общий модуль (`dependencies`/`crud`/`core`)? → его тесты + прямые потребители; полный —
   на финале.
4. Фича готова, иду на bump? → полный fast **один раз** (§7), затем `flake8 app/` + `mypy app/`.
5. Всё зелено? → упаковываю архив (удаление `.venv` — последним, после всех прогонов).
6. Число тестов в CHANGELOG/WATCHLOG — из фактического итога, не на память.
