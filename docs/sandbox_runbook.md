# FINPILOT — Runbook: запуск и работа в песочнице

> **Кому и зачем.** Этот файл — для ассистента/разработчика, начинающего сессию по FINPILOT
> в песочнице (agent-окружение). Здесь собраны все грабли окружения, которые **повторяются каждый раз**, и
> готовые команды-обходы. Цель: прочитал → за пару команд поднял рабочее окружение и
> правильно гоняешь тесты, **не тратя время на повторную диагностику** того, что уже известно.
>
> Если коротко: песочница рабочая, но у неё три особенности (shell, вывод, длинные команды)
> и одна по браузеру. Зная их, всё гоняется без проблем — включая E2E.

## 0. TL;DR — bootstrap за 5 шагов

```sh
# 1. Проект уже распакован в рабочую папку. Перейти в неё.
cd /tmp/finpilot/<inner_dir>          # либо туда, куда распакован архив

# 2. venv (если ещё не создан)
python3 -m venv .venv

# 3. Зависимости (в venv --break-system-packages НЕ нужен)
./.venv/bin/pip install -q -r requirements.txt -r requirements-dev.txt

# 4. Браузеры Playwright для E2E (chromium достаточно; webkit — если нужен)
./.venv/bin/playwright install chromium

# 5. Smoke-проверка лёгким тестом (через ФАЙЛ, не пайп — см. §2)
./.venv/bin/python -m pytest tests/test_currency_engine.py -q > /tmp/check.txt 2>&1
echo "rc=$?"; tail -5 /tmp/check.txt
```

Если шаг 5 дал `rc=0` — окружение готово. Дальше гоняй по правилам §2–§4.

## 1. Что за окружение

- Linux-контейнер, Python 3.12, есть Docker-подобные ограничения.
- Память ~4 ГБ, диск и `/dev/shm` (~2 ГБ) в норме — **ресурсы почти никогда не виноваты**.
- Сеть **ограничена белым списком**: доступны pypi/github/npm (для установки пакетов).
  Внешние боевые хосты — **нет**. В частности:
  - `cbr.ru` **недоступен** → курсы валют тестировать **только по fallback**; реальный
    fetch проверяется на VPS/в Docker, не здесь.
- Файлы проекта в `/mnt/project/` — **read-only**. Любые правки делать в `/home/claude/`
  (или в распакованной копии в `/tmp`) и отдавать через `present_files`.

## 2. Три особенности, из-за которых легко затупить

Это главная часть. Симптом всех трёх одинаковый и обманчивый: команда возвращает
**`returncode -1` с пустым выводом**. Это **НЕ** «песочница умерла» и **НЕ** повод
заводить новый чат — это артефакт того, *как* запущена команда.

### 2.1. Shell — это `/bin/sh` (dash), а не bash

- `${PIPESTATUS[0]}`, `[[ ... ]]`, массивы `arr=(...)` → **`Bad substitution` / синтаксис-ошибки**.
- Использовать только POSIX-sh: `$?`, `[ ... ]`, `case`. Код возврата процесса брать через
  `$?` **после redirect** (не после пайпа).

### 2.2. Вывод — в ФАЙЛ, не в пайп

- `pytest ... | tail` в dash с буферизацией **глотает вывод и валит команду** (тот самый `-1`).
- Правильно: `pytest ... > /tmp/out.txt 2>&1`, потом отдельно `cat /tmp/out.txt`.
- Проверено: один и тот же прогон через `| tail` → `-1`; через `> файл` → нормальный вывод и `rc`.
- **Итоги/счётчики читать из ГОТОВОГО файла:** `tail -n3 /tmp/out.txt` (там строка `N passed`).
  НЕ грепать на ходу и НЕ чейнить `grep` через `&&` — `grep` с 0 совпадений возвращает `1` и обрывает
  цепочку (повторная ошибка зафиксирована: `docs/pitfalls.md`, PIT-001).

### 2.3. Длинные команды (> ~минуты) обёртка рубит на foreground

- Короткая команда (≈2 с) в foreground — ОК. Полный прогон (≈150 с), E2E (≈30–140 с) на
  foreground → **`-1`, вывод потерян**.
- Решение: запускать в **фоне через `setsid` ОДНОЙ строкой** и опрашивать файл-результат
  **отдельной** командой:

```sh
# ЗАПУСК (одна строка, без sleep/pkill внутри!):
cd <project_dir> && rm -f /tmp/r.txt /tmp/r.done && \
setsid sh -c './.venv/bin/python -m pytest -p no:cacheprovider -q > /tmp/r.txt 2>&1; echo "EXIT=$?" > /tmp/r.done' \
< /dev/null > /dev/null 2>&1 & echo "started"

# ОПРОС (отдельная команда, можно повторять):
sleep 90; echo "DONE: $(cat /tmp/r.done 2>/dev/null || echo ещё_идёт)"; tail -8 /tmp/r.txt
```

> **Критично:** в строке *запуска* не должно быть `sleep`, `pkill` или многострочных
> конструкций — только `cd && rm && setsid ... & echo`. Любой `sleep`/`pkill` в той же
> строке снова даёт `-1`. Чистку процессов и ожидание выноси в **отдельные** команды.
>
> Код возврата pytest бери из маркера `/tmp/r.done` (`EXIT=0` = все прошли, `EXIT=1` =
> есть падения). Это надёжнее, чем парсить хвост, — финальную строку `N passed` обёртка
> часто обрезает по лимиту, а exit-код не врёт.

## 3. Браузер (E2E / Playwright)

- В песочнице **нет привилегий для browser-sandbox** у chromium. Без флагов он не стартует
  (та же картина `-1`).
- Нужны флаги: `--no-sandbox --disable-dev-shm-usage --disable-gpu`.
- В проекте это уже зашито в `tests/e2e/conftest.py` (фикстура `browser_type_launch_args`,
  только для chromium). Если по какой-то причине фикстуры нет — добавить её.
- Проверка, что браузер вообще поднимается здесь:

```sh
cd <project_dir>
./.venv/bin/python -c "
from playwright.sync_api import sync_playwright
with sync_playwright() as p:
    b=p.chromium.launch(headless=True, args=['--no-sandbox','--disable-dev-shm-usage','--disable-gpu'])
    pg=b.new_page(); pg.set_content('<h1>ok</h1>')
    print('BROWSER OK:', pg.locator('h1').inner_text()); b.close()
" > /tmp/br.txt 2>&1
echo "rc=$?"; cat /tmp/br.txt
```

### Запуск E2E (фон + опрос, флаги уже в conftest)

```sh
# ЗАПУСК:
cd <project_dir> && rm -f /tmp/e2e.txt /tmp/e2e.done && \
setsid sh -c 'SECRET_KEY=test-secret-key-for-e2e ./.venv/bin/python -m pytest -m e2e --browser chromium -p no:cacheprovider > /tmp/e2e.txt 2>&1; echo "EXIT=$?" > /tmp/e2e.done' \
< /dev/null > /dev/null 2>&1 & echo "E2E started"

# ОПРОС:
sleep 110; cat /tmp/e2e.done; tail -12 /tmp/e2e.txt
```

Эталон зелёного прогона: `10 passed, 431 deselected ... in ~30s`, маркер `EXIT=0`.

## 4. Запуск приложения вручную (если нужно)

- `import app.main` и одиночный запуск `uvicorn` работают. Для ручной отладки эндпоинтов
  удобнее **TestClient внутри pytest** (надёжнее, чем поднимать сетевой сервер): завести
  временный тест-файл, прогнать его через рабочий метод (§2.2) и удалить.
- ENV для запуска: минимум `SECRET_KEY=x` и `DATABASE_URL="sqlite:////tmp/app.db"`
  (4 слэша = абсолютный путь). Переменные ставить **перед** командой:
  `SECRET_KEY=x DATABASE_URL=... ./.venv/bin/python ...` (не после `timeout`/`setsid`).

## 5. Чек-лист «перед началом работы»

1. `cd` в распакованный проект, поднять `.venv` + зависимости (§0).
2. Если задача трогает E2E — `playwright install chromium` и проверить браузер (§3).
3. Гонять тесты **только** через redirect-в-файл; длинное — в фон + опрос (§2).
4. Не использовать bash-измы (`PIPESTATUS`, `[[ ]]`) — shell dash (§2.1).
5. `cbr.ru` и прочие боевые хосты недоступны — это нормально, тестировать fallback (§1).
6. Увидел `returncode -1` с пустым выводом — **не паниковать и не звать новый чат**:
   это §2 (пайп / длительность / sleep в строке запуска), а не смерть песочницы.

---

## 6. Чего НЕ делать (грабли, на которые уже наступали)

- Не делать вывод «песочница сломалась, нужен новый чат» по `-1`. Сначала проверить §2.
- Не гонять pytest через `| tail` / `| grep` — только `> файл`, потом читать файл.
- Не пихать `sleep`/`pkill` в строку фонового запуска — выносить отдельно.
- Не использовать `${PIPESTATUS}` и другие bash-измы.
- Не запускать chromium без `--no-sandbox` в этой среде.
- Не ждать реального ответа от `cbr.ru` — его здесь нет.
