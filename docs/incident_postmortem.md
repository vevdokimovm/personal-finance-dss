# Разбор инцидентов: post-mortem и investigation

> Как разбирать сбои и нетривиальные проблемы так, чтобы они не повторялись. Два формата:
> **post-mortem** (был инцидент с влиянием) и **investigation report** (разобрались в сложной
> проблеме, но дефекта в коде не оказалось). Дефект в коде отдельно оформляется карточкой —
> см. `bug_report_template.md`.

---

## 0. Шаблоны и реестры (быстрые ссылки)

| Что | Когда | Файл |
|---|---|---|
| **Post-mortem** — шаблон + метод | был сбой с влиянием / near miss | `docs/incident_postmortem.md` (этот файл, §3) |
| **Investigation report** — методичка + шаблон | разобрались, но дефекта в коде нет | `docs/investigation_report_guide.md` |
| **Карточка дефекта** BUG-XXX | найден конкретный баг кода | `docs/bug_report_template.md` |
| **Реестр инцидентов** (сводки) | реальные дефекты/влияние | `docs/incidents_summary.md` |
| **Реестр расследований** (сводки) | дефекта в коде нет | `docs/investigations_summary.md` |

Полные разборы (post-mortem и investigation) — отдельными файлами вне репозитория (правило §12);
в репо остаются только сводки-строки в двух реестрах выше.

---

## 1. Когда что писать

| Ситуация | Документ |
|---|---|
| Был сбой с влиянием на пользователей (даунтайм, потеря данных, деградация) | **Post-mortem** |
| Near miss («чуть не упало», но обошлось) | **Post-mortem** (дешёвый урок) |
| Долго/сложно разбирались, нашли причину, но бага в коде нет | **Investigation report** |
| Нашли конкретный дефект кода | Карточка **BUG-XXX** (`bug_report_template.md`) |
| Тривиальная опечатка без влияния | Ничего, просто фикс |

Часто документы комбинируются: post-mortem → внутри ссылка на BUG-XXX (если корень — дефект)
и/или на новый runbook (если будет повторяться).

---

## 2. Главный принцип — blameless

Разбор **не ищет виноватого человека — ищет дыру в системе**. Человеческая ошибка — это
*начало* расследования, а не его конец. Формулировка меняется с «X задеплоил плохой код» на
«деплой без обязательного прогона тестов смог попасть в прод — нет гейта». Первое бесполезно,
второе — пункт действий.

Почему: если за инциденты наказывают — их прячут, и теряется именно то, что нужно для
надёжности. Безопасные системы строятся на честных разборах, а не на страхе.

## 3. Шаблон post-mortem

```markdown
# Post-mortem: <название>, <дата> (SEV<n>)
Авторы: …  Статус: Draft/Final

## Summary
<2–3 предложения: что сломалось, как долго, кого задело>

## Impact
- Длительность: …
- Затронуто пользователей/запросов: …  (в цифрах, не «было плохо»)
- Деньги / SLA / какие функции были недоступны: …

## Timeline (<таймзона>)
- HH:MM — начало (часто видно только постфактум)
- HH:MM — обнаружение (как именно: алерт / мониторинг / пользователь)
- HH:MM — эскалация / начало работ
- HH:MM — митигация (вернули работу)
- HH:MM — полное восстановление

## Detection
<как узнали; если от пользователя, а не от мониторинга — это отдельный action item>

## Root cause
<корневая причина через 5 Whys — не симптом>

## Contributing factors
<что усугубило или замедлило; у серьёзного инцидента редко одна причина>

## Resolution
- Митигация: <как быстро вернули работу, напр. откат>
- Фикс: <как устранили корень — часто другое действие в другое время>

## Lessons learned
- Что сработало хорошо: …
- Что сработало плохо: …
- Где повезло: …  (часто самый ценный пункт — скрытый риск)

## Action items
- [ ] <действие, закрывающее КЛАСС проблем> — @owner — до <дата> — <тикет>
```

### SEV-уровни (тяжесть инцидента)

| Уровень | Значит | Реакция |
|---|---|---|
| SEV1 | Сервис лежит / потеря данных / утечка | Всё бросаем, post-mortem обязателен |
| SEV2 | Серьёзная деградация ключевой функции | Срочно, post-mortem обязателен |
| SEV3 | Частичная деградация, есть обходной путь | По плану |
| SEV4/5 | Косметика, единичный край | Обычный backlog |

> Не путать с Severity/Priority **бага** — то про дефект, это про инцидент.

## 4. Шаблон investigation report

Когда разобрались в сложной проблеме, но дефекта в коде нет (поведение оказалось корректным,
причина внешняя или в подходе) — полная методичка и шаблон вынесены в отдельный файл:
**`docs/investigation_report_guide.md`**.

Коротко, структура отчёта: ID `INV-<тема>` + **стоимость** (шаги + оценка в токенах); вопрос/подозрение
(назвать «страшную гипотезу» честно); TL;DR (вывод в начале: дефекта нет, причина — X); гипотезы (A/B/C);
ход расследования (команды/цифры, решающий шаг выделить); корень/вывод (с границей «от нас не зависит»,
если внешнее); **«что это НЕ»** (снятые подозрения — сердце документа); профилактика (+ хвост каналов
по §11, если остался); уроки. Сводка-строка — в `docs/investigations_summary.md`; полный отчёт —
отдельным файлом владельцу (§12, имя `FINPILOT_<тема>_investigation.md`).

---

## 5. Методы поиска корневой причины (RCA)

- **5 Whys** — задаёшь «почему?» к симптому и каждому ответу (3–5 раз), пока не упрёшься в
  системную/процессную причину. Первый «почему» — технический симптом, последний — дыра в процессе.
- **Contributing factors** — серьёзный инцидент почти никогда не имеет одной причины. Перечисляй
  все факторы — закрыть нужно несколько дыр, иначе следующий инцидент придёт через соседнюю.
- **Timeline** — выложенная хронология часто сама показывает связь («деплой в 14:00 → ошибки в 14:02»).

---

## 6. Антипаттерны

- **Blame** («виноват X») — стоп: где гейт, который должен был это поймать?
- **Симптом вместо причины** — лечишь следствие, болезнь возвращается.
- **Нет action items** или они без owner/срока — ничего не изменится.
- **Action item не про класс** — «починили этот случай» вместо «закрыли весь класс».
- **Пишут через неделю по памяти** — timeline врёт; писать по горячим следам, с логами.
- **Преждевременная капитуляция** — «невозможно/недоступно» до перебора всех обходных путей.

## 7. Примеры из этого проекта

Реальные разборы (внешние файлы, могут передаваться вместе с проектом; сводки — в реестрах
`docs/incidents_summary.md` и `docs/investigations_summary.md`):
- **E2E (CSRF/CORS + флаги браузера)** — *post-mortem*: фикс в несколько строк искался долго
  (симптом `-1` приняли за «смерть песочницы»). BUG-024, BUG-025.
- **Fetch ЦБ (403 / зеркало / парсер)** — *investigation report*: дефекта нет (graceful fallback
  корректен), реальные данные добыты через обход; карточка ограничения LIM-001 (округление).
- **firefox/webkit в песочнице** — *investigation report*: дефекта в коде/конфиге нет; бинари не
  качаются, т.к. CDN Playwright вне allowlist (403). Перебраны все каналы доставки (GitHub releases —
  404, npm без бинарей, системный firefox несовместим с juggler, apt только либы) → внешнее
  ограничение. `FINPILOT_browsers_sandbox_investigation.md`.
- **«Баг ядра» = гонка в E2E** — *investigation report*: подозрение с математического ядра снято
  тремя уровнями доказательств; чинился тест, не продукт. `FINPILOT_mathmodel_falsealarm_*`.

Все иллюстрируют один урок, вынесенный в антипаттерны: **не списывать на «невозможно», не
перебрав все каналы и обходы** (правило §11).
