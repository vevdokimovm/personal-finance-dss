# Безопасность FINPILOT — что реализовано и что осталось

> Привязанный к коду срез безопасности: что **реально** в кодовой базе, где это лежит, и что
> ещё предстоит на проде. Это не обзор стандартов вообще, а карта защиты именно FINPILOT —
> чтобы разработчик/ревьюер видел состояние без чтения всего кода.
>
> Контекст обязательности: для продакшен-сервиса с обработкой ПДн граждан РФ обязателен
> **152-ФЗ** (согласие, уведомление РКН, хранение данных в РФ). Тяжёлый ГОСТ Р 57580 — только
> когда проект становится лицензированной финансовой организацией под надзором ЦБ; сейчас не наш
> случай. Практический ориентир — **OWASP Top 10**.

---

## 1. Что уже реализовано (с привязкой к коду)

### Аутентификация и доступ
- **Хеширование паролей** — `services/security.py` (никогда не в открытом виде).
- **JWT в cookie** + **блокировка после неудачных входов** (account lockout) — `routes_auth.py`,
  поля lockout в модели пользователя (миграция 0014).
- **Изоляция данных по пользователю** — `_owner_filter` по `user_id` во всех CRUD
  (`database/crud.py`); гостевой режим (`user_id` nullable).
- **Админ-эндпоинты под ключом** — `require_admin` (X-Admin-Key) на аналитике и cron-триггерах.

### Защита данных
- **Шифрование чувствительных полей в покое** — `EncryptedString` (Fernet) в
  `database/types.py`; шифруются `display_name` и комментарии (миграция 0015).
- **Секреты только через `.env`** — `config.py`, никаких секретов в коде/гите.
- **Изоляция секретов из логов** — токены/пароли не пишутся в логи (PII-аудит через события
  `pii_access`).

### Защита от типовых атак
- **SQL-инъекции** — закрыто ORM (SQLAlchemy, параметризованные запросы), сырых SQL-строк нет.
- **CSRF** — `CSRFMiddleware` (`middleware.py`): на изменяющих запросах сверяет Origin со
  списком доверенных, если Origin присутствует (браузерный случай).
- **Security-заголовки** — `SecurityHeadersMiddleware` (`middleware.py`): CSP, X-Frame-Options,
  X-Content-Type-Options; HSTS включается в production.
- **XSS** — экранирование пользовательского вывода во фронте (`esc()` в `app.js`).
- **Валидация входа** — Pydantic-схемы на всех эндпоинтах (строгие типы).
- **Rate limiting** — `RateLimitMiddleware` (`main.py`): лимит на чувствительные и дорогие
  эндпоинты (вкл. `auth/*`, `export/*`, `planning/calculate`, `planning/forecast`), по
  IP+путь, окно настраивается (`RATE_LIMIT_REQUESTS` / `RATE_LIMIT_WINDOW_SECONDS`).

### Fail-loud и эксплуатация
- **Fail-loud старт в production** — `validate_production_security` (`config.py`): старт падает
  при дефолтном `JWT_SECRET`, незащищённой cookie или отсутствии `ADMIN_API_KEY`.
- **152-ФЗ — удаление данных** — `delete_user` чистит все связанные данные одной транзакцией.
- **Наблюдаемость** — `/health`, JSON-логи с `request_id`, Sentry-hook (no-op без DSN).
- **Бэкапы + проверка восстановления** — `scripts/backup_db.sh`, `restore_db.sh`,
  `backup_verify.sh` (бэкап, который не восстанавливали, — не бэкап).

## 2. Что осталось (по приоритету)

🔴 критично к запуску · 🟡 важно · 🟢 по мере роста

- 🔴 **TLS на проде** — артефакты готовы (`nginx/` + Let's Encrypt в `docs/DEPLOY.md`),
  применяется на VPS.
- 🔴 **Реальные прод-секреты** — сгенерировать длинные случайные `JWT_SECRET`,
  `ADMIN_API_KEY`, ключ шифрования, разные по окружениям (`.env.prod.example`).
- 🔴 **Хранение данных граждан РФ в РФ** (152-ФЗ) — выбор хостинга на VPS.
- 🟡 **Уведомление Роскомнадзора** об обработке ПДн — организационный шаг.
- 🟡 **MFA** (второй фактор входа) — пока не реализован.
- 🟡 **`pip-audit` в рутину** — регулярный аудит уязвимых зависимостей.
- 🟢 **Мониторинг аномалий / алертинг** — активировать Sentry-DSN (P1.6 роадмапа).
- 🟢 **Ротация секретов** — процедура смены `JWT_SECRET` (смена разлогинит всех — осознанный шаг).

## 3. Домены проверки (по чему смотреть при ревью)

Любой аудит (по ГОСТ или по здравому смыслу) смотрит одни и те же домены — здесь они с нашим статусом:

| Домен | Статус в FINPILOT |
|---|---|
| Управление доступом | ✅ JWT + lockout + `_owner_filter` + admin-key |
| Защита данных (шифрование) | ✅ Fernet в покое; 🔴 TLS в транзите — на проде |
| Аутентификация | ✅ хеш паролей; 🟡 MFA — нет |
| Журналирование/мониторинг | ✅ события + PII-аудит; 🟢 алертинг — Sentry на проде |
| Защита от типовых атак | ✅ ORM/CSRF/заголовки/XSS/валидация/rate limit |
| Управление уязвимостями | 🟡 `pip-audit` в рутину |
| Реагирование на инциденты | см. `incident_postmortem.md` |
