# Расхождения «код vs ВКР/модель» — talking points

Осознанные отличия между `Математической_моделью_v2_0_2.md` (ВКР) и реализацией (v2.1.0),
с обоснованием каждого. Документ для защиты: если комиссия откроет ВКР и увидит другие числа —
это объяснимо, а не ошибка.

---

## 1. Доверительный интервал Monte-Carlo: 80% вместо 95%

- **Модель (§12.2):** 95% интервал `[Q2.5 … Q97.5]`.
- **Код (`forecast.monte_carlo_intervals`, `forecasting.py`):** **80% интервал `[p10 … p90]`**.
- **Обоснование:** 80% даёт более «читаемый» и менее пугающий пользователя диапазон — края 95%
  на коротких финансовых рядах слишком широки и подрывают доверие к прогнозу. Для бытового
  планирования 80% информативнее. Метод (Monte-Carlo вокруг точечного SES) идентичен — отличается
  только ширина перцентилей.

---

## 2. Порог ликвидности в фильтре допустимости: 0 вместо 0.30

- **Модель (§6):** ограничение допустимости `Lt(a) ≥ L_min = 0.30`.
- **Код (`filtering.py`, `LT_CRIT = 0.0`):** фильтр допустимости отсекает альтернативы только по
  `Rt ≥ 0` и `Dt ≤ 0.40`; порог `Lt` по умолчанию мягкий (0).
- **Обоснование:** `L_min = 0.30` сохранён как **норма для диагностики и дашборда** (подсветка
  показателя), но как жёсткий фильтр допустимости он отсекал бы у многих реальных пользователей
  *все* альтернативы (Lt < 0.30 при положительном Rt — частая ситуация), оставляя пользователя без
  рекомендации. Порог вынесен в параметр (`l_min` в запросе) — пользователь может ужесточить его сам.

---

## 3. Точечный прогноз: SES с накоплением баланса

- **Модель (§12.1):** простое экспоненциальное сглаживание ряда.
- **Код (`forecasting.py`):** SES прогнозирует доходы/расходы/платежи, а баланс `Bt` **накапливается**
  по формуле 35 ВКР (`bt_running += cf − p`), и `Rt` считается на накопленном балансе.
- **Обоснование:** это не противоречие, а уточнение — прогноз свободных денег учитывает накопление
  остатка по месяцам, что точнее «плоского» SES по самому Rt.
