# Соответствие «код ↔ каноническая модель v3.0.0»

Канонический документ модели — `docs/Математическая_модель_v3_0_0.md`. Этот файл фиксирует, что
**код соответствует** канонической модели, поясняет параметры со «слабым» дефолтом (поведение
by design) и историческую эволюцию (чтобы не путаться со старыми редакциями модели и заметками).

> Истина о модели — в `docs/Математическая_модель_v3_0_0.md` и в коде (`app/core/`). Если где-то
> встречается другая формулировка — **21 альтернатива / шаг 20%**, **flow-based** `Lt = Rt/(E+ΣP)`,
> интервал **95%** — это **до-v3.0.0** редакция, она устарела (см. §3).

## 2. Параметр со «слабым» дефолтом (by design, не расхождение)

- **Порог `L_min` в фильтре допустимости = 0** (`filtering.py`, `L_MIN = 0.0`). Жёсткий порог
  отсекал бы у многих реальных пользователей *все* альтернативы (Lt в месяцах часто мал при
  положительном Rt), оставляя их без рекомендации. Поэтому фильтр по `Lt` по умолчанию выключен, но
  **выведен в параметр** (`l_min` в запросе) — пользователь может ужесточить сам. Норма `[2.5; 6]`
  мес используется для **диагностики и подсветки** на дашборде, а не как жёсткий отсев.

## 3. Историческая эволюция модели (для контекста)

Прежняя редакция модели (**v2.0.2**, в т.ч. в бакалаврской работе) описывала:
- дискретизацию шагом **0.20 → 21** альтернатива (`C(7,2)`);
- ликвидность **flow-based** `Lt = Rt/(ΣE + ΣP)` (безразмерная доля, норма ≥ 0.30);
- доверительный интервал MC **95%**.

**v3.0.0 пересмотрел** эти три места, и код приведён к v3.0.0:
- **шаг 10% (66 альтернатив)** — тоньше рекомендации почти без потери скорости (расчёт в единицы мс);
- **stock-based `Lt`** (месяцы) — прежняя flow-based формула коррелировала с ресурсом на ≈ 0.9998
  (два критерия свёртки из четырёх дублировались); stock-based делает `Lt` ортогональным `Rt`;
- **интервал 80%** — 95% на горизонте 1–3 мес слишком широк, чтобы быть ориентиром.

Поэтому старые «расхождения код ↔ модель» по этим пунктам **закрыты**: каноническая модель v3.0.0
приняла реальность кода. Документ для защиты против v2.0.2 более не нужен — канон сместился на v3.0.0.

## 4. Продуктовая обвязка вокруг неизменного ядра (v3.0.x)

Сверх алгоритма v3.0.0 добавлены слои данных и доступа, **не меняющие формулы** §7–§14:

- **Деньги `float` → `Decimal`/`numeric(14,2)`.** Десятичная арифметика ради точности денег (нет
  ошибок округления `float`). Результаты SAW/Avalanche/MC по смыслу те же — отличается только тип
  внутреннего представления.
- **Мультивалюта.** Каждый объект хранит валюту; суммы приводятся к базовой валюте пользователя
  (USD-пивот) **до** входа в движок. На вход алгоритма по-прежнему поступают суммы в одной валюте.
- **Многопользовательский режим.** Изоляция по `user_id`; движок вызывается с данными конкретного
  пользователя. Сам расчёт идентичен.
- **Provider-agnostic ingestion.** Контракт `FinancialSnapshot` ↔ `FinanceEngine` (Protocol); реальное
  ядро обёрнуто адаптером `CoreFinanceEngine`. Логика ядра не дублируется и не меняется.
- **`stable_baseline` в прогнозе.** Прогноз отдаёт долю регулярных (`is_recurring`) доходов/расходов —
  характеризует надёжность прогноза, но **не входит** в точечный прогноз или доверительный интервал
  (математика §12 не тронута).
