# Расхождения «код vs ВКР/модель» — talking points

Осознанные отличия между `Математической_моделью_v2_0_2.md` (ВКР) и реализацией (v2.1.0),
с обоснованием каждого. Документ для защиты: если комиссия откроет ВКР и увидит другие числа —
это объяснимо, а не ошибка.

---

## 1. Доверительный интервал Monte-Carlo: 80% вместо 95%

- **Модель (§12.2):** 95% интервал `[Q2.5 … Q97.5]`.
- **Код (`forecast.monte_carlo_intervals`, `forecasting.py`):** **80% интервал `[p10 … p90]`**.
- **Обоснование:** 80% даёт более «читаемый» и менее пугающий пользователя диапазон — края 95%
  на коротких финансовых рядах слишком широки и подрывают доверие к прогнозу. Для бытового
  планирования 80% информативнее. Метод (Monte-Carlo вокруг точечного SES) идентичен — отличается
  только ширина перцентилей.

---

## 2. Порог ликвидности в фильтре допустимости: 0 вместо 0.30

- **Модель (§6):** ограничение допустимости `Lt(a) ≥ L_min = 0.30`.
- **Код (`filtering.py`, `LT_CRIT = 0.0`):** фильтр допустимости отсекает альтернативы только по
  `Rt ≥ 0` и `Dt ≤ 0.40`; порог `Lt` по умолчанию мягкий (0).
- **Обоснование:** `L_min = 0.30` сохранён как **норма для диагностики и дашборда** (подсветка
  показателя), но как жёсткий фильтр допустимости он отсекал бы у многих реальных пользователей
  *все* альтернативы (Lt < 0.30 при положительном Rt — частая ситуация), оставляя пользователя без
  рекомендации. Порог вынесен в параметр (`l_min` в запросе) — пользователь может ужесточить его сам.

---

## 3. Точечный прогноз: SES с накоплением баланса

- **Модель (§12.1):** простое экспоненциальное сглаживание ряда.
- **Код (`forecasting.py`):** SES прогнозирует доходы/расходы/платежи, а баланс `Bt` **накапливается**
  по формуле 35 ВКР (`bt_running += cf − p`), и `Rt` считается на накопленном балансе.
- **Обоснование:** это не противоречие, а уточнение — прогноз свободных денег учитывает накопление
  остатка по месяцам, что точнее «плоского» SES по самому Rt.

---

## 4. Что совпадает с моделью

- Дискретизация альтернатив: шаг 0.20 → до 21 распределения (`C(7,2)`), как в §4.3.
- SAW-свёртка и min-max нормализация критериев — как в §7–8.
- Avalanche с OCR-фильтром `r_bench` — как в §10.
- Веса категорий целей и срочность — как в §11.
- Пороги `Dt ≤ 0.40` (ПДН ЦБ), `BLR ∈ [2.5; 6]` (Greninger) — как в §14.

---

## 5. v3.0.0 — международный слой: что добавлено сверх модели

Математическая модель v2.0.2 описывает **алгоритм** (показатели, целевая функция, фильтры).
Релиз v3.0.0 добавляет **инфраструктуру вокруг алгоритма**, не меняя его. Для защиты важно:
ни одна формула из §1–§14 модели не тронута — добавлены лишь слои данных и доступа.

- **Деньги во `float` → `Decimal`/`numeric(14,2)`.** Модель оперирует вещественными числами;
  реализация v3.0.0 перешла на десятичную арифметику ради точности денег (нет ошибок
  округления `float`). Результаты SAW/Avalanche/MC по смыслу те же — отличается только тип
  внутреннего представления. Это закрывает расхождение DOC-08/REFACTOR-03 из реестра.
- **Мультивалюта (FR-19).** Модель безвалютна (все суммы в одной валюте). v3.0.0 хранит валюту
  на каждом объекте и приводит к базовой валюте пользователя через курсы (USD-пивот) **до**
  входа в движок. На вход алгоритма по-прежнему поступают суммы в одной валюте — модель
  применяется без изменений.
- **Многопользовательский режим (DATA-03, INFRA-06).** Модель описывает расчёт для одного
  набора входных данных. v3.0.0 добавляет изоляцию по `user_id`: движок вызывается с данными
  конкретного пользователя. Сам расчёт идентичен; меняется лишь то, *чьи* данные подаются.
- **Provider-agnostic ingestion (DATA-07, KEEP-07).** Каноническая модель `FinancialSnapshot`
  и `FinanceEngine` Protocol — это контракт «источник ↔ движок». Реальное ядро обёрнуто
  адаптером `CoreFinanceEngine`, который маппит снимок во вход `run_planning` и результат —
  в `Recommendation`. Логика ядра не дублируется и не меняется (KEEP-01..05).
- **r_bench и профили риска** остаются как в модели (§9, §10): профиль 1..5 переносится в
  `RiskProfile` enum один-в-один, `r_bench` берётся из настроек/актива/дефолта 0.14.

**Вывод для защиты:** v3.0.0 — это «продуктовая обвязка» вокруг неизменного научного ядра ВКР.
Если комиссия спросит про Decimal/валюту/пользователей — это слои данных и доступа, а
многокритериальная оптимизация (SAW), Avalanche-фильтр и прогноз SES+Monte-Carlo работают
ровно по модели v2.0.2.
