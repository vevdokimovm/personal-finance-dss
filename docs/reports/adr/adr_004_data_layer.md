# ADR-004: Слой данных — оставляем SQLAlchemy ORM (Postgres в проде, SQLite в тестах)

Дата: 2026-06-29 · Статус: Accepted · Вахта: M

> Зафиксировано по запросу Василия. Рассматривался вариант «убрать SQLAlchemy и перейти
> фул на Postgres». Чтобы решение «оставляем ORM» и его причины не потерялись — и через
> год никто (включая нас на любом из 4 аккаунтов) не начал выпиливать ORM «потому что
> непонятно, зачем она». Оформлено по `docs/adr_template.md`.

---

## Контекст

К v4.24.0 бэкенд FINPILOT — зрелый production-grade слой: FastAPI + **SQLAlchemy 2.0** +
Alembic + Pydantic v2, вся история схемы в Alembic (миграции до 0026), матрица
SQLite + PostgreSQL, сотни тестов, покрытие выше 90%.

Возник вопрос: нужна ли вообще SQLAlchemy — может, «фул на Postgres», убрав ORM и работая с
СУБД напрямую?

**Ключевое, что надо снять сразу:** SQLAlchemy и Postgres — **не альтернативы и не «или-или»**.
SQLAlchemy — это слой доступа к данным, который *разговаривает* с Postgres (и с SQLite). «Убрать
ORM ради Postgres» — категориальная ошибка: Postgres остаётся СУБД в любом случае, вопрос лишь в
том, ходим мы в неё через ORM или сырым SQL руками.

**Силы, действующие на решение:**
- Желание упростить / быть «ближе к железу».
- Требование прод-уровня: надёжность, безопасность, поддерживаемость — и всё это силами одиночного
  разработчика-джуна.
- На ORM завязан весь существующий data-layer и инфраструктура тестирования.

---

## Решение

Оставляем **SQLAlchemy ORM** как слой доступа к данным. **Postgres** — единственная СУБД в проде.
**SQLite** — только в тестах (быстрая дешёвая матрица). Сырой SQL допускается **точечно** — через
`text()` / SQLAlchemy Core на доказанно горячем запросе, **без выноса ORM** из проекта.

---

## Альтернативы

**Выпилить ORM, писать сырой SQL (psycopg напрямую).** Отвергнут. Цена: переписать весь data-layer
(модели `app/database/models.py` + весь `app/database/crud.py`) и вручную маппить строки в объекты;
**потерять Alembic** (он сам построен на SQLAlchemy → исчезает весь механизм версионирования схемы и
история миграций); потерять матрицу SQLite + PostgreSQL; вернуть поверхность **SQL-инъекций**
(`SECURITY.md` прямо называет «только ORM, параметризованные запросы» защитой OWASP A03). Выигрыш —
маргинальная производительность, которая этому приложению не нужна.

**Заменить SQLAlchemy на другой ORM (Tortoise / Peewee / SQLModel).** Отвергнут. Миграционные
издержки без выгоды: SQLAlchemy 2.0 — индустриальный стандарт, типизирован (`Mapped[...]`), Alembic
ему родной. SQLModel вообще построен *поверх* SQLAlchemy + Pydantic — это не «уход от ORM».

**Postgres-only везде, убрать SQLite даже из тестов.** Отвергнут частично. В проде уже Postgres-only
— это правильно. Но в тестах SQLite ценен: zero-setup, быстрый, и **матрица SQLite + PostgreSQL ловит
дивергенцию** между СУБД (SQLite молча игнорирует нарушения FK — баг всплывает именно на PG). Это
актив, а не технический долг.

---

## Последствия

**Плюсы:**
- Сохраняем Alembic + всю историю миграций + версионирование схемы.
- Сохраняем матрицу SQLite + PostgreSQL — дешёвая страховка от дивергенции диалектов.
- Параметризованные запросы из коробки — защита от SQL-инъекций (OWASP A03).
- Типобезопасность моделей (`Mapped[...]`), как `Decimal` для денег — ошибку ловит раньше.
- Ноль миграционных издержек — ничего не переписываем.

**Минусы / на что подписались:**
- Небольшой оверхед ORM-абстракции против сырого SQL — для нагрузок продукта несущественно.
- Нужно держать совместимость схем SQLite ↔ PostgreSQL (дисциплина уже есть: `server_default=
  sa.false()` для boolean и т.п. — задокументировано в `docs/engineering_practices.md`).

**Влияние:**
- Data-layer не меняется. Прод — Postgres, тесты — матрица SQLite + PostgreSQL.
- Сырой SQL — только точечно через `text()` при **доказанной** необходимости (профилирование), а не
  по умолчанию.
- В роадмап отдельным пунктом **не идёт** — это решение класса «не трогать».
