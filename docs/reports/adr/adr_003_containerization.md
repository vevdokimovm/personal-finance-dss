# ADR-003: Контейнеризация — Docker + docker-compose для деплоя

Дата: 2026-06-29 · Статус: Accepted · Вахта: M

> Фиксируем уже принятое и реализованное решение контейнеризировать приложение (Dockerfile +
> docker-compose уже в репозитории), чтобы контекст «почему Docker, а не bare-metal или PaaS» не
> потерялся. ADR фиксирует и состоявшиеся решения — ради ответа на «почему так?» позже. По
> `docs/adr_template.md`.

---

## Контекст

Прод-цель — деплой на **VPS** (веха 9), с хранением данных граждан РФ в РФ (152-ФЗ). Нужны:
воспроизводимость окружения (Python 3.12 + системные зависимости), изоляция, простой повторяемый
деплой, паритет dev ≈ prod.

**Ограничение РФ:** Docker Hub заблокирован для российских IP → базовые образы тянем через зеркала
(`dh-mirror.gitverse.ru`, `dockerhub.timeweb.cloud`). Это уже учтено в инструкциях проекта.

В репозитории уже есть: `Dockerfile`, `docker-compose.yml`, `docker-compose.prod.yml`,
`docker-entrypoint.sh`.

---

## Решение

Контейнеризируем приложение через **Docker**; оркестрация на VPS — **docker-compose** (dev:
`docker-compose.yml`, прод: `docker-compose.prod.yml`). Запуск — `docker compose up --build` (**без
`-v`**, чтобы не сносить тома с данными). Базовые образы — через РФ-зеркала Docker Hub.

---

## Альтернативы

**Bare-metal / systemd на VPS** (venv + systemd-юнит, без контейнера). Отвергнут: ручная настройка
окружения, хуже воспроизводимость, дрейф «works on my machine», сложнее откат.

**Зарубежный PaaS** (Heroku-подобное / облачные контейнеры). Отвергнут: для РФ-аудитории и 152-ФЗ
(хранение данных в РФ) нужен контроль над хостингом; зарубежный PaaS не подходит по data-residency.
VPS + Docker проще и полностью под контролем.

**Не контейнеризировать вовсе.** Отвергнут: прод без изоляции и воспроизводимости — шаг назад от
прод-уровня.

---

## Последствия

**Плюсы:**
- Воспроизводимое окружение, изоляция процессов, простой повторяемый деплой на VPS.
- Паритет dev ≈ prod — меньше класса багов «на сервере иначе».
- Под контролем хостинг → совместимо с требованием хранения данных в РФ.

**Минусы / на что подписались:**
- Оверхед Docker (образы, сборка).
- В РФ — зависимость от зеркал Docker Hub (учтено, не блокер).
- Прод-образ нужно поддерживать в форме (обновления базового образа, безопасность).

**Влияние / что дошлифовать к прод-уровню (веха 9):**
- Multi-stage build (тонкий финальный образ).
- Non-root пользователь в контейнере (**уже есть**, по `SECURITY.md`).
- Healthcheck в compose/Dockerfile.
- Пиннинг базового образа по digest (воспроизводимость + безопасность).
- `.dockerignore` (не тащить в образ лишнее).
- Прод-compose с nginx + TLS (Let's Encrypt) + секреты через env-файлы окружения.
- Отдельным пунктом роадмапа не идёт — это часть **вехи деплоя (9)**.
