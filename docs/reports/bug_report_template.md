# Шаблон баг-репорта

> Формат карточки дефекта, принятый в проекте. Главное поле — последнее («почему не поймали
> раньше»): оно важнее самого фикса, потому что показывает дыру в процессе, а не в коде.

---

## Две оси оценки

Не путать — это разные вещи:

- **Severity** (тяжесть) — насколько сильно ломается продукт. `Critical / Major / Minor / Trivial`.
- **Priority** (срочность) — когда чиним. `P0` (сейчас) … `P3` (когда дойдут руки).

Опечатка на главной = Severity Trivial, но Priority P0. Падение редкого экспорта = Severity
Major, но, может, Priority P2.

---

## Шаблон карточки

```markdown
### BUG-XXX · [Severity / Priority] Краткий заголовок

| Поле | Значение |
|---|---|
| Окружение | OS, браузер (движок!), Docker/локально, версия |
| Компонент | файл/модуль/слой |
| Статус | Open / In Progress / Closed |

**Как проявлялся:**
Что видит пользователь. Симптом, а не догадка о причине.

**Шаги воспроизведения:**
1. …
2. …
3. …

**Ожидаемо / фактически:**
- Ожидалось: …
- Фактически: …

**Корневая причина:**
Реальная техническая причина (не симптом). Почему именно это ломалось.

**Фикс:**
Что изменено и почему именно так.

**Почему не поймали раньше:**
Какая дыра в процессе/тестах это пропустила. Что добавить, чтобы класс таких багов
ловился автоматически. ← самое важное поле.
```

## Пример (реальный)

```markdown
### BUG-019 · [Critical / P0] FK-нарушение при удалении в PostgreSQL

| Поле | Значение |
|---|---|
| Окружение | PostgreSQL 16, Docker |
| Компонент | crud, каскад удаления |
| Статус | Closed |

**Как проявлялся:** удаление цели с привязанной историей → ошибка целостности.

**Шаги воспроизведения:**
1. Создать цель, привязать к ней операции.
2. Удалить цель.
3. На PostgreSQL — исключение FK; на SQLite — проходит.

**Ожидаемо / фактически:**
- Ожидалось: цель и связанные строки удаляются.
- Фактически: ошибка внешнего ключа на PostgreSQL.

**Корневая причина:** SQLite по умолчанию не форсит FK, баг был невидим; на PostgreSQL FK строгие.

**Фикс:** явное удаление дочерних строк перед родителем.

**Почему не поймали раньше:** вся разработка и тесты шли на SQLite. Урок: схемные изменения
проверять на PostgreSQL — теперь это правило (см. CONTRIBUTING §4).
```

## Чеклист при заведении бага

1. Указал ли **движок** браузера (WebKit/Safari, Chromium, Firefox), а не только название?
   Разные движки = разные баги.
2. Отделил ли симптом от причины?
3. Заполнил ли «почему не поймали раньше» — и добавил ли тест/проверку под этот класс?
4. Проставил ли обе оси (Severity и Priority) — они независимы.
