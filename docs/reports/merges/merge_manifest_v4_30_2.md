# Merge-манифест линии v4.30.2 — выравнивание flake8-гейта на весь проект

> **Назначение.** Прикладная карта ОДНОЙ линии (`v4.30.2`, вахта V) для слияния с параллельными
> наработками других вахт (J/M/S). Общая методология — `docs/merge_and_fork_guide.md`; этот файл
> инстанцирует её процедуру (§4) и гейты (§5) под конкретный батч. Цель — чтобы мёржер за минуты понял,
> что эта линия трогает, где риск конфликта и какой инвариант нельзя сломать, не читая весь диф вслепую.

- **База линии:** `v4.30.1` (хвосты вехи 4: фикс reverse-миграции 0009 + чистка чекбоксов 4.4).
- **Задача:** гигиена гейта — привести `flake8 .` (весь проект) к 0. Не фича, не бэк-логика.
- **Тип:** PATCH. Миграций нет. Содержательный код `app/` **не тронут вообще**.
- **Дизайн-решение:** **Вариант A** (выровнять весь проект под `flake8 .` = 0 реально, не игнор-конфигом).
  Решение Василия. Отвергнуты: **B** (сузить CI-гейт до `flake8 app/` — прячет tests от линтинга) и
  **C** (per-file-ignore E501 для tests/ в `.flake8`). **Не переоткрывать на B/C при слиянии** (C остаётся
  задокументированным откатным путём на случай устойчивой merge-боли — см. §8).

## 1. TL;DR для мёржера

Батч **широкий, но неглубокий**: 23 файла `tests/`/`loadtest/` затронуты **форматированием** (разбивка
длинных строк + точечный `# noqa: E501` + 2 реальных F841 + переписанная XML-фикстура) — плюс убран
`black` из pre-commit и обычная обвязка релиза. **Ни один файл `app/` (кроме `config.py`::версия) не
изменён.** Логика, модели, миграции, схемы, роуты — байт-в-байт как в `v4.30.1`.

Главный нюанс не в дифе, а в **смене скоупа гейта**: до этой линии руками гоняли `flake8 app/` (узко),
а CI гоняет `flake8 .` (весь проект). v4.30.2 привёл весь проект к 0. **Последствие для слияния:** после
merge с любой вахтой, чьи `tests/`/`loadtest/` содержат длинные строки или unused-переменные, `flake8 .`
снова станет НЕ-ноль. Мёржер **обязан** прогнать именно `flake8 .` (не `app/`) и привести новые нарушения
тем же методом (§3, §8). Это не косметика на вкус — иначе первый CI-run красный.

Если другая вахта НЕ трогала те же тест-файлы → слияние тривиально (перенести правки + прогнать
`flake8 .`). Если трогала → текстовые конфликты по форматированию (§3), разрешаются в пользу **содержания**
тестов другой вахты с последующим выравниванием её длинных строк.

## 4. Инварианты-контракты — НЕ сломать при слиянии

- **I1. `app/` — вне этой линии.** Содержательный код приложения этой линией не изменён (кроме версии в
  `config.py`). При слиянии любые расхождения в `app/**` (кроме `config.py`) — это правки ДРУГИХ вахт,
  не конфликт с v4.30.2. Не тащить сюда и не «разрешать» их через призму этой линии.
- **I2. Гейт — полный `flake8 .`, не `app/`.** Суть линии: весь проект (не только `app/`) держится в 0.
  После merge проверять `flake8 .`. Не «оптимизировать» проверку до `flake8 app/` — это откат к варианту B,
  под которым `tests/` копили нарушения незаметно (первопричина этого батча).
- **I3. `black` под запретом (§2).** Не возвращать `psf/black` в pre-commit ни в каком виде. Форматирование
  только точечное (`autopep8 --select`). black-хук — латентная merge-мина (`make precommit` переформатирует
  весь проект → массовый диф → конфликты между вахтами).
- **I4. noqa точечный ≠ игнор-конфиг.** E501-подавления живут как `# noqa: E501` **на конкретных строках
  данных**, а НЕ как `per-file-ignores` в `.flake8`. Это намеренно: новый длинный код в тестах flake8
  по-прежнему ловит. Не сворачивать точечные noqa в конфиг-правило (это вариант C — отвергнут; допустим
  лишь как осознанный откат при устойчивой merge-боли, §8).
- **I5. `_CBR_XML` — данные идентичны.** Скобочная конкатенация даёт тот же XML, что исходные тройные
  кавычки (включая `\n`-разделители). Не «чинить» обратно на тройные кавычки без причины — вернётся E501.
  Если правятся данные фикстуры — сохранять построчную форму + noqa и проверять парсер.
- **I6. F841-фиксы сохраняют side-effect.** В `test_cross_household_isolation` вызов `_create_household`
  для `b` обязателен (делает `b` членом семьи-2). Удалено только присваивание в `hid2`, не вызов. Не
  «дочищать» вызов — сломается смысл теста изоляции.

**Почему вариант A, а не B/C (фиксирую, чтобы при слиянии не «улучшили» вбок).** B (сузить CI до `app/`) —
прячет `tests/` от линтера навсегда: F841 и unused-imports в тестах не ловятся, для «hireable» репозитория
слабый сигнал. C (per-file-ignore E501 в `.flake8`) — честнее B, но глушит E501 в тестах целиком, включая
будущий код. A — реально приводит проект к 0 и оставляет enforcement нового кода (точечный noqa виден в
дифе, новую длинную строку без noqa flake8 поймает). Цена A — широкий одноразовый диф форматирования
(принято). Решение Василия.

## 5. Разложение этой линии по измерениям (для шага 3–4 гайда — что здесь сильное)

- **Гигиена гейта.** Устранён скрытый рассинхрон (гоняли `app/`, CI гоняет `.`) — вскрыт до первого
  реального CI-run, когда он был бы красным. Первопричина найдена и зафиксирована (WATCHLOG §4).
- **Метод.** Инженерное разделение: код → автоформат (`autopep8 --select`), данные → точечный noqa,
  автоформатная порча golden → откат+noqa. Не «black всё подряд», не «ignore всё подряд».
- **Безопасность данных.** XML-фикстура переоформлена без изменения байт (доказано зелёными тестами
  парсера), не «на глаз».
- **Обвязка.** Полный релизный цикл (версия/README/CHANGELOG/WATCHLOG §0/§3/§4), §3 ровно 10; попутно
  разморожен зависший бейдж README.

Эта линия по измерению «логика/тесты по существу» **пустая** (ничего содержательного не добавляет) — при
конфликте с вахтой, добавившей реальные тесты/фичу, приоритет её содержанию; форматную часть этой линии
пересобрать поверх.

## 6. Чек-лист слияния ЭТОЙ линии (инстанс процедуры гайда §4)

1. Распаковать все линии в `cmp/<вахта>`; свою (`v4.30.2`) — базой форматирования. Сверить `app/config.py`

2. `diff` по `app/**` между линиями. Если расходится что-то кроме `config.py` — это правки других вахт;
   принять их как истину `app/` (эта линия `app/` не трогала), не мешать в форматный мёрдж.
3. Объединить **содержание** `tests/`/`loadtest/`: базой брать тесты той вахты, что добавляла реальные
   кейсы; форматные хунки v4.30.2 (разбивка строк, noqa) накатывать поверх слитого содержания, не наоборот.
4. `.pre-commit-config.yaml` — привести к версии БЕЗ `black` (I3).
5. `_CBR_XML` — если данные фикстуры не расходятся, взять конкатенацию v4.30.2 (I5); если расходятся —

7. **Прогнать `flake8 .`** (не `app/`!). Привести ВСЕ нарушения из чужих `tests/` методом §2 (I2).
8. Гейты заново (§7 ниже).

## 8. Семантические конфликты — особое внимание (не ловятся текстовым diff)

- **Скоуп гейта после merge.** Самый частый промах: прогнать `flake8 app/` (по привычке) вместо `flake8 .`
  → чужие длинные строки в `tests/` проскочат, CI покраснеет на push. Гнать всегда полный `.` (I2).
- **black-регресс.** Если у другой вахты в pre-commit присутствует `psf/black` (не убирала / вернула) —
  текстовый мёрдж может протащить его обратно. Убрать (I3). Это запрет §2, не «выбор по вкусу».
- **XML-обратный «фикс».** Если другая вахта держит `_CBR_XML` в тройных кавычках, наивный «взять более
  простую версию» вернёт длинные строки → E501 → красный `flake8 .`. Сохранить построчную конкатенацию
  (I5), данные при этом — по актуальной вахте.
- **F841-возврат.** Если сливается версия `test_households.py`, где `b_id`/`hid2` снова присваиваются без
  использования — F841 всплывёт. Сохранить: `b_id` не нужен, `hid2` без присваивания, вызов остаётся (I6).
- **Свёртка noqa в конфиг.** Соблазн «убрать 45 noqa, добавив per-file-ignore в `.flake8`» — это переход
  на вариант C. Допустим ТОЛЬКО как осознанный откат при устойчивой merge-боли по форматным хункам, с
  фиксацией решения; по умолчанию держать точечные noqa (I4), чтобы enforcement нового кода в тестах жил.
