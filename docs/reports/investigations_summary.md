### INV-BROWSERS-SANDBOX — firefox/webkit не запускаются в песочнице

- **Подозрение:** сломана E2E-инфраструктура (`tests/e2e/conftest.py`, флаги Playwright, мультибраузерный прогон настроен криво).
- **Снято с подозрения (корень):** код и конфиг чисты — те же тесты зелёные на chromium (20 E2E). Бинарей firefox/webkit физически нет, потому что Playwright качает их ИСКЛЮЧИТЕЛЬНО с Azure-CDN (`cdn.playwright.dev` / `playwright.download.prss.microsoft.com`), а он вне сетевого allowlist песочницы (403). Граница «от нас не зависит», как ЦБ-403. chromium доступен только потому, что предустановлен в образе.
- **Что опробовано (все каналы перебраны, §11):** (а) GitHub releases — `PLAYWRIGHT_DOWNLOAD_HOST=https://github.com playwright install firefox` строит `github.com/builds/firefox/1495/firefox-ubuntu-24.04.zip` → **404** (хост доступен, билда нет); (б) npm-пакет `playwright@1.56.0` — 866 КБ, 170 файлов, **ноль** бинарей браузеров; (в) системный firefox через `executable_path` — отвергается, `p.firefox.launch()` жёстко привязан к juggler-патченному managed-бинарю; (г) apt — `firefox-esr` без кандидата, и принципиально apt даёт только runtime-либы, не бинарь браузера.
- **Почему не вышло:** Playwright раздаёт кастомные (juggler/WebKit-патченные) сборки только через свой CDN; ни GitHub releases, ни npm-пакет их не несут, системными браузерами не заменить.
- **Что можно попробовать ещё / следующее:** если allowlist песочницы расширят на CDN Playwright — `playwright install firefox webkit` поднимет их без правок (тесты браузер-агностичны). До тех пор: chromium в песочнице, мультибраузер (chromium + firefox + webkit) — в CI (полный тир), где CDN доступен.
- **Статус:** закрыт окончательно (внешнее ограничение, все каналы исчерпаны). Полный разбор — у владельца (правило §12).

### INV-MATHMODEL-FALSEALARM — «одинаковые планы для разных риск-профилей»

- **Подозрение:** `test_profiles_produce_different_plans` давал одинаковые планы для разных профилей → «баг математического ядра» (дорого: ядро — сердце всего продукта).
- **Снято с подозрения (корень):** ядро чисто — это **гонка в самом E2E-тесте**. `/planning` при загрузке делает авто-расчёт с дефолтным `risk=3`; тест читал его результат вместо ручного.
- **Что опробовано (доказано тремя независимыми уровнями):** прямой вызов API с UI-payload (risk=5 → деньги «на цели» — ядро дифференцирует профили); чтение `app.js` (нашёлся авто-расчёт на загрузке страницы); перехват реальных POST из браузера (видны ДВА запроса — авто и ручной).
- **Решение:** `expect_response` на ручной расчёт в тесте; продуктовый код/ядро не трогали.
- **Что запомнить:** прежде чем винить ядро — проверить, не читает ли тест чужой (авто) результат при загрузке страницы.
- **Статус:** закрыт. Породил правило §12. Полный разбор — у владельца (правило §12).

### INV-PG-SANDBOX — «PostgreSQL нельзя тестировать в песочнице»

- **Подозрение:** PG-тестирование объявлено невозможным в песочнице и переложено на владельца как «долг».
- **Снято с подозрения (корень):** дефекта/блокера нет — гипотезу просто не проверили фактически. PostgreSQL **поднимается** в песочнице.
- **Что опробовано / как поднять:** `apt-get install postgresql` (БЕЗ `apt-get update` — он падает на стороннем nodesource-репо 403; ubuntu-репы рабочие) → `pg_ctlcluster 16 main start` → роль/БД через `su postgres -c psql` → `DATABASE_URL=postgresql+psycopg://finpilot:finpilot@localhost:5432/finpilot_test`. Полная матрица (558 тестов) зелёная на PG 16.14.
- **Почему «не выходило» раньше:** не пробовали — отмашка вместо проверки.
- **Что запомнить:** §11 — не объявлять невозможным без фактического перебора каналов.
- **Статус:** закрыт. Породил правило §11 и процедуру запуска PG в WATCHLOG §5. Полный разбор — у владельца (правило §12).

### INV-CBR-403 — `cbr.ru` отдаёт 403 датацентровым IP

- **Подозрение:** ключевая ставка не тянется с `cbr.ru` — не баг ли это нашего фетчера/парсера.
- **Снято с подозрения (корень):** код чист — graceful fallback корректен. 403 — DDoS-защита провайдера на стороне `cbr.ru` для датацентровых IP (внешнее, от нас не зависит).
- **Что опробовано / обход:** в проде — зеркало `cbr-xml-daily.ru`; в коде — graceful fallback `r_bench=0.14` при недоступности; реальные данные добываются через зеркало.
- **Сопутствующая карточка:** LIM-001 (округление в обходном парсере).
- **Что можно попробовать ещё:** при необходимости — кэш ставки в БД, чтобы пережить кратковременную недоступность зеркала. **(Сделано в v4.24.0** для ключевой ставки: durable БД-кэш `cbr_key_rate_cache`.)
- **Сопутствующий инцидент:** INC-CBR-KEYRATE-PARAM — у **ключевой ставки** оказался ОТДЕЛЬНЫЙ реальный дефект кода (регистр `<toDate>`), который маскировался этой же историей про 403. Урок: graceful fallback скрыл реальный баг, потому что симптом совпал с известным внешним ограничением.
- **Статус:** закрыт (внешнее ограничение, обойдено зеркалом). Полный разбор — у владельца (правило §12).

### INV-VISUAL-FLAKINESS — визуальная регрессия флакала между прогонами

- **Подозрение:** второй прогон visual-регрессии (14 страниц × 2 темы) давал 20/28 падений против собственного baseline минутной давности → UI реально нестабилен / вёрстка плывёт / тема не применяется (для финпродукта «прыгающая вёрстка» — серьёзно).
- **Снято с подозрения (корень):** дефекта в продукте НЕТ. Вёрстка стабильна (размеры снимков совпадали 1280×2339), тема применяется. Флак — артефакт способа снятия/сравнения: рендер недетерминирован в мелочах (анимации, canvas-графики Chart.js, async-шапка `auth.js`), а наивный побитовый дифф (`px != (0,0,0)`) засчитывал ЛЮБОЕ микро-отличие, включая субпиксельный антиалиасинг глифов на длинных текстовых страницах.
- **Что опробовано (5 итераций, решающий шаг — последний):** заморозка анимаций + `fonts.ready` + скрытие canvas (20→13); пауза + порог 3% (→9); маска `#auth-widget` (→11, НЕ помогло — значит источник не в шапке); **допуск на канал в диффе (пиксель значим только если канал > 40) → 0**. Диагностический замер показал: размеры идентичны, различия 2.28% рассеяны по тексту (антиалиасинг), а не компактный сдвиг блока.
- **Почему «не выходило» промежуточно:** боролись с симптомами (анимации, шапка), пока не нашли доминирующий источник — субпиксельный антиалиасинг текста, который снимается только допуском-на-канал, а не стабилизацией DOM.
- **Что запомнить / что можно ещё:** «пиксель отличается» ≠ «UI сломан»; отличить артефакт от дефекта дёшево — посмотреть размеры снимка и характер различий (рассеянный микро-разброс = антиалиасинг; компактный сдвиг = регрессия). Эталоны платформо-зависимы → генерировать в CI, не коммитить (`.gitignore`).
- **Статус:** закрыт (артефакт методики теста, не дефект продукта). Полный разбор — у владельца (правило §12).

### INV-A11Y-ICEBERG — «два контраста» оказались системным долгом палитры

- **Подозрение:** расширенный a11y-тест нашёл на dashboard 2 контраст-нарушения (кнопка 3.53, футер 4.23) → выглядело как точечный фикс на пару элементов. Страшная гипотеза: это верхушка айсберга — вся палитра (особенно light) не проходит WCAG AA, а «доступность» продукта — фикция, которую не измеряли.
- **Снято с подозрения / подтверждено:** страшная гипотеза в значительной мере ПОДТВЕРДИЛАСЬ. Полный скан (14 страниц × 2 темы вместо одной) показал ~40 уникальных контраст-нарушений, доминирует light-тема (semantic-цвета на светлом фоне 1.2–3.6, badges, ссылки accent-hl ~2.5, приглушённый текст 3.1–4.4). Долг СИСТЕМНЫЙ, не точечный — точечный фикс закрыл бы 2 из ~40.
- **Что опробовано / решающий шаг:** первый скан (1 страница × 1 тема) дал иллюзию «2 контраста»; точечные фиксы + сужение baseline до 0 → a11y упал 16/28 (значит осталось много); **полный скан 14×2 со сбором уникальных (fg,bg,ratio) пар** вскрыл системный масштаб. Расширение ОХВАТА ИЗМЕРЕНИЯ превратило «2 бага» в «системный долг».
- **Корень (чеховское ружьё):** (1) light-палитра проектировалась «на глаз» под тёмную тему, semantic/приглушённые цвета на светлом фоне не проверялись на контраст; (2) долг был НЕВИДИМ, потому что единственный инструмент (live-a11y) молча не работал (CSP резал axe — см. INC-A11Y-DEADTEST). Ружьё не стреляло не потому что его не было, а потому что крючок был отключён. Тест ожил → выстрелило системно.
- **Что запомнить:** масштаб находки определяется ОХВАТОМ ИЗМЕРЕНИЯ, не самой находкой — мерить системно (все страницы × все темы) с первого раза; не доверять «удобной» первой гипотезе масштаба; «свойство в реестре ≠ свойство в продукте» (галочка без живого измерения — фикция); сломанный инструмент не создаёт долг, но позволяет ему копиться невидимо.
- **Статус:** разобран; сам долг — в работе (ревизия палитры под WCAG AA, поэтапно). Дефект реальный, поэтому пограничный с инцидентом — но предмет разбора это ПАТТЕРН/УРОК, а сам долг зафиксирован отдельно (a11y-ratchet + WATCHLOG §5). Полный разбор — у владельца (правило §12). Парный — INC-A11Y-DEADTEST.

### INV-A11Y-ANIMATION — a11y-скан раздувал контраст-нарушения во время fade-in анимации

- **Подозрение:** на этапе 2 ревизии палитры axe-скан стабильно показывал ~200 color-contrast нарушений в light, причём приглушённый текст помечался как 3.17:1, хотя computed-цвет `#4a5566` даёт 7.5:1 на белом. Страшная гипотеза: light-палитра дефектна катастрофически, даже правильные токены не работают, долг неустраним точечно, подход к WCAG AA провален.
- **Опровергнуто / корень:** страшная гипотеза ОПРОВЕРГНУТА. ~90% «нарушений» — ЛОЖНЫЕ, артефакт МЕТОДИКИ замера. Секции-карточки имеют fade-in (`opacity: 0→1`); `axe.run` срабатывал ДО завершения анимации и замерял текст при `opacity≈0.15` (полупрозрачный, смешан с фоном → контраст занижен). Токены корректны (доказано прямым `getComputedStyle`); проблема в МОМЕНТЕ замера — измеряли переходное состояние, не финальное.
- **Решающий шаг:** обход дерева РОДИТЕЛЕЙ span (а не плоский getComputedStyle элемента) → найден `SECTION opacity=0.157`. Текст имел opacity 1 и верный цвет, но УНАСЛЕДОВАННАЯ opacity секции (анимация) делала его полупрозрачным в композиции — это и видит axe. После заморозки анимаций (`animation/transition-duration:0s` + settle 300 ms) перед `axe.run`: dark=0, light=20 (реальный остаток, добит → color-contrast=0).
- **Что это НЕ:** НЕ дефект палитры (токены AA по computed); НЕ неустранимый долг (ложные снимаются стабилизацией, не правкой цвета); НЕ дефект axe (он корректно учитывает opacity — ошибка КОГДА его звали); НЕ то же что INV-A11Y-ICEBERG (там охват ЗАНИЖАЛ, тут стабилизация ЗАВЫШАЛА — два независимых фактора достоверности замера).
- **Что запомнить:** замораживать анимации/переходы ПЕРЕД любым инструментальным замером рендера (axe, скриншоты, пиксельный дифф) — тот же класс причины, что INV-VISUAL-FLAKINESS; `networkidle` ≠ «анимации завершены»; при противоречии computed↔инструмент идти вверх по дереву искать унаследованную opacity/фон; достоверность замера = охват × стабилизация.
- **Статус:** закрыт (артефакт методики теста, не дефект продукта). Внедрено в `_run_axe`. Полный разбор — у владельца (правило §12). Связанные — INV-VISUAL-FLAKINESS (тот же класс), INV-A11Y-ICEBERG (второй фактор замера).

### INV-EMAIL-FIRE-AND-FORGET — «письма не приходят, а интерфейс пишет отправлено»

- **Подозрение:** давняя заметка владельца — пользователь регистрируется, интерфейс рапортует об успехе, но письмо на почту не приходит. Гипотеза: дефект в коде формирования или отправки письма (`EmailService`), либо эндпоинт врёт об отправке.
- **Снято с подозрения (корень):** дефекта в продуктовом коде НЕТ — это сумма ожидаемого поведения. Доказано на трёх уровнях: (1) **транспорт** `email_service.py` корректен — 9 тестов зелёные (STARTTLS-путь, SSL-путь, fallback отправителя, text+html-альтернативы, встраивание URL в verification/reset, no-op без конфига, сбой SMTP → False); (2) **эндпоинт** `routes_auth.py` не имитирует отправку — в dev (`email_enabled=False`) он возвращает `verification_url` прямой ссылкой в теле ответа; (3) **фронт** `auth.js` в dev честно показывает тост со ссылкой подтверждения («Аккаунт создан. Подтвердите email по ссылке»), а НЕ «письмо отправлено». «Пусто на почте» в dev — намеренный no-op без SMTP-конфига (self-hosted без почты должен работать), а не баг.
- **Что вскрыла диагностика (реальный пробел, не баг текущего поведения):** отправка идёт через `BackgroundTasks` (fire-and-forget), и её результат нигде не фиксировался наблюдаемым образом. На проде с живым SMTP сбой (неверный пароль, заблокированный хостером порт, rate-limit Yandex/Mail.ru, истёкший токен) виден только в текстовом логе `logger.warning` — в таблице `events` ничего. Для критичного канала (верификация email + сброс пароля, 152-ФЗ) это значит: о поломке узнаёшь по жалобам пользователей. Это и создавало ощущение «маскировки».
- **Решение (наблюдаемость, не правка транспорта):** обёртка `app/services/email_dispatch.py` — `dispatch_email()` вызывается из фоновой задачи и пишет событие `email_sent`/`email_skipped`/`email_failed` в `events` (домен вместо адреса — PII не утекает, как в `login_failed`). Подключена в 4 точки `routes_auth.py` через `partial`. Транспорт и синхронность регистрации не тронуты. +7 тестов (100% покрытие модуля).
- **Что запомнить:** «письма не приходят» в dev почти всегда = намеренный no-op без SMTP, не баг; прежде чем «чинить» отправку — проверить, не теряется ли РЕЗУЛЬТАТ фоновой задачи (наблюдаемость), и не отдаёт ли фронт dev-ссылку вместо имитации. Живой SMTP проверяется только на проде (P0.2) — но теперь статус каждой отправки виден в `events`.
- **Статус:** закрыт (ожидаемое поведение + добавлена наблюдаемость). Живая верификация с реальным SMTP — у владельца (P0.2, веха деплоя). Полный разбор — у владельца (правило §12).
