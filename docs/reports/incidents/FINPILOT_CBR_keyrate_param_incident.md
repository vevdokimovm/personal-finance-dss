# FINPILOT — Пост-мортем: INC-CBR-KEYRATE-PARAM (ключевая ставка всегда отдавала дефолт)

> Реальный дефект кода (инцидент), не внешнее ограничение. Парный к расследованию INV-CBR-403
> (там — внешний 403 для курсов валют; здесь — отдельный баг ключевой ставки, который маскировался
> той же историей). Формат — `docs/incident_postmortem_guide.md`. Версия фикса — **v4.24.0**.
>
> **Итог одной строкой:** SOAP-метод `KeyRate` у cbr.ru ASMX чувствителен к регистру параметров;
> в коде стояла строчная `<toDate>` вместо `<ToDate>` → ASMX не распознавал параметр → `ToDate`
> уходил в `DateTime.MinValue` → диапазон пустой → ноль строк `KR` в ответе → парсер возвращал
> `None` → «не удалось извлечь» → вечный fallback 0.16 ПРИ ЖИВОМ и доступном cbr.ru.

---

## Часть A. Что произошло

Кнопка «получить ставку с ЦБ» на `/api/planning/key-rate` всегда возвращала статический fallback
(0.16), хотя cbr.ru доступен. Эмпирическая улика — поле `detail` в ответе эндпоинта:

```
{"key_rate": 0.16, "source": "fallback", "as_of": "2026-06-29",
 "detail": "cbr.ru ответил, но ключевую ставку не удалось извлечь из ответа"}
```

Ключевое в `detail`: **«cbr.ru ответил»** — то есть HTTP 200, не 403 и не таймаут. Запрос дошёл,
ответ получен и распарсился как XML, но в нём не нашлось ни одной строки `KR` с числовым `Rate`.

### Влияние

Реальная ключевая ставка не доходила до пользователя. Хуже: `r_bench` (OCR — посленалоговая
альтернативная доходность, порог фильтра Avalanche «гасить кредит или копить») считался от
устаревшего статического дефолта 0.16 вместо реальной ставки (на момент инцидента ЦБ держал
~14.25%). Неверный `r_bench` смещает рекомендацию о приоритете гашения долга — прямой продуктовый
вред для финансового СППР. Near-miss по серьёзности: приложение ещё не на проде.

---

## Часть B. Корень (5 Whys)

1. **Почему пользователь видит дефолт?** `get_key_rate` ушёл в ветку fallback.
2. **Почему fallback, если cbr.ru ответил 200?** `_parse_latest_rate` вернул `None` — в ответе нет
   строк `KR`.
3. **Почему в ответе нет строк `KR`?** cbr.ru вернул пустой набор данных для запрошенного диапазона
   дат: `fromDate=сегодня−30`, `ToDate=…`.
4. **Почему диапазон пустой?** SOAP-метод `KeyRate` у cbr.ru ASMX **чувствителен к регистру имён
   параметров** и ждёт `<fromDate>` + `<ToDate>` (заглавная T). В коде была строчная `<toDate>`.
   ASMX не сопоставил `<toDate>` параметру `ToDate`, и `ToDate` принял значение по умолчанию
   `DateTime.MinValue` (0001-01-01). Диапазон `2026-… … 0001-01-01` инвертирован/пуст → ноль строк.
5. **Почему это не заметили раньше?** Симптом (fallback) совпал с историей INV-CBR-403 (cbr.ru 403
   датацентровым IP — внешнее). «Решили», что это то же внешнее ограничение, и не посмотрели в
   `detail`, который прямо говорил иное («ответил, но не извлечь» ≠ «нет связи / 403»).

**Подтверждение первоисточником.** Шаблон SOAP-запроса самого cbr.ru (страница метода в
`DailyInfoWebServ/DailyInfo.asmx`) показывает параметры `<fromDate>` и `<ToDate>` — заглавная T.
Тот же регистр виден у соседних методов сервиса (например `DragMetDynamic`). Гипотеза регистра
проверена по источнику, а не угадана.

---

## Часть C. Решение (v4.24.0)

1. **Фикс регистра** — `_build_soap_body` теперь шлёт `<ToDate>` (заглавная T). Регрессия закрыта
   тестом `TestSoapBody::test_uses_capital_todate` (в т.ч. явная проверка отсутствия `<toDate>`).
2. **Durable last-known-good кэш в БД** — новая таблица `cbr_key_rate_cache` (миграция 0026, модель
   `CbrKeyRate`). Рубежи резолюции: process-кэш на сутки → live cbr.ru → **БД-кэш** → статический
   fallback. Переживает перезапуск процесса И прод-сценарий «датацентровый IP → cbr.ru 403» (один раз
   прогрелся с РФ-IP — реальное значение живёт). Кэш работает на собственной короткоживущей сессии —
   транзакции пользовательского запроса не затрагиваются. Это реализация хвоста «что попробовать ещё»
   из INV-CBR-403.
3. **Устойчивый парсер** — `_parse_latest` извлекает дату вступления + ставку; теги `KR/DT/Rate` в
   любом регистре и с namespace/без; `Rate` с запятой или точкой; SOAP-fault / пустой ответ → честно
   `None`. Старый `_parse_latest_rate` сохранён (обратная совместимость).
4. **Самодиагностика** — при коде 200, но неизвлекаемой ставке начало ответа cbr.ru кладётся в
   `detail`. Будущие сбои этого класса читаются без доступа к cbr.ru из песочницы.
5. **Прогрев** — `refresh_key_rate()`: принудительный fetch + запись в кэш, мимо process-кэша и
   backoff (для cron на проде с российским IP).
6. **r_bench** — `get_opportunity_cost_rate` принимает источник `cache_db` наравне с `cbr`/`cache`.

Гейты: fast-прогон зелёный на SQLite (EXIT=0), покрытие 92.17% (порог 90). Миграция 0026 аддитивна
(новая таблица, без FK); PG-матрица для 0023–0026 — отдельным прогоном перед мёржем (проектная матрица
валидна до 0022). +24 теста в `tests/test_cbr_rate.py`. Blast-radius — только
`app/services/cbr_rate.py` (+ модель, миграция, автосброс кэша в conftest).

## Часть D. Честная граница (что проверяемо где)

- **В песочнице проверено:** фикс регистра в теле запроса, устойчивый парсер (включая сценарий
  инцидента «200, но не извлечь»), БД-кэш (upsert-идемпотентность, served-when-down, fallback),
  сетевые ветки (403/500/network), прогрев — всё на фикстурах и моках.
- **Только на проде/РФ-IP:** живой POST к cbr.ru из приложения. Из песочницы недоступен — bash-egress
  отдаёт 403 (та же граница, что в INV-CBR-403). Примечание: `web_fetch` (инфра Anthropic, другой
  канал) cbr.ru **достаёт** GET-ом — этим и подтверждён первоисточник запроса. SOAP — POST, поэтому
  живой вызов именно нашего запроса воспроизводится на машине с РФ-IP.
- **Верификация однозначна:** `curl -s http://127.0.0.1:8000/api/planning/key-rate` на Маке должен
  дать `"source":"cbr"` и реальную ставку; при сбое — `detail` покажет начало ответа cbr.ru.

## Часть E. Уроки

1. **Graceful fallback маскирует реальный баг, если симптом совпадает с известным внешним
   ограничением.** «Всё падает в fallback» выглядело как INV-CBR-403 (внешний 403) — и реальный
   дефект кода жил незамеченным. Смотреть в `detail`/корень, а не списывать на «то же самое».
2. **«cbr.ru ответил (200)» ≠ «cbr.ru недоступен».** Пустой-200, 403, таймаут, parse-error — четыре
   разные ветки с разной обработкой; их нельзя сваливать в один «fallback» без диагностики.
3. **Регистр имён в SOAP/ASMX значим.** При интеграции с внешним SOAP — сверять имена параметров по
   первоисточнику (шаблон запроса/WSDL), не по памяти.
4. **Наблюдаемость окупается.** Сниппет ответа в `detail` превращает «непонятно почему fallback» в
   мгновенный диагноз — особенно ценно, когда источник недоступен из среды разработки.
5. **Устойчивость внешнего источника — это кэш, а не «лучший дефолт».** Для регулируемой финансовой
   величины правильная резилентность — durable last-known-good реального значения, а не подгонка
   статической константы (она тоже устареет).
