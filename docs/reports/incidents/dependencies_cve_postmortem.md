# FINPILOT — Зависимости с CVE: incident report / post-mortem

> **Что это и в какой нотации.** Документ из двух частей в принятом проектном формате:
> - **Часть A — карточка находки** (SEC-001) по образцу `bug_report_template.md`. Про *факт*:
>   какие зависимости уязвимы, какие CVE, чем чинится, что достижимо в FINPILOT.
> - **Часть B — post-mortem** (ретроспектива): почему апгрейд был осознанно отложен
>   («чеховское ружьё»), почему это риск именно для финпродукта, разбор мифа про Python 3.12,
>   план устранения и уроки.
>
> Это не разбор уже починенного бага, а фиксация **отложенного техдолга безопасности** до того,
> как он «выстрелит». Безоценочно: цель — закрыть дыру в процессе (увидели CVE → забыли), а не
> найти виноватого.

**Итог одной строкой:** `pip-audit` (13.06.2026, чат «Разработчик IT») нашёл CVE в 4 пинованных
зависимостях; апгрейд был помечен как отложенный «в блок 7» и держится на `continue-on-error` в
CI. Самая опасная цепочка — **DoS через multipart** (`python-multipart` + `starlette`),
**достижимый с endpoint загрузки CSV-выписок**. Версия **Python 3.12 тут ни при чём** — она
поддерживается до октября 2028 и остаётся рекомендуемым production-таргетом.

# Часть A. Карточка находки

### SEC-001 · [Security / High — отложено] Уязвимые пинованные зависимости

| Поле | Значение |
|---|---|
| Обнаружено | 13.06.2026, чат «Разработчик IT» — `pip-audit -r requirements.txt` локально при настройке CI (INFRA-05) |
| Инструмент | `pip-audit` (OSV / PyPI Advisory DB) |
| Найдено | 12 advisories в 4 пакетах: `jinja2`, `python-multipart`, `starlette`, `python-dotenv` |
| Компонент | `requirements.txt`; CI-job `audit` в `.github/workflows/ci.yml` (`continue-on-error: true`) |
| Статус | **Open** — апгрейд отложен «в блок 7», на момент документа не выполнен |

**Как обнаружено.** При переносе CI добавлен шаг `pip-audit`. Прогон на зафиксированных версиях
вернул CVE — это и было обоснованием `continue-on-error` (иначе пайплайн сразу красный). Запись
тогда дословно: «вход в блок 7 — обновлю зависимости там, аккуратно с совместимостью
fastapi/starlette».

**Уязвимые версии и ремедиация (актуально на 06.2026):**

| Пакет | Сейчас | Ключевые CVE | Тип | Fixed in | Достижимо в FINPILOT | Практический риск |
|---|---|---|---|---|---|---|
| `python-multipart` | 0.0.9 | CVE-2024-53981 | DoS через деформированную границу `multipart/form-data` (CWE-770), CVSS 7.5 | 0.0.18+ | **Да** — путь загрузки CSV-выписок | **High** |
| `starlette` (транзитивно через `fastapi` 0.110.3) | ~0.37.x | CVE-2024-47874 | DoS: нет лимита числа полей/файлов в multipart | 0.40.0 | **Да** — тот же multipart-путь | **High** |
| `jinja2` | 3.1.4 | CVE-2024-56326, CVE-2024-56201 (→3.1.5), CVE-2025-27516 (→3.1.6) | Sandbox-breakout / RCE при рендере **недоверенных** шаблонов | 3.1.6 | Нет — шаблоны авторские, пользовательский ввод в них не попадает как код; данные авто-экранируются | Низкий |
| `python-dotenv` | 1.0.1 | CVE-2026-28684 | Symlink-following в `set_key` → перезапись произвольных файлов (CWE-59), локальный | 1.2.2 | Нет — `.env` только читается через `pydantic-settings`, `set_key`/`unset_key` не вызываются | Низкий |

> Примечание: `pip-audit` отрапортовал ~12 advisories суммарно — в таблице сведены содержательные
> CVE по каждому пакету (advisory-база может считать несколько GHSA-ID на одну проблему). Все
> четыре пакета закрываются одним апгрейдом, поэтому разделять по приоритету для самого факта
> обновления не нужно.

**Корневая причина.** Версии в `requirements.txt` запинованы и не обновлялись; новые CVE
публикуются после пина и копятся молча. `pip-audit` их видит, но шаг информативный
(`continue-on-error`), поэтому находка не блокирует и легко выпадает из фокуса.

**Почему «чеховское ружьё».** Сейчас не стреляет (CI зелёный, фичи едут), но:
1. **multipart-DoS достижим в проде** — endpoint загрузки выписок принимает `multipart/form-data`,
   а это ровно тот путь, который роняет `python-multipart` 0.0.9 / старый `starlette`. Один кривой
   запрос с мусором после границы → высокая загрузка CPU → стоп event-loop → отказ в обслуживании.
   Для финсервиса с планкой Т-Банк/Сбер это неприемлемо.
2. Чем дольше пин, тем больше CVE накопится к запуску — «ружьё» тяжелеет.

# Часть B. Post-mortem

> Безоценочно. Факты, причина отсрочки, разбор мифа, план, уроки.

## B.1. Хронология
1. **13.06.2026** («Разработчик IT») — настроен CI, `pip-audit` нашёл CVE в 4 пакетах. Решение:
   `continue-on-error` + апгрейд отложен «в блок 7».
2. **24.06.2026** («Разработчик IT №5») — всплыло напоминание про «чеховское ружьё», запрос
   собрать репорт; начат поиск формулировки по транскрипту — **чат оборвался на середине** (стал
   слишком большим).
3. **Сейчас** — находка восстановлена через поиск по прошлым чатам, CVE и фиксы сверены с
   актуальными источниками, оформлен этот документ.

## B.2. Почему отложили — и почему это всё-таки риск
Отсрочка была **осознанной и не лишённой логики**: на тот момент приоритетом были фичи/инфра, а
апгрейд `fastapi`/`starlette` тянет проверку совместимости (starlette пинуется внутри fastapi,
ломать стек на ходу опасно). Это нормальный инженерный размен — *при условии, что долг записан и
имеет срок*. Здесь он повис на `continue-on-error` без срока — это и есть дыра в процессе.

Риск именно для FINPILOT: из 4 находок две (multipart-цепочка) лежат **на реальном пути данных**
(загрузка выписок), а не в теории. Остальные две (`jinja2`, `python-dotenv`) практически
недостижимы — но чинятся тем же апгрейдом, так что разделять не нужно.

## B.3. Миф «Python 3.12 устарел»
Отдельным разделом, потому что в памяти эта тема слиплась с зависимостями. **Python 3.12 не EOL и
не «старый».** По данным python.org / endoflife.date: 3.12 в фазе security-fix, **поддержка до
31.10.2028** (~2.5 года запаса от сейчас), EOL Risk Score 22 (Low), и это **рекомендуемый
production-таргет**. Свежее: 3.13 (до окт-2029), последняя активная — 3.14.x. То есть 3.12 на
одну-две минорки позади «самой новой», но это **не риск** — апгрейд рантайма отдельной задачей не
горит. «Ружьё» — это зависимости, не Python.

Откуда могло слипнуться: курсовой шаблон CI был на 3.13, FINPILOT адаптировали под 3.12 — отсюда
ощущение «мы на более старой». Это про «не самую новую», а не про «устаревшую».

## B.4. План устранения (remediation)
По проектному правилу — **TDD-порядок**: сначала зафиксировать зелёную регрессию как baseline,
потом апгрейд, потом полный прогон.
1. Bump в `requirements.txt`: `python-multipart` → 0.0.18+, `jinja2` → 3.1.6, `python-dotenv` → 1.2.2,
   `fastapi` → актуальная (подтянет `starlette` ≥ 0.40 с фиксом DoS).
2. Сверить совместимость связки `fastapi` / `starlette` / `pydantic` (pydantic v2 уже стоит —
   проверить, что мажор не уехал и API не сломался).
3. Полная матрица: unit/integration + E2E, **оба движка** (SQLite + PostgreSQL).
4. **Defense-in-depth на multipart-путь** (нужно независимо от версий): лимит размера тела запроса
   (nginx `client_max_body_size` / прокси), rate-limit на upload, таймаут парсинга. Снимает
   остаточный DoS-риск на загрузке выписок даже при будущих CVE.
5. **Починить процесс:** `pip-audit` оставить, но добавить severity-gate — блокировать пайплайн на
   достижимых High/Critical, не блокировать на шумных Low. Иначе ружьё повиснет снова.
