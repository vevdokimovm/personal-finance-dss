## 2. Детальные карточки дефектов

Формат карточки — продакшен-стандарт: **как проявлялся → шаги воспроизведения → ожидаемо/фактически → корневая причина → фикс → почему не поймали раньше.** Последняя строка («почему не поймали») — то, что в зрелых командах пишут в постмортемах: она важнее самого фикса, потому что показывает дыру в процессе.

### BUG-001 · [Critical / P0] Невозможно удалить цель/обязательство/ликвидный актив в WebKit

| | |
|---|---|
| Окружение | macOS, DuckDuckGo (WebKit). На Chromium/Firefox не воспроизводился |
| Компонент | `frontend/static/js/app.js` — хелпер `api()`; роуты DELETE с ответом 204 |
| Версия фикса | v2.2.1 |

**Как проявлялся.** Пользователь жмёт «удалить» на цели/обязательстве/активе → запись не удаляется, в консоли/тосте ошибка `The string did not match the expected pattern`.

**Шаги воспроизведения.**
1. Открыть приложение в Safari/WebKit-браузере.
2. Создать цель → нажать «удалить».
3. Наблюдать: запись остаётся, всплывает ошибка парсинга.

**Ожидаемо:** запись удаляется, приходит тост подтверждения.
**Фактически:** ошибка, удаление не происходит.

**Корневая причина.** Роуты DELETE возвращали `204 No Content`, но FastAPI проставлял заголовок `content-type: application/json` при **пустом теле**. Фронтовый `api()` делал `response.json()` всегда, когда content-type содержал `json`. WebKit-парсер на пустой строке падает (Chromium и Firefox молча отдают `{}` — поэтому баг был платформенным).

**Фикс.** Хелпер `api()` переписан: читать тело через `.text()`, парсить через `try { JSON.parse(text) }`, а статус `204` пропускать целиком, не пытаясь парсить.

```javascript
async function api(url, opts = {}) {
    const r = await fetch(url, opts);
    let d = {};
    if (r.status !== 204) {
        const text = await r.text();
        if (text) {
            try { d = JSON.parse(text); } catch { d = {}; }
        }
    }
    if (!r.ok) throw new Error(d.detail || 'Ошибка сервера');
    return d;
}
```

**Почему не поймали раньше.** `pytest` гонялся через `TestClient` (httpx), который не воспроизводит браузерный JSON-парсинг — он отдаёт сырые байты. Дефект живёт исключительно на стыке «реальный браузерный движок ↔ пустое тело 204». Классический случай: зелёные тесты, сломанный продукт.

### BUG-002 · [Major / P1] 500 при удалении обязательства (detached object)

| | |
|---|---|
| Компонент | роут DELETE обязательства |
| Версия фикса | v2.2.0 |

**Как проявлялся.** Удаление обязательства падало с 500 ещё до WebKit-проблемы.

**Корневая причина.** После `session.commit()` SQLAlchemy переводит объект в detached-состояние; код пытался обратиться к его полям/вернуть его в теле ответа.

**Фикс.** Роут возвращает `204` без тела, не трогая detached-объект после коммита.

**Почему не поймали раньше.** Тест проверял happy-path создания, но не удаление с последующим обращением к объекту — типичный пропуск тест-кейса на жизненный цикл сущности.

### BUG-003 · [Major / P1] «Ошибка: Load failed» при импорте CSV ~2 МБ (Safari timeout)

| | |
|---|---|
| Окружение | Safari/WebKit, файл выписки ~5000 строк |
| Компонент | `app/database/crud.py` → `create_transaction` |
| Версия фикса | v2.2.1 |

**Как проявлялся.** Импорт крупной выписки (~2 МБ) обрывался с `Load failed`.

**Корневая причина.** `create_transaction` коммитил **каждую строку отдельно**: `SELECT + INSERT + COMMIT × 5000`. Совокупное время запроса превышало таймаут WebKit на длинный fetch.

**Фикс.** Параметр `autocommit=False` в `create_transaction`, один `commit()` после всего цикла (batch commit). Результат: **5000 строк за 2.9 с**.

**Почему не поймали раньше.** Тестировали на мелких файлах. Деградация была нелинейной по объёму — нужен тест с реалистичным размером данных (perf/load case).

### BUG-004 · [Major / P0] Фиксы не применяются — браузер держит устаревший `app.js`

| | |
|---|---|
| Компонент | `base.html`, подключение статики |
| Версия фикса | v2.2.2 |

**Как проявлялся.** После выкатки фиксов поведение в браузере не менялось — выглядело так, будто исправления не сработали.

**Корневая причина.** Браузер отдавал закэшированный старый `app.js` / `styles.css`. Это маскировало ВСЕ остальные фиксы — самый коварный тип дефекта, потому что бьёт по доверию к самому процессу починки.

**Фикс.** Cache-busting на статике: `app.js?v={{ app_version }}`, `styles.css?v={{ app_version }}`. При каждом бампе версии URL меняется → браузер тянет свежий файл.

**Почему не поймали раньше.** Кэш — не баг кода, а баг доставки. Не ловится unit-тестами в принципе; ловится только дисциплиной «hard refresh при проверке» или, лучше, архитектурно — через версионирование ассетов.

### BUG-005 · [Major / P1] AssertionError на 204-роутах на части версий FastAPI

| | |
|---|---|
| Компонент | роуты DELETE goals / obligations / liquid-assets |
| Версия фикса | v2.2.2 |

**Как проявлялся.** В venv пользователя приложение ассертило `Status code 204 must not have a response body`.

**Корневая причина.** Декоратор роута с `status_code=204` совместно с `response_model` — на некоторых версиях FastAPI это вызывает внутренний assert.

**Фикс.** Убрать `status_code` из декоратора, возвращать явно `Response(status_code=204)` из тела функции.

**Почему не поймали раньше.** Версия FastAPI в sandbox-окружении и в venv пользователя различались — environment drift. Ловится pinned-зависимостями + матрицей версий в CI.

### BUG-007 · [Major / P1] Падение pytest из-за окружения

**Как проявлялся.** Прогон `pytest` падал у пользователя локально (проблемы с путями/окружением).

**Принцип.** Тяжёлые/редкие зависимости — лениво. Отсутствие опциональной фичи не должно ронять ядро.

### BUG-009 · [Critical / P0] Письмо подтверждения email не приходит в production без SMTP

| | |
|---|---|
| Компонент | `app/api/routes_auth.py` |
| Окружение | self-hosted, `ENVIRONMENT=production`, SMTP не настроен |

**Как проявлялся.** Пользователь регистрируется или жмёт «отправить письмо» → видит «отправлено», но письмо не приходит, и подтвердить email невозможно → онбординг заблокирован.

**Корневая причина.** Dev-ссылка для подтверждения отдавалась в ответе только при условии `not is_production`. На self-hosted без почтового сервера среда `production`, SMTP выключен → ссылка скрыта, а письма физически нет. Два «нормальных» условия пересеклись в дыру.

**Фикс.** Отдавать `verification_url` всегда, когда `email_enabled = False`, не привязываясь к `is_production`:
```python
dev_link = verify_url if not settings.email_enabled else None  # без SMTP отдаём ссылку (self-hosted)
```

**Почему не поймали раньше.** Тестировали в dev-режиме, где ссылка и так показывалась. Прод-сценарий без SMTP не входил в матрицу окружений. Ловится тестом на конфигурацию `production + email_disabled`.

### BUG-010 · [Critical / P0] Аккаунты пропадают после каждого redeploy

| | |
|---|---|
| Компонент | `docker-compose.yml` |

**Как проявлялся.** После пересборки/редеплоя все аккаунты исчезали — база чистая.

**Корневая причина.** Имя Docker-тома по умолчанию выводится из имени папки проекта. Деплой из новой папки (`finpilot-new`) → это другой Docker-проект → другой том → чистая база. Данные оставались в томе старой папки. Потеря пользовательских данных — максимально тяжёлый класс.

**Фикс.** Явно зафиксировать имя тома — теперь данные переживают redeploy в любой папке:
```yaml
volumes:
  pgdata:
    name: finpilot_pgdata
```

**Почему не поймали раньше.** Инфраструктурный дефект, не проявляется в коде приложения. Ловится только тестом полного цикла deploy → данные → re-deploy → данные на месте.

### BUG-011 · [Major / P1] Ставка обязательства показана как `0.085%` вместо `8.5%`

**Как проявлялся.** В карточке обязательства ставка 8.5% отображалась как `0.085%`.
**Корневая причина.** Внутри ставка хранится как доля (`0.085`), а на экран выводилась без нормализации доля→процент.
**Фикс.** Нормализация при отображении: `0.085` → `8.5%`.
**Почему критично для финпродукта.** Неверное число на экране в денежном приложении подрывает доверие сильнее, чем падение. Severity Major именно из-за домена. Ловится тестом форматтера на граничных значениях (0.085, 0.3, 1.0, 30).

### BUG-013 · [Major / P1] При отрицательном Rt — бесполезное «денег нет»

**Как проявлялся.** Когда доступный ресурс Rt < 0, система выдавала плоское «денег нет» вместо полезного разбора.
**Корневая причина.** Не обрабатывался сценарий отрицательного ресурса при наличии накоплений.
**Фикс.** Считать буфер: на сколько месяцев накоплений (подушка + ликвидные) хватит покрывать дефицит, с пометкой «это временный буфер». Проверено на кейсе Михаила: `Rt = −3000`, накоплено `120 000` → «хватит на 40 мес, но временно», вместо «денег нет».
**Класс дефекта.** UX/логический: алгоритм давал технически верный, но бесполезный ответ там, где мог дать ценный. Ловится edge-кейсом `Rt < 0` в тестах сценариев.

### BUG-017 · [Major / P1] Тесты `test_calculate_all_profiles` падают — устаревший контракт API

| | |
|---|---|
| Компонент | `tests/test_api_planning.py` |
| Версия фикса | №3 |

**Как проявлялся.** Тест падал: проверял ключи `recommendation`/`alternatives`, которых в ответе нет.

**Корневая причина.** Реальный ответ `POST /api/planning/calculate` отдаёт `best` / `ranked` / `top3` / `input_summary`. Тест отстал от фактического контракта — рассинхрон теста и API.

**Фикс.** Assertions переписаны под реальный контракт:
```python
body = resp.json()
assert "best" in body
assert "ranked" in body and len(body["ranked"]) > 0
assert "top3" in body
assert "input_summary" in body
```

**Почему опасно.** Тест, проверяющий несуществующие ключи, — это false negative: он либо всегда красный (и его глушат), либо был зелёным по случайности. Ловится контрактными/snapshot-тестами ответов API.

### BUG-018 · [Major / P1] Отрицательный `target_amount` принимается без валидации — **OPEN**

| | |
|---|---|
| Компонент | Pydantic-схема создания цели (`GoalCreate`) |
| Статус | **Обнаружен. Валидацию рекомендуется добавить.** |

**Как проявлялся.** `POST /api/goals` с `target_amount = -100` → запрос **принимается**, цель с отрицательной целевой суммой создаётся.

**Шаги воспроизведения.**
1. `POST /api/goals` с телом `{"name":"x","target_amount":-100,"current_amount":0,"deadline":"2027-01-01T00:00:00","category":"material"}`.
2. Ответ — успех, валидации нет.

**Ожидаемо:** `422 Unprocessable Entity`.
**Фактически:** цель создана.

**Корневая причина.** В схеме отсутствует ограничение `gt=0` / `ge=0` на сумму.

**Рекомендуемый фикс.**
```python
from pydantic import BaseModel, Field

class GoalCreate(BaseModel):
    name: str = Field(min_length=1, max_length=200)
    target_amount: float = Field(gt=0)
    current_amount: float = Field(ge=0)
    # ... + проверка current_amount <= target_amount
```
Та же дыра вероятна и в других денежных полях (доход, расход, платёж, ставка, баланс) — см. раздел 4, паттерн «нет валидации входа».

**Почему не поймали раньше.** Нет негативных тест-кейсов на границы. В этой сессии дефект был зафиксирован при ревизии контракта, но в схему ограничение пока не внесено.

## 3. Как QA тестирует это в продакшене (методология)

### 3.1. Severity ≠ Priority

Две независимые оси, их постоянно путают:

- **Severity** — насколько сильно ломается продукт технически. Падение оплаты = Critical, опечатка = Trivial.
- **Priority** — как срочно чинить с точки зрения бизнеса. Опечатка в названии компании на лендинге = Trivial по severity, но P0 по приоритету.

Пример из этого реестра: BUG-015 (фавикон) — Trivial severity, P3. BUG-010 (потеря аккаунтов) — Critical severity, P0. BUG-004 (кэш) — «всего лишь» Major по severity, но P0, потому что блокировал проверку всех остальных фиксов.

| Severity | Что значит | Примеры из реестра |
|----------|-----------|--------------------|
| Critical | Потеря данных, неработающая ключевая функция, блок онбординга | BUG-001, 009, 010 |
| Major | Важная функция работает неверно/частично | BUG-002, 003, 011, 013 |
| Minor | Неудобно, но есть обходной путь | BUG-006, 012, 016 |
| Trivial | Косметика | BUG-015 |

### 3.2. Жизненный цикл бага (Bug Lifecycle)

```
New → Triage (severity/priority) → Assigned → In Progress
    → Fixed → Verified (QA перепроверил) → Closed
                   �‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾↗ Reopened (если не починилось)
```

Ключевой шаг, который чаще всего пропускают соло-разработчики — **Verified**: баг не закрывается словами «я починил», он закрывается воспроизведением исходных шагов и подтверждением, что симптом исчез. В этом проекте роль Verified исполняет ручной прогон пользователя в реальном браузере после доставки — это правильно.

### 3.3. Типы тестирования и какой поймал бы какой баг

| Тип тестирования | Что проверяет | Поймал бы |
|------------------|---------------|-----------|
| Unit | Отдельная функция/формула | расчёт буфера (BUG-013), форматтер ставки (BUG-011) |
| Integration / API | Контракт эндпоинта, статусы, тело | контракт `/calculate` (BUG-017), валидация цели (BUG-018) |
| Contract / snapshot | Ключи и форма ответа не «уехали» | BUG-017 |
| Cross-browser (E2E) | Реальный движок браузера | BUG-001 (WebKit 204), BUG-003 (Safari timeout) |
| Performance / load | Поведение на больших данных | BUG-003 (5000 строк) |
| Visual regression | Вёрстка, темы | BUG-006, BUG-012 |
| Config / environment | Поведение в prod-окружении | BUG-005, BUG-009 |
| Infra / deploy | Полный цикл деплоя и сохранность данных | BUG-010 |

**Главный вывод по процессу:** большинство дефектов FINPILOT жили там, где `pytest + TestClient` слеп по определению — в браузере, в окружении, в инфраструктуре. Зелёные unit-тесты создавали ложное чувство покрытия. Это нормальная ловушка; лечится не «больше unit-тестов», а **сдвигом тестирования к границам системы** (E2E на реальном движке + проверка в prod-like окружении).

## 4. QA-аудит FINPILOT: системные слабые места

Если перестать смотреть на баги по одному и сгруппировать по корневой причине — видно пять системных паттернов. Это и есть карта рисков продукта.

### Паттерн A · WebKit/Safari — главное слепое пятно
**Дефекты:** BUG-001, BUG-003 (+ за пределами этих чатов — поломка drawio-парсера на тех же WebKit-причинах).
**Суть:** разработка и автотесты идут на Chromium/httpx, а пользователь сидит на WebKit. Всё, что WebKit обрабатывает строже (пустое тело 204, длинные fetch, парсинг), — потенциальная мина.
**Что делать:** smoke-набор E2E на реальном WebKit (Playwright с webkit-движком) на каждый релиз. Минимум: удаление сущностей, импорт большого CSV, расчёт плана.

### Паттерн B · Нет валидации входных данных
**Дефекты:** BUG-018 (отрицательный target_amount). Высокая вероятность той же дыры в income/expense/payment/rate/balance.
**Суть:** Pydantic-схемы принимают то, что не имеет финансового смысла (отрицательные суммы, ставка > 100%, current > target).
**Что делать:** ревизия всех Create/Update-схем, проставить `Field(gt=0 / ge=0)`, кросс-полевые валидаторы (`current_amount <= target_amount`), верхние границы. Негативные тест-кейсы на каждое числовое поле.

### Паттерн C · Граничные случаи финансовой логики
**Дефекты:** BUG-013 (Rt < 0). Сюда же — известный по UX-стандарту 500 на пустом профиле (ST-04), риск деления на ноль в коэффициентах при нулевых расходах/доходе.
**Суть:** ядро считает корректно на «нормальных» данных, но ломается/бесполезно на краях.
**Что делать:** параметризованные тесты на Rt=0, Rt<0, пустой профиль, один кредит, доход=0, расход=0, 100+ целей.

### Паттерн D · Окружение и инфраструктура
**Дефекты:** BUG-005 (version drift FastAPI), BUG-009 (prod без SMTP), BUG-010 (Docker volume), BUG-004 (кэш).
**Суть:** «работает у меня» ≠ «работает в проде». Среды расходятся по версиям, конфигам, путям, кэшу.
**Что делать:** pinned-зависимости, тест на конфигурацию `production + email_disabled`, тест полного цикла redeploy с проверкой сохранности данных, версионирование статики (уже сделано).

### Паттерн E · Представление данных ≠ внутренняя модель
**Дефекты:** BUG-011 (доля 0.085 vs процент 8.5%).
**Суть:** внутри — доли/копейки/секунды, на экране — проценты/рубли/даты. Каждая граница преобразования — место для ошибки на ×100.
**Что делать:** изолировать форматтеры в один модуль, покрыть unit-тестами на граничных значениях. Для денег — задуматься о `Decimal` (уже в планах v3.0.0 MAJOR).

## 5. Регрессионный smoke-чеклист

Минимальный прогон перед каждым релизом. Гонять на **реальном WebKit-браузере**, не только в sandbox. Это прямая дистилляция найденных багов — каждый пункт закрывает конкретную исторযoric-мину.

**Аутентификация и онбординг**
1. Регистрация нового аккаунта → приходит/показывается ссылка подтверждения (в т.ч. в prod-конфиге без SMTP) — *BUG-009*.
2. Подтверждение email по ссылке → статус «подтверждён».
3. Логин/логаут, перезаход.

**CRUD сущностей (на WebKit!)**
4. Создать и **удалить** цель — *BUG-001*.
5. Создать и **удалить** обязательство — *BUG-001, BUG-002*.
6. Создать и **удалить** ликвидный актив — *BUG-001*.
7. «Вернуть» (undo) после удаления каждой сущности.

**Импорт**
8. Импорт CSV ~5000 строк завершается без таймаута (целевое < 3 с) — *BUG-003*.
9. Импорт битого/пустого CSV → внятная ошибка, не 500.

**Расчёт и отображение**
10. Расчёт плана на всех 5 профилях риска → ответ содержит `best/ranked/top3/input_summary` — *BUG-017*.
11. Ставка обязательства отображается в процентах корректно (8.5%, не 0.085%) — *BUG-011*.
12. Профиль с Rt < 0 при наличии накоплений → расчёт буфера, не «денег нет» — *BUG-013*.
13. Пустой профиль (нет данных) → не 500, а понятное пустое состояние.

**Инфраструктура / визуал**
14. Светлая и тёмная тема — карточки читаемы — *BUG-012*.
15. После выкатки фиксов браузер тянет свежий `app.js` (проверить версию ассета) — *BUG-004*.
16. Redeploy Docker → аккаунты и данные на месте — *BUG-010*.

## 6. Каталог тест-кейсов по областям

Расширенный набор для постепенной автоматизации. Формат: **область → что проверяем**.

**Валидация входа (негативные кейсы — приоритет, закрывает паттерн B)**
- target_amount: отрицательный, ноль, гигантский, не-число → 422.
- current_amount: отрицательный → 422; current > target → 422.
- Доход/расход/платёж: отрицательные → 422.
- Ставка: < 0 и > 1 (или > 100%) → 422.
- Дедлайн цели: в прошлом → предупреждение/422.
- Имя цели: пустое, 10 000 символов, HTML/`<script>` (XSS) → экранируется.

**Граничные значения финансовой логики (паттерн C)**
- Rt = 0, Rt < 0 (с накоплениями и без).
- Доход = 0, расход = 0 (деление на ноль в Lt/Dt).
- Пустой профиль: 0 транзакций, 0 целей, 0 обязательств.
- 1 кредит / 100+ целей / 0 кредитов.

**Корректность алгоритма (инварианты — должны держаться ВСЕГДА)**
- Сумма долей любой альтернативы = 100% (xd+xr+xg).
- U-score ∈ [0, 1] для каждой альтернативы.
- После применения плана: Dt(a) ≤ 0.40, Lt(a) ≥ L_min.
- Avalanche: все кредиты дешевле r_bench → xd уходит в цели (x_d_eff = 0).
- Avalanche: кредит ровно на пороге r_bench — граничное поведение определено.
- Веса профиля соответствуют выбранному риску (Conservative … Aggressive).

**Импорт CSV**
- Банки: Тинькофф, Сбер, Альфа, ВТБ, Райффайзен — каждый формат.
- Неизвестный банк / Universal-парсер.
- Дубликаты строк, не-UTF8 кодировка, пустой файл, файл 10 МБ.

**API-контракт (паттерн, закрывает BUG-017)**
- Snapshot ключей ответа `/calculate`, `/goals`, `/obligations`.
- Все статусы: 200/201/204/401/403/404/422.

**Cross-browser E2E (паттерн A — наивысший ROI)**
- Полный happy-path на WebKit, Chrome, Firefox.

**Безопасность (актуально после auth в v3.0.0)**
- Доступ к чужому user_id → 403.
- Истёкший/битый JWT → 401.
- Регистрация с существующим email → понятная ошибка.

**Идемпотентность**
- Двойной сабмит формы создания → не два дубля (известный риск FRM-05 из UX-стандарта).

## 7. Что автоматизировать в первую очередь

Порядок по соотношению «риск × частота поломки / стоимость написания»:

1. **Негативная валидация входа** (паттерн B). Дёшево писать (parametrize по полям и невалидным значениям), закрывает целый класс. Старт — `target_amount` из BUG-018.
2. **Инварианты алгоритма** (сумма долей = 100%, U ∈ [0,1], Dt ≤ 0.40). Ядро продукта — здесь регрессия дороже всего, а проверки простые.
3. **Контрактные/snapshot-тесты ответов API** (BUG-017). Дёшево, ловит «уехавший» контракт автоматически.
4. **E2E smoke на реальном WebKit** (паттерн A). Дороже в настройке (Playwright + webkit), но именно тут жили Critical-баги. Минимальный набор из 3 сценариев: удаление сущности, импорт большого CSV, расчёт плана.
5. **Тест полного цикла redeploy** (BUG-010). Один раз настроить — навсегда закрыть риск потери данных.

**Принцип приоритизации:** автоматизируй сначала то, что (а) уже один раз сломалось и (б) ломает ключевой сценарий. Пункты 1–3 можно поднять за один заход в `pytest`; пункты 4–5 — отдельная инфраструктурная задача, но с наивысшей отдачей по Critical-рискам.
