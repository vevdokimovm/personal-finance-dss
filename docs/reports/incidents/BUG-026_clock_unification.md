### BUG-026 · [Minor / P0] datetime.now() (локальное время сервера) вместо UTC в «сейчас»-логике

| Поле | Значение |
|---|---|
| Окружение | Linux/Docker, PostgreSQL (prod) / SQLite (test), v4.25.2 → фикс в v4.25.3 |
| Компонент | `app/services/spending.py`, `app/services/statement_parser.py`, `app/services/bank_api.py`, `app/schemas/obligation.py`, `app/database/crud.py` |
| Статус | Closed (v4.25.3) |
| Найден | код-ревью (до проявления на проде) |

**Как проявлялся:**
На UTC-окружении (песочница, CI) — никак: дефект невидим, потому что локальное время процесса
совпадает с UTC. На прод-VPS в РФ (типично MSK, UTC+3) появился бы стабильный сдвиг в 3 часа между
«сейчас»-логикой приложения и временем, записанным в БД (которое везде UTC). Наблюдаемые следствия:

- Месячная сводка трат на стыке месяца относит свежие операции не в тот месяц (в первые 3 часа суток
  1-го числа по MSK локальные часы уже в новом месяце, а UTC-дата операции — ещё в старом).
- Окна «расходы за последние N дней» сдвинуты на оффсет сервера.
- Импортированные без даты операции получают fallback-дату в локальном времени, записанную в
  naive-UTC колонку (рассинхрон с остальными датами).
- «Сколько месяцев уже выплачивается» по обязательству считается от локального «сейчас».

**Шаги воспроизведения:**
1. Выставить часовой пояс процесса на не-UTC (например, MSK / UTC+3).
2. Создать расход с UTC-датой, попадающей в «хвост месяца» по UTC, но уже «следующий месяц» по локали.
3. Запросить сводку трат → операция бакетится не в свой UTC-месяц (current_period расходится с датой).

**Ожидаемо / фактически:**
- Ожидалось: вся «сейчас»-логика в одной системе координат с хранилищем (UTC).
- Фактически: 13 вызовов `datetime.now()` возвращали naive-локаль, остальной код и колонки — naive-UTC.

**Корневая причина:**
`datetime.now()` без аргумента возвращает локальное время процесса (naive, в зоне сервера). Релиз
v4.25.2 заменил deprecated `datetime.utcnow()` единым helper `utcnow()` (naive UTC), но прямые
`datetime.now()` остались вне этой замены. На UTC-CI локаль == UTC, поэтому ни тесты, ни ревью раньше
не различали их.

**Фикс:**
13 вызовов `datetime.now()` → `app/utils/time.py::utcnow()` в 5 файлах. Контракт замены — naive↔naive
(helper отдаёт naive UTC), поэтому риск сравнений naive↔aware (TypeError) не вносится. Намеренно
**не** тронуты aware-вызовы `datetime.now(timezone.utc)` в `cbr_rate.py`/`cbr_fx.py`/`security.py` —
там корректный UTC. Костыль `obligation.py` `start_date.replace(tzinfo=None)` **оставлен**: он гасит
aware-значение, протекающее из Pydantic (ISO-строка с оффсетом на входе) — это отдельная задача, не
относящаяся к унификации «сейчас», и снимется только при будущем переходе на aware-везде (вариант B),
если он вообще понадобится. Осиротевший импорт `datetime` убран из `bank_api.py`.

Полная корректность таймзон (конвертация хранимого UTC в зону пользователя перед извлечением
месяца/дня/дня-недели для слоя внутримесячных паттернов) — **отдельный заход** в слое агрегации; там
же заводится поле `tz` у User (сейчас его нет). Тип колонок (`timezone=True`) этого не решает — это
вопрос зоны на чтении, а не метаданных хранилища.

**Почему не поймали раньше:** ← самое важное
Ни один тест не эмулировал не-UTC окружение — на UTC-CI локальное и UTC «сейчас» неотличимы, поэтому
весь класс таких дефектов был слеп. Закрыто `tests/test_clock_unification.py` (7 тестов): мокаем
`utcnow` фиксированным «невозможным» моментом и проверяем, что результат функции привязан к нему, а не
к системным часам. Любой новый `datetime.now()` в этих путях теперь падает на этих тестах. Контракт
«всё „сейчас“ — через `utcnow()`; прямой `datetime.now()` в `app/` запрещён (кроме явных aware-точек)»
зафиксирован в WATCHLOG §4.
