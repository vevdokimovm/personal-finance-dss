# INC-CRUD-DELETE — удаление/восстановление obligations/goals/transactions

> Полный разбор (post-mortem) по правилу §12 WATCHLOG. Живёт ВНЕ репозитория (у владельца +
> в Claude-проекте). Краткая сводка — строкой в `docs/incidents_summary.md`.

- **Категория:** функциональный дефект + дыра в покрытии тестами.
- **Версия закрытия:** v4.16.28 (6 CRUD-E2E).
- **Связанные правила:** §13 (E2E добавляются вместе с unit; базовые CRUD ОБЯЗАНЫ иметь E2E).

---

## 1. Сводка

Базовый сценарий удаления кредитного обязательства через UI был сломан на протяжении нескольких
версий, при этом юнит-тесты на соответствующие функции оставались зелёными. Отдельно — функция
восстановления (undo/restore) была добавлена недавно и тоже не имела сквозной проверки через
браузер. Корень — структурный: юнит-тест проверяет функцию в изоляции, но НЕ проверяет, что
пользователь реально может выполнить действие через интерфейс, то есть не ловит разрыв между UI
(кнопка, её `data-*`-атрибут, JS-обработчик) и API (эндпоинт, формат запроса). Закрыто шестью
браузерными CRUD-E2E (создание/удаление/восстановление для obligations, goals, transactions),
добавленными в v4.16.28.

---

## 2. Хронология

1. В ходе работ обнаружено, что удаление обязательства через UI не отрабатывает, хотя юнит-тесты
   на удаление зелёные несколько версий подряд.
2. Восстановление (restore) добавлено недавно как отдельный путь — но без сквозной E2E-проверки.
3. Диагностирована природа разрыва: дефект живёт «между слоями» (UI ↔ API), куда юнит не дотягивается.
4. Написаны 6 E2E через реальный браузер на полный жизненный цикл каждой сущности.
5. Зафиксировано правило §13: базовые CRUD-флоу обязаны иметь E2E.

---

## 3. Что именно закрыто (6 CRUD-E2E, `tests/e2e/test_crud_e2e.py`)

| Тест | Проверяет |
|---|---|
| `test_obligation_create` | создание обязательства появляется в UI |
| `test_obligation_delete_and_restore` | удаление убирает элемент; restore (undo) возвращает |
| `test_goal_create` | создание цели |
| `test_goal_delete_and_restore` | удаление + восстановление цели |
| `test_transaction_create` | создание транзакции |
| `test_transaction_delete_and_restore` | удаление + восстановление транзакции |

Элемент находится по `data-<entity>-id` из ответа создания; удаление ждёт `state="detached"`.

**Нюанс restore (важен для понимания тестов):** у obligations и goals восстановление — это POST на
базовый `/api/<entity>` со снапшотом в теле (сущность ПЕРЕсоздаётся, id может смениться); у
transactions — `POST /api/transactions/<id>/restore` (тот же id). Поэтому E2E проверяет возврат
по имени сущности, а не по совпадению id.

---

## 4. Корневая причина (5 × «почему»)

1. **Почему** удаление обязательства было сломано незаметно? → Не было сквозной проверки через UI.
2. **Почему** юнит-тесты не поймали? → Юнит проверяет функцию удаления в изоляции, а ломался путь
   UI → API (атрибут кнопки / формат запроса / обработчик), которого юнит не касается.
3. **Почему** этого пути не было в тестах? → E2E писались точечно (dashboard/planning/auth), а
   базовые CRUD-операции не были включены в обязательный E2E-набор.
4. **Почему** не были включены? → Не было правила, что CRUD обязан иметь E2E; считалось, что юнит
   + integration достаточно.
5. **Почему** считалось достаточно? → Недооценён класс багов «разрыв между UI и API», который виден
   только в реальном браузере. → **Структурная дыра в стратегии тестирования.**

---

## 5. Resolution

- 6 браузерных CRUD-E2E на полный жизненный цикл трёх сущностей (v4.16.28); E2E на chromium = 20.
- Правило §13 WATCHLOG: для любого изменения, затрагивающего фронт или API-эндпоинт, — сразу E2E;
  базовые CRUD-флоу обязаны иметь E2E.
- Бэк восстановления (симметричный soft-delete + restore) закрыт ранее в v4.16.18 (миграция 0018,
  Obligation/Goal/LiquidAsset; транзакции — с BUG-03). UI-модалки undo — в дизайн-блоке.

---

## 6. Профилактика и уроки

- **Урок.** Зелёные юнит-тесты не доказывают, что фича работает для пользователя. Класс багов
  «UI ↔ API» ловится только сквозным прогоном в браузере. Поэтому базовые пользовательские действия
  (создать/удалить/восстановить ключевые сущности) — это E2E-обязанность, а не «по желанию».
- **Профилактика.** §13 встроено в чек-лист: трогаешь фронт/эндпоинт — добавляешь E2E в том же
  батче. CRUD-E2E теперь регрессионный гард: повторно сломать удаление незаметно нельзя.
- **Связь со стратегией.** Это часть общей трёхуровневой системы (`docs/testing_infrastructure.md`):
  unit ловит логику, E2E ловит интеграцию UI↔API, property ловит математику. Каждый слой закрывает
  свой класс ошибок; выпадение любого оставляет слепую зону — как здесь выпал E2E на CRUD.
