# Incident Report — ложная тревога «баг математической модели», оказавшаяся гонкой в E2E

| | |
|---|---|
| **ID** | INC-MATHMODEL-FALSEALARM |
| **Дата** | 2026-06-25 |
| **Версия** | выявлено на v4.16.24 (ветка docs/QA); затронутый тест существовал с ранних версий |
| **Severity** | Low (тест, не прод) — но с высокой ценой ошибки при неверном выводе |
| **Тип** | False-positive диагностика / гонка в автотесте |
| **Статус** | Закрыт — корень устранён, ядро подтверждено чистым |

Разбор blameless. Главная ценность инцидента — не в самом дефекте теста, а в **методе**:
как подозрение «сломана математика» было доведено до настоящего корня, а не списано на
«флак» или «чужой баг».

---

## 1. Сводка

При добавлении E2E-тестов для auth/guest всплыл `test_profiles_produce_different_plans`
(зона planning E2E), который **стабильно падал**: консервативный (`risk=1`) и агрессивный
(`risk=5`) профили давали идентичный план «Всё в резерв» вместо разных. В журнале он
числился как «флак на перепрогоне зелёный».

Первая, пугающая гипотеза: **баг в математическом ядре** — ранжирование (SAW) не различает
риск-профили. Для финансового продукта это критично: профиль риска — одна из центральных осей
рекомендации.

Расследование показало: **ядро и UI чисты**, а тест падал из-за **гонки в самом себе**.
Страница `/planning` при загрузке делает автоматический расчёт с дефолтным `risk=3`, а тест
читал результат этого авто-расчёта вместо своего ручного (с выбранным профилем).

---

## 2. Воздействие

- Прод-воздействия нет: дефект жил только в E2E-тесте.
- Риск **неверного вывода** был высоким: если бы «флак» оставили как есть или поспешно
  «починили» правкой ядра — могли внести реальный баг в математику ради зелёного теста, либо
  надолго оставить ложную запись «возможно, ядро не дифференцирует профили».
- Скрытый процессный долг: тест, помеченный «флаком», маскировал детерминированную гонку.

---

## 3. Хронология

1. Полный E2E-прогон на ветке → `test_profiles` падает (оба профиля «в резерв»).
2. Изолированный перепрогон → падает снова (дважды) ⇒ это **не флак**, а детерминированный фейл.
3. Гипотеза A (баг теста/данных) vs B (баг ядра). Проверка демо-данных: у кейса anna есть и
   долги, и цели — данных для расхождения профилей достаточно, значит «нечего распределять» как
   объяснение отпало.
4. **Прямой вызов API в обход UI** (тот же payload, что шлёт фронт): `risk=5,l_min=0,r_bench=0.14`
   → «Всё на цели» (x_goals=39500); `risk=1` → «Всё в резерв». **Профили различаются — ядро ✓.**
   Гипотеза B отвергнута.
5. Чтение `app.js`: submit формы шлёт `risk_tolerance = planRisk`, где `planRisk` берётся из
   кликнутой кнопки. **UI ✓.**
6. Первый фикс (ожидание инициализации + подтверждение `.active`) — тест всё равно падает.
7. **Перехват реального POST из браузера** (`page.on("request")`) → видно **ДВА** запроса:
   `{"risk_tolerance":3,...}` (авто-расчёт при загрузке) и `{"risk_tolerance":5,...}` (ручной).
   Корень найден: тест читал результат авто-расчёта.
8. Финальный фикс — `expect_response` на ручной POST + ожидание рендера. Тест зелёный;
   полный E2E — 14 passed.

---

## 4. Корневая причина

**Детерминированная гонка чтения в E2E-тесте**, замаскированная под «флак»:

- `/planning` при загрузке инициирует автоматический расчёт с дефолтным `risk=3` (UX-решение —
  показать план сразу).
- Старый `_calculate` дожидался лишь появления результата в `#best-container` (текст «оценка») и
  читал его — но первым контейнер заполнял **авто-расчёт** (`risk=3` → «в резерв»), а не ручной.
- Для `risk=1` дефект невидим: и авто (`risk=3`), и ручной (`risk=1`) дают «в резерв».
- Для `risk=5` тест читал чужой план («в резерв») вместо «на цели» → ложное «профили не
  различаются».
- «Иногда зелёный» (отсюда ярлык «флак») — когда тайминг случайно давал прочитать ручной
  результат.

Ядро (SAW-ранжирование, веса профилей) и UI (`app.js`) к дефекту отношения не имеют.

---

## 5. Resolution

В `tests/e2e/test_planning_e2e.py::_calculate`: дождаться завершения авто-расчёта
(`networkidle`), выбрать профиль, затем дождаться ответа **именно на ручной** POST
(`page.expect_response("**/api/planning/calculate")`) и применения рендера. Это убирает гонку
авто-vs-ручной детерминированно. Прод-код (`app/`) не менялся.

---

## 6. Профилактика

1. **Доказательство до корня перед выводом** (закреплено в памяти проекта и в `WATCHLOG`):
   гипотезу «сломано ядро» подтверждать/опровергать прямым вызовом нижнего слоя (API в обход UI),
   а не догадками по верхнему.
2. **«Флак» — это диагноз-заглушка, а не объяснение.** Тест, помеченный флаком, должен быть
   доведён до детерминированной причины; до тех пор он маскирует реальный дефект.
3. **Учитывать авто-расчёт `/planning`** в E2E (зафиксировано в `WATCHLOG` §4): на странице
   всегда есть фоновый POST `risk=3` при загрузке — ждать ответ именно на свой расчёт.
4. Не «чинить» верхнеуровневый симптом правкой ядра без доказательства, что баг в ядре.

---

## 7. Уроки

- Самый громкий симптом («профили не различаются» ⇒ «математика сломана») часто имеет тихую
  причину в слое автоматизации. Цена неверного вывода здесь была бы максимальной — правка ядра
  ради зелёного теста.
- Опровержение гипотезы на **нижнем слое** (API) мгновенно сняло подозрение с математики и
  сфокусировало поиск на тесте.
- Перехват фактического сетевого трафика (`page.on("request")`) — решающий инструмент: он
  показал два POST там, где по верхнему слою казался один.
- Метод важнее частного фикса: до конца выяснить, перепробовать все уровни проверки, потом
  делать вывод.
