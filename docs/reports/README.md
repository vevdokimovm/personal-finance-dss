# docs/reports/ — отчёты проекта

> Единый диспетчер «что писать когда и куда» — **`reporting_policy.md`**. Конвенция имён — `../naming_convention.md`.

Единый дом для всех типов отчётов. Каждый вид — в своей папке.

## Структура
- **adr/** — Architecture Decision Records (принятые архитектурные решения).
- **incidents/** — инциденты: дефекты продукта с влиянием на пользователя, post-mortem'ы.
- **investigations/** — расследования: неочевидная причина без дефекта в коде.
- **merges/** — merge-манифесты и отчёты о слияниях параллельных веток + гайд по форку/мёржу.

## Реестры и методология (корень reports/)
- `incidents_summary.md`, `investigations_summary.md` — сводные реестры.
- `incident_postmortem_guide.md`, `investigation_report_guide.md` — как писать отчёты.
- `bug_report_template.md`, `report_types.md` — шаблон баг-репорта и таксономия отчётов.

## Таксономия (кратко)
Инцидент = дефект с impact на юзера. Расследование = разобрались в неочевидной причине,
дефекта в коде нет. Bug card = найден новый дефект в коде. ADR = зафиксированное
архитектурное решение.

## Автогенерация (веха 5)
Отчёты создаются **автоматически по триггеру**, без отдельной просьбы, и коммитятся в репозиторий
(живут с кодом, мёржатся, видны в истории):
- дефект в продуктовом коде → инцидент в `incidents/` + строка в `incidents_summary.md`;
- искали причину, дефекта в коде нет → расследование в `investigations/` + строка в `investigations_summary.md`;
- принято архитектурное решение → ADR в `adr/` (следующий свободный номер, без пропусков);
- слияние разошедшихся веток → манифест в `merges/`;
- повторная ошибка воркфлоу → строка в `docs/pitfalls.md` (не сюда);
- найден баг → карточка по `bug_report_template.md`.

Имя файла — по конвенции проекта: английский snake_case, без префикса `FINPILOT_`
(например `crud_delete_incident.md`, `browsers_sandbox_investigation.md`).
