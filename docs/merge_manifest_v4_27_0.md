# Merge-манифест линии v4.27.0 — инвалидация сессий при смене/сбросе пароля

> **Назначение.** Прикладная карта ОДНОЙ линии (`v4.27.0`, вахта J) для слияния с параллельными
> наработками других вахт (V/M/S). Общая методология — `docs/merge_and_fork_guide.md`; этот файл
> инстанцирует её процедуру (§4) и гейты (§5) под конкретный батч. Цель — чтобы мёржер за минуты понял,
> что эта линия трогает, где риск конфликта и какие инварианты нельзя сломать, не читая весь диф вслепую.

- **База линии:** `v4.26.0` (консолидированная — слияние трёх линий JWT-ревокации M/J/S).
- **Задача:** раздел 4.4, follow-up к JWT-ревокации — смена/сброс пароля гасит все сессии.
- **Тип:** MINOR (новое security-поведение). Миграций нет.
- **Дизайн-решение:** **Option A** (банковская планка) — гасим ВСЕ сессии, включая текущую. Решение
  Василия, обосновано в §4 ниже. **Не переоткрывать на Option B при слиянии.**

## 3. Зоны риска конфликта с другими вахтами

| Зона | Риск | Как разрешать |
|------|------|---------------|
| `routes_auth.py` :: `change_password` / `reset_password` | **Высокий, семантический.** Если другая вахта тоже взяла «инвалидацию при смене пароля» — обе добавили `bump`. Дубль `bump` безвреден функционально, но грязно. | Взять ОДНУ реализацию (см. §6 — критерии). НЕ склеивать два `bump` подряд. Сверить, что выбранная делает Option A (bump + точечный jti + cookie на change; bump на reset). |
| `routes_auth.py` :: импорты / другие эндпоинты | Средний, текстовый. Другая вахта могла менять соседние ручки (`logout`, `login`). | Обычный текстовый мёрдж по ручкам; импорты — объединить множества, дублей не плодить. |
| `bump_tokens_valid_since` (в `revocation.py`) | **Критический, если расходится.** Если другая вахта изменила округление рубежа (микросекунды вместо `.replace(microsecond=0)`) — меняется контракт re-login и same-second. | Сохранить округление до секунды (см. §4 инвариант I2). Если линии разошлись — это не «выбор по вкусу», а решение по контракту: округление нужно для немедленного re-login после reset/logout-all. |
| `docs/WATCHLOG.md` §3 | Гарантированный текстовый конфликт — все вахты добавляют версии. | Слить списки версий, оставить **ровно 10** свежих, порядок по убыванию; дубли версий схлопнуть. |
| `CHANGELOG.md` / `ROADMAP.md` | Append-конфликты. | Объединить записи по версиям, историю НЕ перекраивать (гайд §7). |
| Счётчик тестов | Число «N passed» у каждой линии своё. | **Пересчитать после объединения** тест-файлов (гайд §5), не списывать. |

## 4. Инварианты-контракты — НЕ сломать при слиянии

- **I1. ADR-002 (naive↔naive UTC).** Всё «сейчас» — через `utcnow()` (naive). JWT-таймстампы (`iat`/`exp`)
  нормализуются `revocation.epoch_to_naive_utc`. `routes_auth` таймстампы сам не конвертирует — отдаёт
  сырой `exp` в `revoke_token`. Не вводить inline naive/aware-конверсии в роутере.
- **I2. Округление рубежа до секунды.** `bump_tokens_valid_since` ставит `utcnow().replace(microsecond=0)`,
  сравнение в `tokens_invalidated_for` — строгий `<`. Это **намеренно**: даёт немедленный re-login после
  reset/logout-all (свежий токен той же секунды переживает рубеж). Цена — sub-секундное окно, где чужой
  токен той же секунды может ускользнуть от рубежа; на change-password оно закрыто точечным отзывом по
  `jti`. Не «чинить» округление на микросекунды — сломается re-login.
- **I3. Связка отзыва на change-password = как у `logout-all`.** `bump` (массово) + `_revoke_request_token`
  (точечно текущий jti) + `delete_cookie`. Точечный отзыв обязателен: без него токен-инициатор той же
  секунды переживёт округлённый рубеж. Не упрощать до «только bump».
- **I4. Инвалидация только при УСПЕХЕ.** `bump`/отзыв идут ПОСЛЕ `verify(current_password)` и
  `update_user_password`. Неверный текущий пароль → 400, рубеж не двигается, сессии живы. Не выносить
  `bump` выше проверок.
- **I5. Симметрия и асимметрия флоу.** change (залогинен) — bump + jti + cookie; reset (не залогинен) —
  только bump (нет текущего токена/cookie). Не добавлять cookie-очистку в reset (нет `Response`,
  пользователь не в сессии) и не добавлять точечный отзыв (нет предъявленного токена).
- **I6. Option A.** Гасим текущую сессию тоже. Не переоткрывать на Option B (сохранить текущую) — см.
  обоснование ниже.

**Почему Option A, а не B (фиксирую, чтобы при слиянии не «улучшили» обратно).** Финпродукт, планка
банков: «перелогинься после смены пароля» — сильнее и ожидаемее. Option B (оставить текущую сессию,
как Google/GitHub) требует ре-выпуска токена + нового cookie + чтобы фронт подхватил новый токен; фронт
— отдельная веха (Jinja+vanilla JS), в бэк-батч не тащим. Option A — zero frontend coordination,
симметрично, строго. Решение Василия.

## 5. Разложение этой линии по измерениям (для шага 3–4 гайда — что здесь сильное)

- **Роуты/API.** Хуки переиспользуют существующий `_revoke_request_token` (DRY) и повторяют проверенный
  паттерн `logout-all` — единообразие, минимум нового кода, изменён один файл.
- **Тесты.** 7 кейсов, включая негативный (привязка к успеху) и изоляцию между пользователями;
  детерминизм через форж прошлого `iat` (не зависит от границы секунды).
- **Дизайн-выбор.** Option A с явным обоснованием под доменные требования (финансы) — задокументирован
  в коде, CHANGELOG, WATCHLOG §4 и здесь.
- **Обвязка.** Полный релизный цикл (версия/CHANGELOG/ROADMAP/WATCHLOG §0/§3/§4), §3 ровно 10.

Если у другой вахты сильнее тесты (больше кейсов, например на TTL/просроченный reset-токен) или чище
роутер — брать их по этому измерению, мою логику-инвариант (§4) сохранять.

## 6. Чек-лист слияния ЭТОЙ линии (инстанс процедуры гайда §4)

1. Распаковать все линии, метящие в эту фичу, в `cmp/<вахта>`; свою (`v4.27.0`) — базой. Сверить
   `app/config.py` — все ли про одну версию.
2. `diff` по `app/api/routes_auth.py` между линиями. Файлы из §2 «не тронуто» — пропустить (если у
   кого-то отличаются, это его отдельные правки, эскалировать Василию, не мешать в этот мёрдж).
3. Если другая вахта **не трогала** парольные ручки → просто перенести два хука (change + reset) и
   тест-файл в каноническую базу. Перейти к п.6.
4. Если **трогала** → выбрать одну реализацию change/reset по критериям: соблюдает I1–I6 (§4) > DRY
   (переиспользует `_revoke_request_token`, а не дублирует логику) > покрытие тестами > читаемость.
   Зафиксировать выбор и довод.
5. Согласовать стыки (I1–I5): `change_password` имеет `Request`+`Response` в сигнатуре; вызовы идут
   ПОСЛЕ verify+update; reset без cookie/jti; `bump` использует ту же округлённую семантику.
6. Тесты — объединение: базовый `test_password_session_revocation.py` + уникальные ценные кейсы других
   линий (без дублей). Негативный кейс и изоляцию сохранить обязательно.
7. Гейты заново (§7 ниже).
8. Обвязка: версия остаётся `v4.27.0` (фича одна — слияние не меняет номер, гайд §3); CHANGELOG/ROADMAP/
   WATCHLOG объединить; в **WATCHLOG §4** дописать след «консолидация линий v4.27.0: что откуда и почему».
9. Упаковать одну каноническую `finpilot_v4_27_0_intl` + WATCHLOG отдельным файлом.

## 8. Семантические конфликты — особое внимание (не ловятся текстовым diff)

- **Двойной `bump`.** Если две линии добавили инвалидацию в одну ручку, текстовый мёрдж может оставить
  два `bump_tokens_valid_since` подряд. Функционально безвредно, но грязно — оставить один.
- **Option A vs B.** Если другая вахта реализовала Option B (сохранение текущей сессии через ре-выпуск
  токена) — это НЕ «более полная версия для слияния поверх», а другое продуктовое решение. Выбор —
  Option A (директива Василия). Куски Option B (ре-выпуск/cookie-set) в каноническую линию НЕ тащить.
- **Округление рубежа.** Если линии разошлись по `bump` (округление vs микросекунды) — выбирать не по
  «чище», а по контракту I2 (округление, ради re-login). Это решение, а не вкус.
- **Порядок проверок.** Убедиться, что `bump`/отзыв везде ПОСЛЕ verify+update (I4) — если у другой линии
  раньше, старые сессии гасились бы даже при неуспешной смене (тихий баг).
