# Типы разборов: что и когда писать (роутер)

> Какой формат выбрать под ситуацию. Это **навигатор**, а не сам шаблон: критерии и таблица
> «когда что», ссылки на практические гайды по каждому типу и на общую теорию. Сами шаблоны —
> в файлах-гайдах своих типов.

В проекте четыре типа разбора + общая методичка-теория:

| Тип | Когда | Практический гайд | Реестр |
|---|---|---|---|
| **Post-mortem (инцидент)** | был сбой с влиянием / near miss | `docs/incident_postmortem_guide.md` | `docs/incidents_summary.md` |
| **Investigation report (расследование)** | долго разбирались, причину нашли, но дефекта в коде НЕТ | `docs/investigation_report_guide.md` | `docs/investigations_summary.md` |
| **Bug card (карточка дефекта)** | найден конкретный дефект кода | `docs/bug_report_template.md` | — (в трекере/CHANGELOG) |
| **Pitfall (частая ошибка воркфлоу)** | повторяющаяся ошибка процесса, причина известна сразу | — (строка-урок) | `docs/pitfalls.md` |

Общая теория (культура blameless, RCA-методы, runbook, ADR/RFC, SLO/error budget) — отдельно
в `docs/documentation_methodology.md`.

## 1. Когда что писать

| Ситуация | Документ |
|---|---|
| Сбой с влиянием на пользователей (даунтайм, потеря данных, деградация) | **Post-mortem** |
| Near miss («чуть не упало», обошлось) | **Post-mortem** (дешёвый урок) |
| Долго/сложно разбирались, нашли причину, но бага в коде нет (внешнее, артефакт теста, подход) | **Investigation report** |
| Найден конкретный дефект кода | Карточка **BUG-XXX** |
| Повторяющаяся ошибка воркфлоу (кривая команда, привычка), причина ясна сразу | **Pitfall** (строка в `pitfalls.md`) |
| Тривиальная опечатка без влияния | Ничего, просто фикс |

Различение по сути:
- **Инцидент vs расследование** — был ли дефект в продуктовом коде / влияние. Да → инцидент; причину
  искали, но дефекта в коде нет → расследование.
- **Расследование vs pitfall** — была ли причина неочевидной. Искали, докапывались → расследование
  (отдельный файл). Причина известна сразу, это повтор привычки → pitfall (одна строка, без файла).
- **Bug card** — ортогонально: фиксирует сам дефект; часто рождается ВНУТРИ post-mortem (если корень —
  баг) и ссылается оттуда.

Форматы комбинируются: post-mortem → внутри ссылка на BUG-XXX и/или на новый runbook (если будет
повторяться).

## 2. Разборы оформляются автоматически (правило §14)

Как только событие квалифицируется по таблице выше — разбор делается **сам, в момент события**,
не дожидаясь отдельной просьбы (WATCHLOG §6, правило §14). Полные разборы (post-mortem,
investigation) — отдельными файлами владельцу (правило §12); в репо остаются сводки-строки в
реестрах. Pitfall и bug card — легче, живут строками/карточками.

## 3. Примеры из этого проекта (какой тип под что)

- **E2E (CSRF/CORS + флаги браузера)** — *post-mortem*: фикс в несколько строк искался долго (симптом
  `-1` приняли за «смерть песочницы»). BUG-024, BUG-025.
- **Fetch ЦБ (403 / зеркало / парсер)** — *investigation*: дефекта нет (graceful fallback корректен),
  данные добыты обходом; ограничение LIM-001.
- **firefox/webkit в песочнице** — *investigation*: дефекта в коде/конфиге нет; бинари не качаются (CDN
  Playwright вне allowlist). Перебраны все каналы → внешнее ограничение.
- **«Баг ядра» = гонка в E2E** — *investigation*: подозрение с мат-ядра снято тремя уровнями
  доказательств; чинился тест, не продукт.
- **grep в `&&`-цепочке обрывает вывод** — *pitfall* (PIT-001): причина известна сразу (код 1 при 0
  совпадений), повтор привычки → строка в `pitfalls.md`, без отдельного файла.

Общий урок (он же антипаттерн в гайде инцидентов): **не списывать на «невозможно», не перебрав все
каналы и обходы** (правило §11).
