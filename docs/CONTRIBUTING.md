# Как разрабатывать FINPILOT (CONTRIBUTING)

> Рабочий процесс и стандарты проекта. Читается перед первым коммитом — отвечает на вопрос
> «как тут принято работать». Для любого участника: разработчика, ревьюера, будущего себя.

---

## 1. Стек и запуск локально

- Python 3.12, FastAPI, SQLAlchemy 2.0, Alembic, Pydantic v2, Jinja2, vanilla JS.
- БД: SQLite (локально, быстро) / PostgreSQL (production и проверка схемы).

```
# окружение
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# миграции + запуск
alembic upgrade head
uvicorn app.main:app --reload     # http://127.0.0.1:8000
```

Секреты — только через `.env` (`python-dotenv`), никогда в коде. Шаблон — `.env.example`.

---

## 2. TDD — обязателен

Цикл на каждое изменение поведения:

1. **Тест сначала.** Пишешь/обновляешь тест под желаемое поведение.
2. **Red.** Прогоняешь — он падает (значит тест действительно проверяет то, чего ещё нет).
3. **Реализация.** Пишешь минимальный код, чтобы тест прошёл.
4. **Green.** Прогоняешь — зелёный.

Почему жёстко так: в проекте много взаимосвязанных компонентов (ядро алгоритма, расчёты,
миграции). Без теста, написанного заранее, легко не заметить функциональную или математическую
регрессию.

---

## 3. Прогон тестов

```
# весь набор
SECRET_KEY=test-secret-key-for-ci python3 -m pytest -p no:warnings

# конкретный файл/класс
SECRET_KEY=test-secret-key-for-ci python3 -m pytest tests/test_referral.py -p no:warnings

# с покрытием (gate ≥90%)
SECRET_KEY=test-secret-key-for-ci python3 -m pytest --cov=app --cov-report=term-missing
```

Полный прогон занимает ~2.5 мин. Покрытие не должно опускаться ниже 90% (`.coveragerc`).

---

## 4. PostgreSQL — обязательная проверка схемных изменений

**Правило:** любое изменение моделей/миграций проверяется на живом PostgreSQL, не только на SQLite.

Причина: SQLite по умолчанию не форсит внешние ключи и мягче к типам. Реальный баг (нарушение
FK при каскадном удалении) был невидим на SQLite и проявлялся только на PostgreSQL. Прогон на
SQLite даёт ложное «всё зелено».

```
DATABASE_URL="postgresql+psycopg://<user>:<pass>@127.0.0.1:5432/finpilot_test" \
SECRET_KEY=test-secret-key-for-ci python3 -m pytest tests/test_<schema_change>.py
```

## 5. Ядро алгоритма под защитой

`app/core/` (оптимизация `U(a)`, отбор альтернатив `a*`) **не меняется без явного согласования**.
Канонический спек — `docs/Математическая_модель_v3_0_0.md`.

Исключение: `app/core/spending_advice.py` — чисто информационный слой, он **не входит** в целевую
функцию `U(a)` и не влияет на оптимальное действие. Его расширять безопасно.

## 6. Definition of Done

Фича/фикс считается готовым, когда:

1. Тесты написаны до реализации (TDD), все зелёные.
2. Покрытие не упало ниже 90%.
3. Схемные изменения проверены на PostgreSQL.
4. Сделан живой смоук (не только pytest — поднять приложение, проверить эндпоинт).
5. Для UI — проверка в WebKit/Safari и на мобильном вьюпорте (там исторически были баги).
6. Обновлён `CHANGELOG.md` и поднята версия (см. ниже).

---

## 7. Версионирование

Строго [SemVer](https://semver.org/lang/ru/):

- **MINOR** (`4.15.0 → 4.16.0`) — новая функциональность.
- **PATCH** (`4.16.0 → 4.16.1`) — исправление без новой функциональности.
- **MAJOR** — несовместимые изменения (редко, осознанно).

На каждый релиз:
- bump `APP_VERSION` в `app/config.py`;
- запись в `CHANGELOG.md` (формат Keep a Changelog: что добавлено/исправлено/проверено);
- git-тег `vX.Y.Z` и GitHub Release.

## 8. Стиль кода

- PEP 8 строго (snake_case переменные, PascalCase классы).
- Type hints — на сигнатурах функций.
- Комментарии — минимальные, код говорит сам за себя; комментарий объясняет «почему», не «что».
- Обработка ошибок — там, где сбой вероятен или критичен, без избыточности.
- Без глобальных переменных. Без Python 2 синтаксиса.
- Избегать массового реформатирования (`black .` на всём) — ломает merge между ветками.
