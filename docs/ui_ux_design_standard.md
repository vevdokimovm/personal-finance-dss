# FINPILOT — Спецификация требований к UI/UX (Frontend Design Standard)

> **Назначение.** Нормативный документ, описывающий требования к фронтенду и дизайну FINPILOT по шести классическим UX-доменам: информационная архитектура и навигация, состояния интерфейса, консистентность компонентов, доступность (a11y), поведение форм, обратная связь на действия. Документ детализирует поверхностный UX-блок реестра требований (`UX-01..13`) до уровня, по которому разработчик может реализовать интерфейс, а ревьюер — принять работу.
>
> **Жанр.** Это стандарт (design requirements specification), а не туториал. Требования сформулированы нормативным языком, у каждого есть проверяемый критерий приёмки и привязка к внешнему стандарту (WCAG / ISO / ГОСТ) и к карточке реестра.
>
> **Статус.** Рабочий baseline. Дополняет реестр `FINPILOT_Requirements_MASTER_v3.md`, не заменяет его.

---

## 0. Преамбула

### 0.1. Область применения

Документ распространяется на весь веб-интерфейс FINPILOT (Jinja2-страницы + клиентский JS + CSS). Он задаёт **минимальный обязательный уровень** UI-качества и **рекомендуемые практики**. Бэкенд-логика, алгоритмы и API — вне области (см. основной реестр).

### 0.2. Как читать требования (нормативный язык)

Ключевые слова трактуются по RFC 2119 / ISO (рус. эквиваленты):

| Слово | Англ. | Смысл |
|-------|-------|-------|
| **ДОЛЖЕН / ДОЛЖНА** | SHALL / MUST | Обязательное требование. Невыполнение = дефект, блокирующий приёмку. |
| **СЛЕДУЕТ** | SHOULD | Настоятельная рекомендация. Отклонение допустимо при явном обосновании. |
| **МОЖЕТ** | MAY | Опционально, на усмотрение разработчика. |
| **НЕ ДОЛЖЕН** | SHALL NOT / MUST NOT | Прямой запрет. |

Каждая карточка требования имеет формат: **ID**, нормативная формулировка (по возможности в шаблоне EARS: *When/While/If/Where … система ДОЛЖНА …*), **приоритет** (MoSCoW), **критерий приёмки** (как проверить), **стандарт** (внешний источник), **связь** (карточка реестра `UX-XX` или другой домен).

ID доменно-префиксные, чтобы не конфликтовать с реестром:
`TOK-` токены · `IA-` архитектура/навигация · `ST-` состояния · `CMP-` компоненты · `A11Y-` доступность · `FRM-` формы · `FB-` обратная связь.

### 0.3. Технический контекст (важно для реализуемости)

FINPILOT — **server-rendered** приложение: FastAPI отдаёт HTML через Jinja2, интерактивность добавляется ванильным JS. SPA-фреймворка (React/Vue) нет. Это влияет на требования:

1. **Прогрессивное улучшение (progressive enhancement).** Базовый сценарий ДОЛЖЕН работать на серверном HTML и нативных элементах; JS — это слой улучшения, а не условие работоспособности. Форма отправляется и при выключенном JS (нативный `<form method=post>`), JS лишь добавляет inline-валидацию и optimistic UI.
2. **Состояния — это разметка, а не только JS-стейт.** `empty`/`error`-состояния серверных страниц рендерятся сервером (Jinja2-условие), `loading` для AJAX-фрагментов — клиентом. Оба пути нормируются (домен ST).
3. **Никаких тяжёлых зависимостей ради мелочей.** Компонентная консистентность достигается CSS-переменными и переиспользуемыми Jinja2-партиалами/макросами, а не подключением UI-библиотеки.

### 0.4. Нормативные и справочные источники

| Источник | Роль | Применение в документе |
|----------|------|------------------------|
| **WCAG 2.2, Level AA** | нормативный (целевой уровень) | домен A11Y, контраст токенов, формы, фокус |
| **ISO 9241-110:2020** (Interaction principles) | нормативный | 7 принципов диалога — каркас доменов IA/ST/FRM/FB |
| **ISO 9241-112** (Presentation of information) | справочный | состояния, иерархия информации |
| **ISO 9241-143** (Forms) | справочный | домен FRM |
| **ISO/IEC 25010** (Interaction capability) | справочный | recognizability, learnability, user error protection |
| **ГОСТ Р ИСО 9241-210** (человеко-ориентированное проектирование) | справочный (для ВКР) | обоснование процесса, привязка к защите |
| **WAI-ARIA Authoring Practices (APG)** | нормативный для паттернов | домен CMP, A11Y — роли/состояния компонентов |

### 0.5. Аудитория продукта и проектные ограничения от персон

UI проектируется не «в среднем», а под реальные персоны из юзабилити-сессий. Они задают жёсткие ограничения:

1. **R01 — 56 лет, оператор, низкая фин-грамотность.** На сессии: «не поняла, что вводить», «не нашла, что тыкать», «пропустила интерактивные элементы». → Требования к крупным целям нажатия, явным аффордансам, инструкциям-подсказкам, простому языку (домены IA, A11Y, FB).
2. **Бухгалтер.** Ментальная модель из 1С: ждёт раздельные доходы/расходы, полные термины («досрочное погашение», не «досрочка»), возможность проверить каждую цифру, табличный drill-down. → Требования к консистентной терминологии, читаемости данных, заземлению чисел (домены CMP, FRM).
3. **Мобильный пользователь.** ~76.7% респондентов хотят мобайл; фидбэк «на телефоне мелко, ничего не понятно». → Требования к адаптивности и размерам касаемых целей (домены A11Y, CMP).

### 0.6. Связь с реестром требований

Документ разворачивает карточки реестра: `UX-01` (разгрузка экрана) → домен IA; `UX-02` (без формул) → IA/CMP/FRM; `UX-03` (инструкции) → ST/FB; `UX-04` (состояния) → весь домен ST; `UX-05` (графики) → IA; `UX-06` (упростить ввод) → FRM; `UX-07/UX-12` (термины, консистентность) → CMP; `UX-08/09` (темы, мобайл) → TOK/A11Y; `UX-10` (обнаруживаемость) → IA/CMP; `UX-11` (уведомления) → FB; `UX-13` (undo, доступность 55+) → FB/A11Y. Полная трассировка — раздел 9.

## 1. Фундамент: дизайн-токены и система

Прежде чем нормировать домены, нужен общий словарь визуальных решений — **дизайн-токены**. Это именованные переменные (цвет, отступ, шрифт, тень), на которые ссылаются все компоненты. Без токенов «консистентность» (домен CMP) и «контраст» (домен A11Y) недостижимы: каждый экран будет считать пиксели заново, и значения разъедутся. Токены — это контракт, единый источник правды для оформления.

**Реализация в стеке FINPILOT:** токены объявляются как CSS custom properties в `:root`, тема переключается атрибутом `data-theme="light|dark"` на `<html>`. Никакого JS для применения токенов не требуется — каскад делает всё сам.

### [TOK-01] Все цвета — через семантические токены, не «сырые» значения
- **Описание:** Компоненты НЕ ДОЛЖНЫ использовать «сырые» цветовые значения (`#0a0a0a`, `rgb(...)`) напрямую; цвет ДОЛЖЕН задаваться семантическим токеном (`--color-bg-surface`, `--color-text-primary`, `--color-accent`, `--color-danger`, `--color-success`, `--color-border`).
- **Rationale:** Семантические роли (а не «синий») позволяют переключать тему и менять палитру в одном месте. Прямые значения = рассинхрон и невозможность светлой темы (UX-08).
- **Приоритет:** **Must**
- **Критерий приёмки:** в CSS компонентов нет хардкод-цветов вне файла токенов; смена `data-theme` перекрашивает весь UI без правок компонентов.
- **Стандарт:** ISO 9241-112 (консистентное кодирование); связь TOK-02, CMP-01, UX-08.

### [TOK-02] Парные токены для светлой и тёмной темы; бренд ≠ фон рабочих экранов
- **Описание:** Система ДОЛЖНА определять полный набор цветовых токенов для обеих тем (`light` и `dark`). Фирменный неон логотипа ДОЛЖЕН использоваться как **акцент** (CTA, выделение), а НЕ как фон рабочих экранов.
- **Rationale:** Сессия №2: бухгалтер «привыкла к светлому экрану, чёрные цифры непривычны». Паспорт логотипа и реестр постов: «светлая палитра, чтобы не получился ещё один банк-онлайн» — айдентика держится на акценте, а не на заливке всего экрана неоном.
- **Приоритет:** **Must** (светлая тема — Could по срокам, но токены закладываются сразу)
- **Критерий приёмки:** определены оба набора токенов; в светлой теме фон рабочих областей светлый, неон — только акценты; контраст сохраняется в обеих темах (см. A11Y-01).
- **Стандарт:** WCAG 2.2 (1.4.3); связь UX-08, `FINPILOT_Паспорт_логотипа_v3.md`, `FINPILOT_Posts_Registry.md`.

### [TOK-03] Типографическая шкала (модульная), не произвольные кегли
- **Описание:** Размеры текста ДОЛЖНЫ браться из фиксированной шкалы (например `--font-size-xs/sm/base/lg/xl/2xl/3xl`), а не задаваться произвольными `px` на местах. Базовый размер тела текста ДОЛЖЕН быть ≥ 16px.
- **Rationale:** Шкала даёт визуальную иерархию и консистентность (CMP). Базис 16px — против жалобы «на телефоне мелко» и под аудиторию 55+ (A11Y).
- **Приоритет:** **Must**
- **Критерий приёмки:** все размеры текста — токены шкалы; базовый body ≥ 16px; иерархия (заголовок > подзаголовок > тело) визуально считывается.
- **Стандарт:** ISO 9241-112; WCAG 2.2 (1.4.4 — масштабирование до 200%); связь UX-09, UX-13.

### [TOK-04] Шкала отступов (8-pt grid) и единый ритм
- **Описание:** Отступы и промежутки ДОЛЖНЫ кратны базовому шагу (рекомендуется 4/8px: `--space-1=4px … --space-8=32px`).
- **Rationale:** Единый ритм вертикали/горизонтали = ощущение порядка, ключ к «разгрузке экрана» (UX-01). Хаотичные отступы читаются как «перегружено».
- **Приоритет:** **Should**
- **Критерий приёмки:** значения `margin/padding/gap` берутся из шкалы; нет случайных `7px`, `13px`.
- **Стандарт:** ISO 9241-112; связь CMP-01, UX-01.

### [TOK-05] Радиусы, тени (elevation), границы — токенами
- **Описание:** Скругления (`--radius-*`), тени (`--shadow-*` как уровни возвышения) и цвет/толщина границ ДОЛЖНЫ задаваться токенами и применяться консистентно по смыслу (например, тень = «приподнятый» слой: модалка, поповер).
- **Приоритет:** **Should**
- **Критерий приёмки:** одинаковые по смыслу элементы имеют одинаковый радиус/тень; уровни возвышения не противоречат друг другу.
- **Стандарт:** ISO 9241-112; связь CMP-01.

### [TOK-06] Токены движения + уважение `prefers-reduced-motion`
- **Описание:** Длительности и кривые анимаций ДОЛЖНЫ быть токенами (`--motion-fast=150ms`, `--motion-base=250ms`, `--ease-standard`). Система ДОЛЖНА отключать/смягчать нефункциональную анимацию при `@media (prefers-reduced-motion: reduce)`.
- **Rationale:** Консистентная микроанимация (FB) + a11y: для части пользователей анимация вызывает дискомфорт/вестибулярные реакции (WCAG 2.3.3).
- **Приоритет:** **Should**
- **Критерий приёмки:** анимации используют токены; при включённом reduce-motion переходы отключены или сведены к мгновенным.
- **Стандарт:** WCAG 2.2 (2.3.3); связь FB-04, A11Y-09.

### [TOK-07] Токен видимого фокуса (focus ring)
- **Описание:** Система ДОЛЖНА определять единый видимый стиль фокуса (`--focus-ring`) с достаточным контрастом и применять его ко всем интерактивным элементам; `outline` НЕ ДОЛЖЕН глобально отключаться (`outline: none` без замены запрещён).
- **Rationale:** Видимый фокус — фундамент клавиатурной навигации (A11Y). Частая ошибка — «убрать некрасивую обводку» — ломает доступность.
- **Приоритет:** **Must**
- **Критерий приёмки:** при навигации Tab каждый интерактивный элемент имеет заметный фокус-индикатор; контраст индикатора ≥ 3:1.
- **Стандарт:** WCAG 2.2 (2.4.7 Focus Visible, 2.4.11 Focus Appearance); связь A11Y-04.

### [TOK-08] Контрольные точки адаптива (breakpoints)
- **Описание:** Адаптивная вёрстка ДОЛЖНА опираться на согласованный набор контрольных точек (например `sm 480 / md 768 / lg 1024`) и проектироваться mobile-first.
- **Rationale:** Единые breakpoints = предсказуемое поведение на всех экранах (UX-09).
- **Приоритет:** **Should**
- **Критерий приёмки:** раскладка ломается только на заявленных точках; базовая вёрстка — для узкого экрана, расширяется вверх.
- **Стандарт:** связь A11Y-08, UX-09.

**Антипаттерны фундамента (NOT DO):**
1. Цвета/отступы/кегли «по месту» в каждом шаблоне — гарантированный рассинхрон.
2. `outline: none` без альтернативного фокуса.
3. Неон как заливка рабочих экранов (визуальный шум + контрастные проблемы).
4. Базовый шрифт < 16px на мобильном.

## 2. Домен A — Информационная архитектура и навигация (IA)

**Принцип.** Пользователь в любой момент ДОЛЖЕН понимать три вещи: *где я, что здесь главное, что делать дальше*. В FINPILOT этот домен проваливался ярче всего — R01 «не нашла, что тыкать», первый экран перегружен, симулятор и шкалы остались незамеченными. Архитектура — это приоритизация: один экран = одна основная задача. Опора: ISO 9241-110 (self-descriptiveness, conformity with expectations), эвристики Нильсена №1 (видимость статуса системы), №6 (узнавание вместо припоминания).

### [IA-01] Один экран — одна основная задача и одно главное действие
- **Описание:** When пользователь открывает экран, интерфейс ДОЛЖЕН выделять ровно один primary-элемент (главный показатель/ответ) и не более одного primary-CTA; второстепенное — визуально подчинено.
- **Rationale:** «Перегружен», «непонятно с чего начать» — топ раздражителей. Разворачивает UX-01; на дашборде главный показатель — свободный ресурс `Rt` (FR-04).
- **Приоритет:** **Must**
- **Критерий приёмки:** на каждом ключевом экране ≤ 1 primary-CTA; в коридорном тесте пользователь за < 5 сек называет «что здесь главное» и «что нажать».
- **Стандарт:** ISO 9241-112 (информационная иерархия); Нильсен №8 (эстетика и минимализм); связь UX-01, FR-04.

### [IA-02] Видимая модель навигации: где я и как вернуться
- **Описание:** The system shall показывать постоянный навигационный каркас (основные разделы) и текущую позицию (active-состояние пункта); с любого экрана ДОЛЖЕН быть очевиден возврат назад/на главную.
- **Rationale:** Без индикации позиции пользователь теряется (особенно 55+). Нильсен №1 (видимость статуса).
- **Приоритет:** **Must**
- **Критерий приёмки:** активный раздел визуально помечен; пользователь находит «назад/домой» без подсказки; структура разделов постоянна между экранами.
- **Стандарт:** ISO 9241-110 (self-descriptiveness); связь UX-12.

### [IA-03] Глубина навигации к ключевым действиям ≤ 2 шага
- **Описание:** The system shall обеспечивать доступ к ключевым задачам (внести доход/расход, получить рекомендацию, посмотреть «сколько свободно») за ≤ 2 действия от главного экрана.
- **Rationale:** Чем глубже, тем выше отвал; time-to-aha — гейт активации.
- **Приоритет:** **Should**
- **Критерий приёмки:** в тесте пользователь достигает каждой ключевой задачи за ≤ 2 клика/тапа.
- **Стандарт:** Нильсен №7 (гибкость и эффективность); связь UX-01, FR-05.

### [IA-04] Каждый экран заканчивается следующим шагом (no dead ends)
- **Описание:** The system shall на каждом экране предлагать явное следующее действие, а не оставлять пользователя наедине со статистикой.
- **Rationale:** Пользователи уходят от «голой статистики» (P0.3). Разворачивает UX/FR-05.
- **Приоритет:** **Must**
- **Критерий приёмки:** на каждом экране есть CTA или подсказка к следующему шагу; нет экранов-тупиков.
- **Стандарт:** ISO 9241-110 (suitability for the task); связь FR-05, UX-03.

### [IA-05] Интерактивные элементы выглядят интерактивными (аффорданс)
- **Описание:** The system shall визуально отличать кликабельные элементы (кнопки, ссылки, шкалы, карточки-триггеры, симулятор) от статичных — формой, цветом-акцентом, курсором, hover/focus-состоянием.
- **Rationale:** Сессии №1/№2: симулятор и шкалы выбора «не нашла/не тыкала» — фичи есть, но не читаются как интерактив. Разворачивает UX-10.
- **Приоритет:** **Must**
- **Критерий приёмки:** в тесте пользователь без подсказки обнаруживает симулятор и шкалы; у всех интерактивных элементов есть hover- и focus-аффорданс.
- **Стандарт:** Нильсен №6 (узнавание); ISO 9241-112; связь UX-10, FR-07.

### [IA-06] Прогрессивное раскрытие сложности (progressive disclosure)
- **Описание:** Where информация второстепенна или экспертна (формульные детали, расширенные альтернативы, технические показатели), система ДОЛЖНА скрывать её за раскрытием (аккордеон/«подробнее»), а не показывать всё сразу.
- **Rationale:** Баланс между объяснимостью (KEEP-02) и перегрузом: новичку — простой ответ, желающему — детали по запросу. Поддерживает «без формул на главном» (UX-02) при сохранении прозрачности.
- **Приоритет:** **Should**
- **Критерий приёмки:** первичный экран показывает суть; детали (раскладка альтернатив, расчёт показателя) доступны по явному раскрытию.
- **Стандарт:** Нильсен №8; ISO 9241-112; связь FR-03, UX-02, UX-05.

### [IA-07] Осознанная визуализация: график только со смыслом
- **Описание:** The system shall показывать график, только если он помогает принять решение; декоративные/дублирующие графики НЕ ДОЛЖНЫ перегружать экран.
- **Rationale:** «Слишком много графиков» — раздражитель. Разворачивает UX-05.
- **Приоритет:** **Should**
- **Критерий приёмки:** каждый график на экране отвечает на конкретный вопрос пользователя; лишние удалены.
- **Стандарт:** ISO 9241-112; связь UX-05.

**Антипаттерны IA (NOT DO):**
1. Дашборд «всё сразу» без выделенного главного — повтор провала первого экрана.
2. Скрытый primary-флоу (как незамеченный симулятор).
3. Экран-тупик: цифры есть, действия нет.
4. Несколько конкурирующих primary-кнопок на одном экране.

**Чек-лист приёмки домена IA:**
1. На экране один главный элемент и ≤ 1 primary-CTA.
2. Видна текущая позиция; возврат очевиден.
3. Ключевые задачи — за ≤ 2 шага.
4. Нет тупиков: всегда есть следующий шаг.
5. Интерактив отличим от статики; симулятор/шкалы обнаруживаются.
6. Экспертные детали — за прогрессивным раскрытием.

## 3. Домен B — Состояния интерфейса (ST)

**Принцип.** Любая область, отображающая данные, имеет не одно, а **пять** состояний: `loading`, `empty`, `error`, `partial` (частично/мало данных) и `ideal` (данные есть). Проектировать только «идеальное» — главная причина «ступора» нового пользователя: пустой экран без подсказки не говорит, что делать. У FINPILOT на экране «Ликвидные активы» empty-подсказка уже есть — её паттерн ДОЛЖЕН стать общим. Опора: Нильсен №1 (видимость статуса), ISO 9241-110 (self-descriptiveness, use error robustness). Разворачивает UX-04.

### [ST-01] Полный набор состояний для каждой области данных
- **Описание:** The system shall определять для каждой области/экрана, отображающего данные, состояния `loading`, `empty`, `error` и `ideal` (и `partial`, где применимо).
- **Rationale:** Незапроектированное состояние = баг по умолчанию (белый экран, спиннер навечно, голая ошибка). Разворачивает UX-04.
- **Приоритет:** **Must**
- **Критерий приёмки:** для каждого экрана данных в коде/макете явно заданы все применимые состояния; нет «необработанных» пустых/ошибочных видов.
- **Стандарт:** ISO 9241-110 (self-descriptiveness); Нильсен №1; связь UX-04.

### [ST-02] Loading: индикация процесса и защита от «вечного спиннера»
- **Описание:** While выполняется запрос дольше ~300мс, система ДОЛЖНА показывать индикатор загрузки (предпочтительно skeleton для контента, спиннер для действий); при затяжке/таймауте ДОЛЖНА переходить в `error` с возможностью повторить.
- **Rationale:** Без индикации пользователь думает, что «зависло» и уходит/дублирует действие. Skeleton снижает воспринимаемое ожидание.
- **Приоритет:** **Should**
- **Критерий приёмки:** долгие операции показывают loading; нет состояния «крутится вечно» — есть таймаут → error+retry; кнопка, инициировавшая запрос, переходит в loading-состояние (см. FB-02).
- **Стандарт:** Нильсен №1; ISO 9241-112; связь FB-02.

### [ST-03] Empty: объясни, что это и что добавить (actionable empty state)
- **Описание:** When область данных пуста, система ДОЛЖНА показать: что это за область, почему пусто, и **CTA для первого действия** (например «Добавьте первый доход»). Пустое состояние НЕ ДОЛЖНО быть просто белым экраном или «нет данных».
- **Rationale:** Для нового пользователя (особенно 55+) пустой экран = ступор «что вводить». Превращаем пустоту в онбординг-точку.
- **Приоритет:** **Must**
- **Критерий приёмки:** каждое пустое состояние содержит заголовок, поясняющий текст и CTA; в тесте пользователь из пустого экрана понимает, что делать.
- **Стандарт:** ISO 9241-110 (suitability for learning); Нильсен №10 (помощь); связь UX-03, UX-04, FR-04.

### [ST-04] Error: что случилось, почему, что делать — человеческим языком
- **Описание:** If операция завершилась ошибкой, then система ДОЛЖНА показать понятное сообщение (что произошло + как исправить/повторить), БЕЗ кодов, стектрейсов и сырых исключений. Деструктивные/системные ошибки ДОЛЖНЫ предлагать действие восстановления (retry, назад, поддержка).
- **Rationale:** Сырые 500/ZeroDivision пугают и не помогают (ср. BUG-04). Нильсен №9 (помощь в распознавании и устранении ошибок).
- **Приоритет:** **Must**
- **Критерий приёмки:** пользовательские ошибки не содержат технических деталей; у каждой есть путь восстановления; на пустом профиле/нулевом доходе — понятный диагноз, не 500 (связь BUG-04).
- **Стандарт:** ISO 9241-110 (use error robustness); Нильсен №9; связь BUG-04, NFR-09.

### [ST-05] Partial / недостаточно данных: честный частичный результат
- **Описание:** Where данных мало для полного расчёта (нет целей, один месяц истории), система ДОЛЖНА показывать частичный результат с пометкой об ограничении и подсказкой, что добавить для полноты, а не пустоту и не ложную точность.
- **Rationale:** Поддерживает прогрессивный онбординг (FR-10): черновой план на неполных данных полезнее «приходите, когда заполните всё». Соответствует «инженерной честности» проекта.
- **Приоритет:** **Should**
- **Критерий приёмки:** при неполных данных выдаётся частичный результат + явная пометка ограничения + подсказка к дозаполнению.
- **Стандарт:** ISO 9241-110 (self-descriptiveness); связь FR-10, NFR-09.

### [ST-06] Состояния консистентны между экранами
- **Описание:** The system shall использовать единый визуальный паттерн для одноимённых состояний на всех экранах (loading везде выглядит одинаково, empty — по одному шаблону).
- **Rationale:** Разнобой состояний путает; единый паттерн — узнаваемость (CMP). Обобщает существующий empty на «Ликвидных активах».
- **Приоритет:** **Should**
- **Критерий приёмки:** loading/empty/error визуально идентичны across экранов; реализованы переиспользуемыми партиалами/макросами.
- **Стандарт:** ISO 9241-110 (conformity with expectations); связь CMP-01, UX-12.

**Антипаттерны ST (NOT DO):**
1. Белый экран или «Нет данных» без подсказки и CTA.
2. Спиннер без таймаута (вечная загрузка).
3. Сырой стектрейс/код ошибки в лицо пользователю.
4. Ложная точность на неполных данных вместо честного «частично».
5. Разный вид одного состояния на разных экранах.

**Чек-лист приёмки домена ST:**
1. Для каждой области данных заданы все применимые состояния.
2. Loading с таймаутом → error+retry; инициирующая кнопка в loading.
3. Empty = заголовок + причина + CTA.
4. Error = человеческий текст + путь восстановления; нет технических деталей.
5. Partial — честный частичный результат с пометкой.
6. Одноимённые состояния выглядят одинаково везде.

## 4. Домен C — Консистентность компонентов (CMP)

**Принцип.** Один и тот же смысл ДОЛЖЕН везде выглядеть и вести себя одинаково: кнопка «сохранить» — один компонент, термин «досрочное погашение» — одно слово, показатель ликвидности — одно название и формат. Консистентность снижает когнитивную нагрузку (узнавание вместо изучения) и убирает ощущение «сырого» интерфейса, на которое жаловался бухгалтер. Достигается компонентной библиотекой (Jinja2-макросы + токены), а не копипастой. Опора: ISO 9241-110 (conformity with user expectations), Нильсен №4 (консистентность и стандарты). Разворачивает UX-07, UX-12.

### [CMP-01] Единая библиотека переиспользуемых компонентов
- **Описание:** The system shall реализовать повторяющиеся элементы (кнопки, поля ввода, карточки, модалки, таблицы, бейджи, тосты) как переиспользуемые Jinja2-макросы/партиалы поверх токенов; дублирование разметки одного компонента НЕ ДОЛЖНО существовать.
- **Rationale:** Источник консистентности и единственная точка правки. Без библиотеки кнопки разъезжаются по цветам/размерам.
- **Приоритет:** **Must**
- **Критерий приёмки:** каждый повторяющийся UI-элемент имеет один макрос-источник; на ревью нет двух разных реализаций одной кнопки.
- **Стандарт:** Нильсен №4; ISO 9241-110; связь TOK-01, UX-12.

### [CMP-02] Иерархия и единое поведение действий (button variants)
- **Описание:** Кнопки ДОЛЖНЫ иметь чёткие варианты (`primary`/`secondary`/`ghost`/`danger`) с консистентным применением: один primary на экран (см. IA-01), деструктивные действия — вариант `danger`.
- **Rationale:** Визуальная иерархия действий помогает понять, что главное и что опасно (связь FB-03 — подтверждение деструктивного).
- **Приоритет:** **Should**
- **Критерий приёмки:** варианты кнопок применяются по смыслу консистентно; удаление всегда оформлено как danger.
- **Стандарт:** Нильсен №4; связь IA-01, FB-03.

### [CMP-03] Единая терминология и словарь UI (полные термины)
- **Описание:** The system shall использовать одно и то же слово для одного понятия на всех экранах и писать финансовые термины полностью («досрочное погашение», а не «досрочка»; «коэффициент ликвидности», единообразно).
- **Rationale:** Сессия №2: «пишите нормально, не сокращённо». Разнобой `Lt`/BLR/«ликвидность» путает. Опирается на глоссарий (реестр DOC-01).
- **Приоритет:** **Should**
- **Критерий приёмки:** один термин = одно слово везде; нет разговорных сокращений в UI; копирайт сверен с глоссарием.
- **Стандарт:** ISO 9241-110 (conformity); связь UX-07, DOC-01.

### [CMP-04] Единые форматы данных (деньги, даты, проценты)
- **Описание:** The system shall форматировать деньги, даты и проценты единообразно во всём UI (например, `12 500 ₽`, `01.06.2026`, `8,5 %`), с консистентными разделителями и локалью (ru-RU).
- **Rationale:** Бухгалтерская аудитория особенно чувствительна к формату чисел; разнобой выглядит как ошибка данных.
- **Приоритет:** **Should**
- **Критерий приёмки:** один формат денег/дат/процентов на всех экранах; реализован общими фильтрами/хелперами форматирования.
- **Стандарт:** ISO 9241-112; связь FR-02.

### [CMP-05] Никакой формульной нотации в пользовательском UI
- **Описание:** Компоненты, отображающие показатели/объяснения, НЕ ДОЛЖНЫ показывать сырую формульную запись (`Rt = Bt + (It−Σej) − ΣP`, «по форм. 8», `U(a)`); только человекочитаемые формулировки.
- **Rationale:** Скрины v2.0.2 показывают формулы в UI — главный «чёрный ящик» по сессии №2. Это компонентное правило, дополняющее REFACTOR-01/02. Прозрачность сохраняется через заземление (FR-02), а не нотацию.
- **Приоритет:** **Must**
- **Критерий приёмки:** ни один компонент не выводит формульную нотацию пользователю; показатели сопровождаются простым пояснением.
- **Стандарт:** ISO/IEC 25010 (recognizability); связь UX-02, FR-01, FR-02, REFACTOR-01, REFACTOR-02.

### [CMP-06] Консистентные паттерны компонентов по WAI-ARIA APG
- **Описание:** Where используется сложный компонент (модалка, аккордеон, табы, тултип, дропдаун), он ДОЛЖЕН следовать паттерну WAI-ARIA Authoring Practices: корректные роли, состояния и клавиатурное поведение.
- **Rationale:** Самописные компоненты без ARIA-паттерна ломают доступность и ведут себя непредсказуемо. APG — готовый эталон поведения.
- **Приоритет:** **Should**
- **Критерий приёмки:** интерактивные компоненты имеют корректные `role`/`aria-*` и клавиатурное управление согласно APG (см. также домен A11Y).
- **Стандарт:** WAI-ARIA APG; WCAG 2.2 (4.1.2 Name, Role, Value); связь A11Y-05, A11Y-06.

### [CMP-07] Адаптивное поведение компонентов (mobile-first)
- **Описание:** Компоненты ДОЛЖНЫ корректно адаптироваться к ширине экрана (таблицы → карточки/скролл, многоколоночное → одноколоночное), сохраняя читаемость без горизонтального скролла на ~380px.
- **Rationale:** «На телефоне мелко/ничего не понятно»; ~76.7% хотят мобайл. Разворачивает UX-09.
- **Приоритет:** **Should**
- **Критерий приёмки:** на ширине ~380px ключевые компоненты читаемы, без горизонтального скролла; таблицы имеют мобильный паттерн.
- **Стандарт:** WCAG 2.2 (1.4.10 Reflow); связь UX-09, TOK-08.

**Антипаттерны CMP (NOT DO):**
1. Две разные кнопки «сохранить» с разным видом.
2. Один показатель под тремя именами на разных экранах.
3. Разные форматы денег/дат в одном продукте.
4. Формулы в интерфейсе.
5. Самописная модалка без ролей и клавиатуры.

**Чек-лист приёмки домена CMP:**
1. Повторяющиеся элементы — из единой библиотеки макросов.
2. Варианты кнопок применяются консистентно; danger для удаления.
3. Один термин = одно слово; термины полные.
4. Единый формат денег/дат/процентов.
5. Нет формульной нотации в UI.
6. Сложные компоненты следуют WAI-ARIA APG.
7. Компоненты адаптивны (≤ 380px без горизонт. скролла).

## 5. Домен D — Доступность (A11Y)

**Принцип.** Интерфейс ДОЛЖЕН быть пригоден для людей с разными возможностями и в разных условиях — включая прямую ЦА FINPILOT: пользователей 55+ с низкой фин-грамотностью (R01) и тех, кто смотрит с телефона. Доступность здесь не «галочка для галочки», а условие активации реальной аудитории. **Целевой уровень — WCAG 2.2 Level AA.** Опора: WCAG 2.2, ISO 9241-110, ISO/IEC 25010 (accessibility). Разворачивает UX-13 (доступность 55+) и часть UX-08/09.

### [A11Y-01] Контраст текста и интерактивных элементов (WCAG 1.4.3 / 1.4.11)
- **Описание:** The system shall обеспечивать контраст текста к фону ≥ 4.5:1 (обычный) и ≥ 3:1 (крупный ≥ 24px/19px bold); нетекстовые элементы (границы полей, иконки-смыслы, индикаторы) — ≥ 3:1. Требование ДОЛЖНО выполняться в обеих темах.
- **Rationale:** Низкий контраст — барьер для 55+ и на ярком экране телефона. Прямо связан с токенами (TOK-02) и светлой темой (UX-08).
- **Приоритет:** **Must**
- **Критерий приёмки:** проверка контраст-чекером проходит для всех текстов/контролов в light и dark; нет пар ниже порога.
- **Стандарт:** WCAG 2.2 (1.4.3, 1.4.11); связь TOK-01, TOK-02, UX-08.

### [A11Y-02] Полная клавиатурная доступность (WCAG 2.1.1 / 2.1.2)
- **Описание:** The system shall делать все интерактивные функции доступными с клавиатуры (Tab/Shift+Tab/Enter/Space/Esc/стрелки где уместно); НЕ ДОЛЖНО быть «ловушек фокуса» (кроме корректно реализованной модалки, см. A11Y-06).
- **Rationale:** Базис доступности и удобства; многие сценарии (формы ввода) быстрее с клавиатуры.
- **Приоритет:** **Must**
- **Критерий приёмки:** весь основной флоу проходится только клавиатурой; из любого элемента можно выйти; порядок Tab логичен.
- **Стандарт:** WCAG 2.2 (2.1.1, 2.1.2); связь A11Y-04, TOK-07.

### [A11Y-03] Семантический HTML и программная структура (WCAG 1.3.1 / 4.1.2)
- **Описание:** The system shall использовать нативные семантические элементы (`<button>`, `<nav>`, `<main>`, `<h1..h6>`, `<label>`, `<table>` с заголовками) по назначению; «кнопка» НЕ ДОЛЖНА быть `<div onclick>` без роли. Иерархия заголовков ДОЛЖНА быть логичной (один `<h1>` на страницу).
- **Rationale:** Семантика даёт доступность «бесплатно» (скринридеры, клавиатура, SEO) и упрощает код. Самый дешёвый способ быть доступным на SSR-стеке.
- **Приоритет:** **Must**
- **Критерий приёмки:** интерактив — на нативных элементах или с корректной ролью; структура заголовков валидна; landmark-теги присутствуют.
- **Стандарт:** WCAG 2.2 (1.3.1, 4.1.2); ISO 9241-110; связь CMP-06.

### [A11Y-04] Видимый и логичный фокус (WCAG 2.4.7 / 2.4.11)
- **Описание:** The system shall показывать видимый индикатор фокуса на всех интерактивных элементах (через TOK-07); фокус НЕ ДОЛЖЕН скрываться под липкими шапками/панелями.
- **Rationale:** Без видимого фокуса клавиатурная навигация бессмысленна. Частая регрессия — `outline:none`.
- **Приоритет:** **Must**
- **Критерий приёмки:** при Tab фокус всегда виден и не перекрыт; контраст индикатора ≥ 3:1.
- **Стандарт:** WCAG 2.2 (2.4.7, 2.4.11 Focus Appearance); связь TOK-07, A11Y-02.

### [A11Y-05] Доступные имена и подписи (WCAG 1.1.1 / 2.4.6 / 4.1.2)
- **Описание:** The system shall давать доступное имя каждому контролу: поля — через связанный `<label for>`, иконки-кнопки — через `aria-label`, изображения-смыслы — через `alt`; декоративные изображения — `alt=""`.
- **Rationale:** Без имён скринридер читает «кнопка, кнопка». Также помогает голосовому управлению.
- **Приоритет:** **Must**
- **Критерий приёмки:** у каждого контрола есть доступное имя; иконки-кнопки озвучиваются осмысленно; декор помечен пустым alt.
- **Стандарт:** WCAG 2.2 (1.1.1, 2.4.6, 4.1.2); связь FRM-02, CMP-06.

### [A11Y-06] Доступные модалки и оверлеи (focus management)
- **Описание:** When открывается модалка/диалог, система ДОЛЖНА перевести фокус внутрь, удержать его в пределах диалога (focus trap), вернуть на триггер при закрытии и закрывать по Esc; фон ДОЛЖЕН получать `aria-hidden`/`inert`.
- **Rationale:** Неуправляемый фокус в модалке — классический барьер. Связано с подтверждением действий (FB-03), которое часто живёт в модалке.
- **Приоритет:** **Should**
- **Критерий приёмки:** модалка ловит и возвращает фокус, закрывается по Esc, фон недоступен для Tab.
- **Стандарт:** WAI-ARIA APG (Dialog); WCAG 2.2 (2.1.2); связь CMP-06, FB-03.

### [A11Y-07] Не только цветом (WCAG 1.4.1)
- **Описание:** The system shall не передавать смысл одним лишь цветом: статусы/ошибки/категории ДОЛЖНЫ дублироваться текстом, иконкой или паттерном.
- **Rationale:** Дальтонизм + телефон на солнце. Ошибка формы «красным» без текста — недоступна.
- **Приоритет:** **Must**
- **Критерий приёмки:** ни одно состояние не различается только цветом; ошибки/статусы имеют текст/иконку.
- **Стандарт:** WCAG 2.2 (1.4.1); связь FRM-04, ST-04.

### [A11Y-08] Масштабирование и reflow (WCAG 1.4.4 / 1.4.10)
- **Описание:** The system shall сохранять работоспособность при увеличении до 200% и на узком экране (reflow до ~320–380px) без потери контента и без горизонтального скролла.
- **Rationale:** 55+ часто увеличивают шрифт системно; мобильные — узкие. Прямо связано с жалобой «на телефоне мелко».
- **Приоритет:** **Should**
- **Критерий приёмки:** при zoom 200% и на 320–380px весь контент доступен без горизонт. скролла и обрезки.
- **Стандарт:** WCAG 2.2 (1.4.4, 1.4.10); связь CMP-07, UX-09, TOK-08.

### [A11Y-09] Целевые размеры касания и уважение к движению (WCAG 2.5.8 / 2.3.3)
- **Описание:** Интерактивные цели на тач-экранах ДОЛЖНЫ быть не меньше ~24×24px (рекомендуется 44×44px) с достаточным разнесением; нефункциональная анимация ДОЛЖНА смягчаться при `prefers-reduced-motion` (через TOK-06).
- **Rationale:** Мелкие цели — барьер на телефоне и для 55+ («тыкала не туда»). Reduce-motion — вестибулярная доступность.
- **Приоритет:** **Should**
- **Критерий приёмки:** касаемые цели ≥ 24px (стремиться к 44px); reduce-motion отключает анимации.
- **Стандарт:** WCAG 2.2 (2.5.8 Target Size, 2.3.3); связь TOK-06, UX-13.

**Антипаттерны A11Y (NOT DO):**
1. `<div onclick>` вместо `<button>`.
2. `outline: none` без замены фокуса.
3. Ошибка только красной рамкой, без текста.
4. Иконка-кнопка без `aria-label`.
5. Модалка без управления фокусом и Esc.
6. Текст контрастом ниже 4.5:1 (особенно неон-на-тёмном или светло-серый).

**Чек-лист приёмки домена A11Y (WCAG 2.2 AA — минимум):**
1. Контраст текста ≥ 4.5:1, контролов ≥ 3:1, в обеих темах.
2. Весь флоу проходится с клавиатуры, без ловушек.
3. Семантический HTML; валидная иерархия заголовков; landmarks.
4. Видимый фокус везде; не перекрыт.
5. У всех контролов доступные имена; alt у изображений.
6. Модалки управляют фокусом, закрываются по Esc.
7. Смысл не только цветом.
8. Zoom 200% и reflow 320–380px — без потерь.
9. Касаемые цели ≥ 24px; reduce-motion уважается.

## 6. Домен E — Поведение форм (FRM)

**Принцип.** В FINPILOT ручной ввод — основной путь данных (в РФ нет агрегатора, KEEP-06), поэтому формы — это не второстепенная деталь, а главный рабочий инструмент. Форма ДОЛЖНА быть короткой, прощающей ошибки и понятной без эксперта. Сюда же относятся реальные баги ввода (дубль первой записи, невозможность удалить/восстановить). Опора: ISO 9241-143 (Forms), WCAG 2.2 (Labels, Error Identification/Suggestion), ISO 9241-110 (use error robustness), Нильсен №5 (предотвращение ошибок). Разворачивает UX-06.

### [FRM-01] Минимум полей и шагов; умные дефолты
- **Описание:** When пользователь вносит сущность (доход/расход/долг/цель/актив), форма ДОЛЖНА требовать минимум обязательных полей и предлагать разумные значения по умолчанию (текущая дата, последняя категория); необязательное — скрыто или помечено как опциональное.
- **Rationale:** «Вносить данные вручную» — раздражитель; для РФ это основной путь, его трение критично. Разворачивает UX-06; снижает порог онбординга (FR-10).
- **Приоритет:** **Must**
- **Критерий приёмки:** базовый ввод сущности — за минимум полей; есть дефолты (дата и т.п.); число обязательных полей онбординга ≤ 4 (связь FR-10).
- **Стандарт:** ISO 9241-143; Нильсен №7; связь UX-06, FR-10.

### [FRM-02] Видимые постоянные подписи (не только placeholder)
- **Описание:** Поля ДОЛЖНЫ иметь видимый постоянный `<label>`, связанный с контролом; placeholder НЕ ДОЛЖЕН заменять подпись (он исчезает при вводе и недоступен).
- **Rationale:** Placeholder-as-label — барьер для памяти и доступности (особенно 55+): начав ввод, человек забывает, что это за поле.
- **Приоритет:** **Must**
- **Критерий приёмки:** у каждого поля видимый label, связанный `for/id`; placeholder только как пример формата, не как единственная подпись.
- **Стандарт:** ISO 9241-143; WCAG 2.2 (3.3.2 Labels or Instructions, 1.3.1); связь A11Y-05.

### [FRM-03] Понятная подсказка формата и ограничений до ввода
- **Описание:** Where у поля есть формат/ограничение (валюта, диапазон ставки, дата), система ДОЛЖНА показать подсказку формата заранее (пример/единица измерения), а не только в момент ошибки.
- **Rationale:** Нильсен №5 — предотвращать ошибки лучше, чем сообщать о них. Связано с NFR-12 (валидация нереалистичного ввода) и заземлением (FR-02).
- **Приоритет:** **Should**
- **Критерий приёмки:** поля с форматом показывают единицу/пример (например «₽», «% годовых»); ограничения видны до отправки.
- **Стандарт:** ISO 9241-143; Нильсен №5; связь NFR-12.

### [FRM-04] Валидация: инлайн, конструктивная, не только цветом
- **Описание:** When ввод не проходит проверку, система ДОЛЖНА показать ошибку рядом с конкретным полем, текстом объяснить что не так и как исправить, и НЕ полагаться только на цвет (см. A11Y-07). Валидация СЛЕДУЕТ выполнять на blur/submit, не «крича» на каждый символ.
- **Rationale:** Ошибка «общим красным сверху» или только рамкой не помогает найти и исправить. Конструктивность = меньше отвала на форме.
- **Приоритет:** **Must**
- **Критерий приёмки:** ошибка привязана к полю, имеет текст с подсказкой исправления, дублирует цвет иконкой/текстом; фокус переводится к первому ошибочному полю.
- **Стандарт:** WCAG 2.2 (3.3.1 Error Identification, 3.3.3 Error Suggestion); ISO 9241-143; связь A11Y-07, A11Y-05.

### [FRM-05] Защита от двойной отправки (дубликаты)
- **Описание:** When пользователь отправляет форму, система ДОЛЖНА предотвращать повторную/двойную отправку (блокировка кнопки в loading + защита на сервере, напр. идемпотентность/POST-redirect-GET).
- **Rationale:** Прямо адресует BUG-01 (дубль при первом добавлении). UI-блокировки мало — нужна и серверная защита.
- **Приоритет:** **Must**
- **Критерий приёмки:** двойной клик/повтор не создаёт две записи; кнопка переходит в loading и блокируется до ответа (связь FB-02, BUG-01).
- **Стандарт:** Нильсен №5; связь BUG-01, FB-02.

### [FRM-06] Сохранение введённого при ошибке (no data loss)
- **Описание:** If отправка не удалась (валидация/сеть), then система ДОЛЖНА сохранить уже введённые значения и не очищать форму.
- **Rationale:** Потеря заполненного — сильнейший раздражитель и причина отказа. Особенно болезненно на длинных формах.
- **Приоритет:** **Must**
- **Критерий приёмки:** после ошибки все введённые поля сохранены; пользователь правит только проблемное.
- **Стандарт:** ISO 9241-110 (use error robustness); Нильсен №9; связь ST-04.

### [FRM-07] Подтверждение и обратимость деструктивных операций в формах
- **Описание:** When действие удаляет/необратимо меняет данные, система ДОЛЖНА запросить подтверждение и/или дать отмену (undo); удаление НЕ ДОЛЖНО быть «в один тык без возврата».
- **Rationale:** Прямо адресует BUG-02 (нельзя удалить накопление — обратная проблема: функция должна быть, но безопасной) и BUG-03 (нельзя восстановить удалённое). Передаёт в домен FB.
- **Приоритет:** **Must**
- **Критерий приёмки:** удаление сопровождается подтверждением или undo; удалённую запись можно вернуть (связь FB-03, BUG-02, BUG-03, UX-13).
- **Стандарт:** Нильсен №3 (контроль и свобода), №5; связь FB-03, BUG-02, BUG-03, UX-13.

### [FRM-08] Корректные типы полей и клавиатуры (мобайл)
- **Описание:** Поля ДОЛЖНЫ использовать подходящий тип/режим ввода (`inputmode="decimal"` для сумм, `type="date"` для дат), чтобы на мобильном открывалась правильная клавиатура и работала нативная валидация.
- **Rationale:** Снижает трение ввода на телефоне (UX-09) и ошибки формата.
- **Приоритет:** **Should**
- **Критерий приёмки:** числовые поля дают цифровую клавиатуру на мобильном; даты — нативный пикер; типы соответствуют данным.
- **Стандарт:** ISO 9241-143; WCAG 2.2 (1.3.5 Identify Input Purpose); связь UX-09.

**Антипаттерны FRM (NOT DO):**
1. Placeholder вместо подписи.
2. Длинная форма с кучей обязательных полей на старте.
3. Очистка формы при ошибке.
4. Удаление в один клик без подтверждения/undo.
5. Ошибка одним красным цветом без текста.
6. Нет защиты от двойной отправки (дубли).

**Чек-лист приёмки домена FRM:**
1. Минимум обязательных полей + дефолты.
2. Видимые постоянные labels (не только placeholder).
3. Подсказка формата/единиц до ввода.
4. Инлайн-валидация: текст + не только цвет + фокус на первую ошибку.
5. Защита от двойной отправки (UI + сервер).
6. Введённое сохраняется при ошибке.
7. Деструктивное — с подтверждением/undo.
8. Корректные `inputmode`/`type` для мобильной клавиатуры.

## 7. Домен F — Обратная связь на действия (FB)

**Принцип.** На каждое действие система ДОЛЖНА отвечать: показать, что действие принято, идёт, завершилось успехом или ошибкой. Молчаливый интерфейс заставляет гадать «сработало или нет» — и кликать повторно (привет дублям). Сюда же — управляемость и обратимость (undo), которых не хватало в багах удаления. Опора: ISO 9241-110 (controllability, self-descriptiveness), Нильсен №1 (видимость статуса), №3 (контроль и свобода). Разворачивает UX-13, UX-11, UX-03.

### [FB-01] Немедленная реакция на каждое действие
- **Описание:** When пользователь инициирует действие (клик, отправка, переключение), система ДОЛЖНА в течение ~100мс дать видимый отклик (нажатое состояние, переход в loading, изменение состояния).
- **Rationale:** Без мгновенного отклика интерфейс ощущается «сломанным/зависшим». Нильсен №1.
- **Приоритет:** **Must**
- **Критерий приёмки:** у каждого действия есть мгновенная визуальная реакция; нет «немых» кликов.
- **Стандарт:** ISO 9241-110; Нильсен №1; связь ST-02.

### [FB-02] Статус выполнения долгих операций (loading на источнике)
- **Описание:** While выполняется операция дольше ~300мс, инициировавший элемент (кнопка) ДОЛЖЕН перейти в loading-состояние и блокироваться от повтора; по завершении — вернуться с результатом (успех/ошибка).
- **Rationale:** Связывает FB с состояниями (ST-02) и защитой от дублей (FRM-05). Пользователь видит, что система работает.
- **Приоритет:** **Must**
- **Критерий приёмки:** кнопка/область показывает loading и не принимает повтор; результат явно отображается.
- **Стандарт:** Нильсен №1; связь ST-02, FRM-05.

### [FB-03] Подтверждение успеха и деструктивных действий + undo
- **Описание:** When действие успешно завершилось, система ДОЛЖНА подтвердить это (тост/инлайн-сообщение). When действие деструктивно (удаление), система ДОЛЖНА запросить подтверждение или предложить отмену (undo, например «Удалено · Отменить» в тосте).
- **Rationale:** Прямо закрывает BUG-02/BUG-03 (удаление без возврата) и UX-13 (обратная связь + undo). Нильсен №3 (свобода действий) и №1.
- **Приоритет:** **Must**
- **Критерий приёмки:** успешные действия подтверждаются; удаление обратимо (undo) или подтверждается; пользователь не теряет данные по случайному клику.
- **Стандарт:** Нильсен №1, №3; ISO 9241-110 (controllability); связь UX-13, FRM-07, BUG-02, BUG-03.

### [FB-04] Консистентная система уведомлений (тосты/баннеры) — неназойливая
- **Описание:** The system shall иметь единый механизм уведомлений с понятной семантикой (success/info/warning/error), авто-скрытием неважных и сохранением важных; уведомления НЕ ДОЛЖНЫ быть назойливыми и ДОЛЖНЫ поддаваться настройке частоты (для напоминаний).
- **Rationale:** «Напоминания достают» (UX-11). Единый паттерн = узнаваемость (CMP); разные ad-hoc алерты путают.
- **Приоритет:** **Should**
- **Критерий приёмки:** один компонент уведомлений с вариантами; неважные авто-скрываются; типы напоминаний настраиваются/отключаются (связь FR-11).
- **Стандарт:** Нильсен №1; ISO 9241-112; связь UX-11, FR-11, CMP-01.

### [FB-05] Доступное озвучивание динамических изменений (ARIA live)
- **Описание:** When контент меняется без перезагрузки (валидация, тост, обновление показателя), система ДОЛЖНА сообщать это вспомогательным технологиям через `aria-live`/`role="status"`/`role="alert"`.
- **Rationale:** Иначе пользователь скринридера не узнает об успехе/ошибке. Связывает FB с доступностью.
- **Приоритет:** **Should**
- **Критерий приёмки:** важные динамические сообщения объявляются скринридером; ошибки — `assertive`, статусы — `polite`.
- **Стандарт:** WCAG 2.2 (4.1.3 Status Messages); связь A11Y-03, FRM-04.

### [FB-06] Обратная связь ведёт к следующему шагу
- **Описание:** When действие завершилось, обратная связь СЛЕДУЕТ дополнять подсказкой следующего шага (после добавления дохода — «Хотите распределить эти деньги?»), а не просто «Сохранено».
- **Rationale:** Закрывает intention-action gap (FR-09) и держит флоу (IA-04). Превращает подтверждение в продолжение пути.
- **Приоритет:** **Should**
- **Критерий приёмки:** ключевые успешные действия предлагают релевантный следующий шаг.
- **Стандарт:** ISO 9241-110 (suitability for the task); связь FR-09, IA-04.

**Антипаттерны FB (NOT DO):**
1. «Немой» клик без реакции → повторные нажатия → дубли.
2. Удаление без подтверждения и без undo.
3. Зоопарк разных алертов вместо единой системы.
4. Назойливые неотключаемые уведомления.
5. Динамические изменения, невидимые для скринридера.

**Чек-лист приёмки домена FB:**
1. Мгновенный отклик (~100мс) на каждое действие.
2. Долгие операции: loading на источнике + блок повтора.
3. Успех подтверждается; деструктив — подтверждение/undo.
4. Единая неназойливая система уведомлений; напоминания настраиваемы.
5. Динамические изменения озвучиваются (`aria-live`).
6. Обратная связь предлагает следующий шаг.

## 8. Сквозной чек-лист ревью фронта (UI Definition of Done)

Этот список ревьюер прогоняет **для каждого экрана / каждого PR** перед приёмкой. Сгруппирован по доменам; отмечены Must-пункты, без которых экран не принимается.

**Фундамент (TOK)**
1. ☐ Цвета/отступы/кегли/радиусы — только из токенов (Must).
2. ☐ Видимый фокус есть, `outline:none` без замены отсутствует (Must).
3. ☐ Работает в light и dark; неон — акцент, не фон.

**Информационная архитектура (IA)**
4. ☐ Один главный элемент + ≤ 1 primary-CTA (Must).
5. ☐ Видна позиция; возврат очевиден (Must).
6. ☐ Нет тупика: есть следующий шаг (Must).
7. ☐ Интерактив отличим от статики; ключевые фичи обнаруживаются (Must).
8. ☐ Ключевые задачи — за ≤ 2 шага.

**Состояния (ST)**
9. ☐ Заданы loading/empty/error (+partial) (Must).
10. ☐ Empty = заголовок + причина + CTA (Must).
11. ☐ Error = человеческий текст + путь восстановления, без техники (Must).
12. ☐ Loading с таймаутом; нет вечного спиннера.

**Консистентность (CMP)**
13. ☐ Компоненты из единой библиотеки макросов (Must).
14. ☐ Нет формульной нотации в UI (Must).
15. ☐ Один термин = одно слово, термины полные.
16. ☐ Единый формат денег/дат/процентов.
17. ☐ Компонент адаптивен (≤ 380px без горизонт. скролла).

**Доступность (A11Y / WCAG 2.2 AA)**
18. ☐ Контраст текста ≥ 4.5:1, контролов ≥ 3:1, в обеих темах (Must).
19. ☐ Весь флоу с клавиатуры, без ловушек фокуса (Must).
20. ☐ Семантический HTML; валидные заголовки; landmarks (Must).
21. ☐ У всех контролов доступные имена; alt у изображений (Must).
22. ☐ Смысл не только цветом (Must).
23. ☐ Модалки управляют фокусом, Esc закрывает.
24. ☐ Zoom 200% / reflow 320–380px без потерь; цели ≥ 24px.

**Формы (FRM)**
25. ☐ Видимые labels (не только placeholder) (Must).
26. ☐ Инлайн-валидация: текст + не только цвет + фокус на ошибку (Must).
27. ☐ Защита от двойной отправки: UI + сервер (Must).
28. ☐ Введённое сохраняется при ошибке (Must).
29. ☐ Деструктив — подтверждение/undo (Must).
30. ☐ Минимум полей + дефолты; корректные `inputmode/type`.

**Обратная связь (FB)**
31. ☐ Мгновенный отклик (~100мс) на действие (Must).
32. ☐ Loading на кнопке-источнике + блок повтора (Must).
33. ☐ Успех подтверждается; динамика озвучивается (`aria-live`).
34. ☐ Единая неназойливая система уведомлений.
35. ☐ Обратная связь предлагает следующий шаг.

> **Правило приёмки:** все Must-пункты, релевантные экрану, закрыты — иначе экран возвращается в работу. Should/May фиксируются как замечания, не блокируют, но требуют явного решения.

## 10. Приложение: краткий глоссарий UI-терминов

Чтобы команда говорила на одном языке (поддерживает CMP-03 и реестровый DOC-01):

1. **Дизайн-токен** — именованная переменная оформления (цвет/отступ/шрифт), единый источник правды; в стеке — CSS custom property.
2. **Аффорданс** — визуальный признак того, что элемент можно использовать (кнопка выглядит нажимаемой).
3. **Состояние интерфейса** — режим области данных: loading / empty / error / partial / ideal.
4. **Skeleton** — «скелет» контента (серые плейсхолдеры) во время загрузки вместо спиннера.
5. **Progressive disclosure** — постепенное раскрытие сложности: сначала суть, детали по запросу.
6. **Progressive enhancement** — базовая работа на HTML без JS, JS — слой улучшения.
7. **Primary / secondary CTA** — главное / второстепенное призывное действие; на экране один primary.
8. **Focus trap** — удержание фокуса внутри модалки на время её открытия.
9. **Landmark** — семантическая область страницы (`<nav>`, `<main>`, `<header>`) для навигации скринридером.
10. **ARIA live region** — область, об изменениях которой вспомогательные технологии сообщают автоматически.
11. **Optimistic UI** — интерфейс показывает результат действия сразу, не дожидаясь сервера, с откатом при ошибке.
12. **Reflow** — переформатирование контента под узкий экран без горизонтального скролла.
13. **WCAG AA** — целевой уровень соответствия требованиям доступности.
14. **Toast** — временное всплывающее уведомление (успех/ошибка), часто с действием (undo).
