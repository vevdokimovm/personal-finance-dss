# QA-регламент FINPILOT

Уровни тестирования и формат фиксации дефектов.

---

## Уровни тестирования

### 1. Smoke (дымовой)
Проверка, что приложение вообще работает после изменений:
- приложение стартует (`docker compose up` / `uvicorn`);
- главные страницы отвечают 200: `/dashboard`, `/planning`, `/transactions`;
- ключевые API не падают: `POST /api/planning/calculate` на демо-данных.

### 2. Sanity (проверка вменяемости)
Основные сценарии дают разумный результат:
- загрузка демо-кейса → рекомендация совпадает с эталоном (`docs/reference_profiles.md`);
- дефицитный профиль (`mikhail`) → отказ (fail-loud), а не пустая рекомендация;
- импорт CSV → транзакции категоризируются;
- бюджет с превышением → подсвечивается.

### 3. Unit (модульные)
Автотесты бизнес-логики:
```bash
pytest                                   # все тесты
pytest --cov=app/core --cov-report=term  # с покрытием (целевое ≥ 80%)
```
Покрытие ядра — 88%. CI прогоняет их на каждый push (`.github/workflows/ci.yml`).

## Формат баг-репорта

Файл по конвенции: `BUG-v<версия>-<NN>-<краткое-описание>.md`
(пример: `BUG-v2.1.0-03-undo-not-restoring.md`).

Содержание:

| Поле | Описание |
|---|---|
| **Заголовок** | Краткая суть дефекта |
| **STR (Steps To Reproduce)** | Пошаговое воспроизведение |
| **Ожидаемо** | Что должно происходить |
| **Фактически** | Что происходит на самом деле |
| **Severity** | Критичность для системы: blocker / critical / major / minor / trivial |
| **Priority** | Срочность починки: high / medium / low |
| **Окружение** | ОС, способ запуска (Docker/локально), версия, БД |
| **Вложения** | Скриншоты, логи, дамп ответа API |

## Перед релизом

1. Smoke зелёный.
2. Sanity по эталонным профилям совпадает.
3. `pytest` зелёный, покрытие ядра ≥ 80%.
4. `ruff check app/ tests/` без ошибок.
5. `pip-audit -r requirements.txt` без известных уязвимостей.
