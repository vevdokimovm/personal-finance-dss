# Стратегия universal-парсера выписок

> **Задача (роадмап §5.5 → реализация в §7).** Покрыть импорт выписок от ~200 банков РФ **без парсера
> под каждую комбинацию банк × формат** (сотни комбинаций — неподъёмно и нежизнеспособно из-за дрейфа
> форматов). Ниже — стратегия, опирающаяся на уже написанный движок (`app/services/statement_parser.py`).
> Вселенная банков — `knowledge/product/banks_registry.md`; матрица форматов и их дрейф —
> `knowledge/product/banks_statements_reference.md`.

---

## 1. Проблема

- ~200 банков × {CSV, XLSX, PDF-текст, PDF-сетка, 1C, MT940, TXT} ≈ сотни комбинаций.
- Форматы — движущаяся мишень: банки убирают/меняют экспорт (Сбер менял PDF несколько раз за год,
  Т-Банк убрал CSV/XLSX физлицам в 2025, ВТБ убрал Excel в 2023 — см. `banks_statements_reference.md` §4).
- Парсер, жёстко завязанный на «банк X, формат Y», ломается при каждом редизайне. Ручное
  сопровождение 200 парсеров нереально для соло-разработчика.

**Ключевой инсайт исследования.** Форматы схлопываются в несколько **семейств**, а семейства
пересекаются между банками. Значит: **парсер на семейство формата, а не на банк** + движок детекции,
который направляет файл в нужное семейство по содержимому, а не по `bank_id`.

---

## 2. Текущее состояние (что уже есть)

Всё ниже — фактический код `app/services/statement_parser.py` (563 строки) на v5.8.0.

**CSV.**
- Универсальный движок `parse_universal_csv`: сам находит строку-заголовок (`_is_header_row` — есть
  колонка даты И суммы/прихода-расхода), решает кейс метаданных над таблицей (`_table_rows`, Альфа и
  пр.), матчит колонки по словарю синонимов (`_H_DATE`, `_H_AMOUNT`, `_H_DESC` — нормализованный матч
  по подстроке), разбирает число с эвристикой десятичного разделителя (`_num`: последний из `.`/`,` —
  десятичный; RU `−2 000,50` и EN `3,000.00` — оба корректно), обрабатывает split-колонки
  Приход/Расход (`_rows_to_transactions`).
- Выделенные: `parse_tinkoff_csv`, `parse_sber_csv` (авто-детект разделителя `;`/`,`).

**XLSX.** `parse_xlsx` — универсальный, тем же движком колонок.

**PDF (4 банка).**
- Текстовые: `parse_tinkoff_pdf`, `parse_sber_pdf` (regex по строкам операций).
- Табличная сетка: `parse_vtb_pdf`, `parse_raiffeisen_pdf` (через `pdfplumber.extract_tables`, разбор
  по позициям колонок — знаковая сумма у ВТБ, split Debit/Credit у Райффайзена).

**1C.** `parse_1c_exchange` — **один парсер на все бизнес-счета**: секции
`СекцияДокумент…КонецДокумента`, знак операции по тому, чей счёт (`РасчСчет` владельца) плательщик/
получатель, сумма из `Сумма=`, описание из `НазначениеПлатежа=`.

**Диспетчер.**
- `parse_bank_statement(content, bank_id)`: если контент начинается с `1CClientBankExchange` →
  `parse_1c_exchange`; иначе роутинг по `bank_id` (`BANK_PARSERS`, дефолт `parse_universal_csv`).
- `parse_bank_pdf(raw, bank_id)`: роутинг по `bank_id` (`PDF_PARSERS`, дефолт Тинькофф).
- Приём файла и дедуп — `app/api/routes_banks.py::upload_statement`.

**Вывод по заделу.** Универсальный CSV/XLSX-движок и 1C-парсер уже закрывают львиную долю: любой банк,
чья выписка — таблица с датой и суммой (или 1C-обмен), парсится **без единой строки нового кода**.
Реальная дыра — только PDF у банков, которые кроме PDF ничего не дают физлицам, и роутинг PDF по
`bank_id` (сейчас требует знать банк заранее).

## 3. Стратегия покрытия — послойно

Не 190 парсеров, а **1 движок детекции + 4 семейных парсера + расширяемый словарь колонок**.

**Слой 0 — детекция формата по содержимому (не по `bank_id`).** Новый роутер определяет семейство
сам:
- расширение + magic-байты: XLSX = ZIP (`PK\x03\x04`), PDF = `%PDF`, иначе текст;
- сигнатура текста: `1CClientBankExchange` в начале → 1C; `:20:`/`:61:` → MT940; наличие
  `;`/`,`-таблицы → CSV;
- PDF: текстовый слой есть (`pdfplumber` даёт текст) → текст/сетка; пусто → скан (нужен OCR, вне
  области — см. §6).

Это убирает зависимость «нужно заранее знать банк». `bank_id` из формы остаётся как подсказка для
меток/дедупа, но не как условие разбора.

**Слой 1 — структурированные форматы (CSV/XLSX): почти бесплатно.** Движок уже универсален. Работа —
**расширять словарь синонимов** `_H_DATE`/`_H_AMOUNT`/`_H_DESC` по мере встречи новых заголовков (это
данные, не код). Каждый новый банк с табличной выпиской → +0 парсеров.

**Слой 2 — формат 1C: бесплатно для всех бизнес-счетов.** `parse_1c_exchange` уже покрывает почти все
юрлица/ИП. Работа — только тесты на образцах разных банков (формат стандартизирован, расхождения
минимальны).

**Слой 3 — PDF, обобщение (главная работа).** Две подстратегии:
- **Табличные PDF → универсальный табличный путь.** Обобщить подход ВТБ/Райффайзена: `extract_tables`
  → строки ячеек → **тот же движок колонок, что и для CSV/XLSX** (`_is_header_row` + словарь
  синонимов + `_num`). Тогда любой банк с PDF-сеткой парсится без выделенного кода — как ещё один
  «CSV, извлечённый из PDF».
- **Текстовые PDF → регэксп + фолбэк.** Самое трудное для обобщения (Тинькофф/Сбер). Общий путь:
  `PDF → текст → построчные эвристики` (дата в начале строки, знаковая сумma в конце). Где эвристика
  не берёт — **выделенный парсер только для гигантов, которые физлицам дают исключительно PDF**
  (Сбер, Т-Банк, ВТБ — уже есть). Для остальных — деградация (Слой 4).

**Слой 4 — MT940 (опционально, корпоратив/SWIFT).** Один парсер, формат хорошо специфицирован
(`:61:` — операции, `:86:` — назначение). Низкий приоритет: у розницы почти не встречается.

**Слой 5 — деградация с пользой.** Если формат не распознан (новый редизайн, экзотический банк):
понятное сообщение + предложение выгрузить формат, который движок берёт гарантированно
(**CSV/XLSX/1C**). Это превращает «не смогли» в «сделайте так» и держит покрытие хвоста без кода под
каждый банк.

## 4. Приоритизация реализации (веха 7)

По охвату пользователей, а не по алфавиту:

1. **Слой 0 — роутер детекции.** Снимает зависимость от `bank_id`, включает universal-путь по
   умолчанию. Максимальный охват на единицу работы.
2. **Слой 3a — универсальный табличный PDF.** Обобщает уже написанные ВТБ/Райффайзен на любой
   PDF-сеткой банк.
3. **Расширение словаря колонок (Слой 1)** — по мере корпуса реальных выписок.
4. **Слой 5 — деградация** — закрывает длинный хвост без кода.
5. **Слой 3b/4 — текст-PDF гигантов и MT940** — точечно, по необходимости.

**Итог одной фразой.** Покрытие ~190 банков без 190 парсеров = **роутер детекции формата + 4 семейных
парсера (structured CSV/XLSX, 1C, табличный PDF, текст-PDF) + расширяемый словарь колонок + вежливая
деградация**. Выделенный код — только под текст-PDF гигантов, которые уже написаны.

## 5. Что тестируется в вехе 7

- **Корпус фикстур** — по одному анонимизированному образцу на семейство формата (CSV RU/EN, XLSX,
  PDF-текст, PDF-сетка, 1C, MT940). Реальные образцы — из `__screenshots__`/тест-данных, обезличенные.
- **Контрактные тесты** на каждое семейство (движок стабильно отдаёт `{amount, description, type,
  date}`), а не на каждый банк.
- **Роутер детекции** — тесты, что формат определяется по содержимому корректно (в т.ч. XLSX-как-ZIP,
  PDF-скан → «нужен OCR»).
- **Табличный PDF-путь** — что обобщённый парсер даёт тот же результат, что выделенные ВТБ/Райффайзен.
- **Дедуп при импорте** (уже есть в `routes_banks`) — не задваивать при повторной загрузке.
- **Property-based** (hypothesis, по образцу `docs/testing_infrastructure.md`): знак суммы, границы
  дат, устойчивость `_num` к разделителям.

## 6. Риски и грабли

- **Дрейф PDF-формата** (Сбер меняет часто; см. `docs/pitfalls.md`). Митигация: universal-табличный
  путь + деградация; контрактные тесты ловят регресс на образце.
- **Скан-PDF без текстового слоя** — требует OCR. **Вне области покрытия** движка: детектим и говорим
  пользователю (рус-OCR в песочнице недоступен — см. историю OCR в отчётах; на проде OCR — отдельное
  решение, не для MVP импорта).
- **Валютные операции** (ВТБ): в PDF не конвертируются в валюту счёта → итог по сумме может не
  сходиться. Не падать на несходящемся балансе; операции заносить, расхождение логировать.
- **Кодировка** (CSV/1C): CP1251 vs UTF-8. Детектить и декодировать; частая причина «кракозябр».
- **Открытый multi-line заголовок** и агрессивный autopep8 при вставке импортов (см. `docs/pitfalls.md`)
  — при доработке парсера править вручную.

## 7. Инварианты и безопасность

- **Минимум PII.** Выписка обрабатывается в памяти при загрузке; в БД идут только операции
  (`amount/description/type/date`), не сырьё выписки. Логи — без содержимого выписки (по образцу
  `email_dispatch` — только метаданные).
- **Разбор — не доверие.** Числа/даты валидируются; битые строки пропускаются, а не роняют импорт.
- **Публичное зеркало.** Реальные выписки/образцы с PII — только в приватном репо и `.gitignore`; в
  `finpilot` (public) — синтетические фикстуры.
