# Безопасность FINPILOT

Краткий security-документ: что защищаем, какие меры внедрены (со ссылками на код),
как реагировать на инцидент. Покрывает требования NFR-04, NFR-06, NFR-07, INFRA-08, INFRA-14.

---

## 1. Что защищаем

FINPILOT хранит личные финансовые данные пользователя: доходы, расходы, остатки,
кредитные обязательства и цели накопления. Это чувствительная информация — её
раскрытие или искажение наносит прямой вред. Основные угрозы:

- несанкционированный доступ к финансовым данным;
- подмена или удаление операций (целостность);
- утечка секретов (ключи БД, в будущем — банковские токены);
- межсайтовые атаки (CSRF, XSS) при работе через браузер;
- перебор и злоупотребление эндпоинтами.

> **Это единый security-документ FINPILOT:** что защищаем, что реализовано (со ссылками на код),
> что осталось, домены ревью и план реагирования на инцидент.
>
> **Контекст обязательности.** Для прод-сервиса с ПДн граждан РФ обязателен **152-ФЗ** (согласие,
> уведомление РКН, хранение данных в РФ). Тяжёлый **ГОСТ Р 57580** — только когда проект становится
> лицензированной финорганизацией под надзором ЦБ; сейчас не наш случай. Практический ориентир — **OWASP Top 10**.

С v3.0.0 приложение работает в **многопользовательском режиме с аутентификацией**: JWT в cookie,
bcrypt-хеширование паролей, блокировка после неудачных входов, изоляция данных по `user_id`
(`_owner_filter`), шифрование чувствительных полей «в покое» (Fernet). Single-user/no-auth — устаревшее
описание эпохи v2.1.0.

## 2. Внедрённые меры

| Мера | Где в коде | Что закрывает |
|---|---|---|
| **Параметризованные запросы** — только ORM, нет сырых SQL | `app/database/crud.py`, модели `app/database/models.py` | SQL-инъекции (OWASP A03) |
| **Экранирование вывода** — `escapeHtml()` для всех пользовательских строк | `frontend/static/js/app.js` (мерчанты, описания) | XSS (OWASP A03) |
| **Валидация ввода** — строгие схемы Pydantic v2 | `app/schemas/*.py` | некорректные/вредоносные данные |
| **CSRF-защита** — проверка `Origin` на изменяющих методах | `app/middleware.py::CSRFMiddleware` | CSRF (OWASP A01) |
| **Security-заголовки** — `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`, CSP | `app/middleware.py::SecurityHeadersMiddleware` | clickjacking, MIME-sniffing, утечка referrer |
| **Rate-limit** — скользящее окно по IP на чувствительных эндпоинтах | `app/middleware.py::RateLimitMiddleware` | перебор, злоупотребление (OWASP A04) |
| **Изоляция секретов** — конфигурация через `.env`, секреты не хардкодятся | `app/config.py`, `.env.example` | утечка ключей (OWASP A05) |
| **Маскирование в логах** — `mask_sensitive()` вырезает пароли/токены; в аналитику пишутся только не-PII поля | `app/services/event_logger.py` | утечка секретов и PII в логи (NFR-06) |
| **Сужение CORS** — только домены фронтенда, без `allow_origins=["*"]` | `app/main.py`, `app/config.py::cors_origins_list` | межсайтовые запросы |
| **Зависимости без известных CVE** — `pip-audit` в CI | `.github/workflows/ci.yml` | уязвимые библиотеки (OWASP A06) |
| **TLS/HTTPS** — за reverse-proxy в продакшене; HSTS включается автоматически | `SecurityHeadersMiddleware(hsts=is_production)` | перехват трафика (NFR-07) |
| **Аудит security-событий** — `csrf_blocked`, `rate_limit_exceeded` пишутся в журнал | `app/middleware.py` → `event_logger` (таблица `events`) | расследование инцидентов (INFRA-08) |
| **Отзыв сессий** — logout гасит токен по `jti`, logout-all и смена/сброс пароля сдвигают рубеж `tokens_valid_since` | `app/database/revocation.py`, `app/dependencies.py`, `app/api/routes_auth.py` | угон/утечка токена, смена учётных данных (раздел 4.4) |
| **Двухфакторная аутентификация (TOTP)** — опциональный второй фактор; recovery-коды (хеш); промежуточный токен между паролем и кодом не даёт сессии | `app/services/mfa.py`, `app/database/mfa_store.py`, `app/api/routes_auth.py` | компрометация пароля (раздел 4.4) |

## 3a. Харденинг сверх минимума (раздел 4.4)

Остаток до прод-уровня поверх ИБ-минимума. Бэкенд-задачи делаются здесь; часть завязана на
инфру/фронт и относится к вехе деплоя/фронта (помечено). Привязка к коду — в таблице §2 выше и напрямую в модулях `app/`.

- [x] **Отзыв JWT** — **сделано (v4.26.0).** `/auth/logout` реально гасит текущий токен (блок-лист по
  `jti`, таблица `revoked_tokens`); `/auth/logout-all` — выход со всех устройств (рубеж
  `users.tokens_valid_since`). Модуль `app/database/revocation.py`, миграция `0027`.
- [x] **Инвалидация сессий при смене/сбросе пароля** — **сделано (v4.27.0).** `reset-password` сдвигает
  `tokens_valid_since` → все ранее выпущенные токены умирают. `change-password` — связка как у logout-all:
  сдвиг рубежа + точечный отзыв текущего токена по `jti` (закрывает sub-секундное окно инициатора) +
  чистка auth-cookie. Option A (гасим всё, включая текущую сессию), бэкенд-only.
- [x] **Ротация ключа шифрования (Fernet MultiFernet)** — **сделано (v4.28.0).** `TokenCipher` переведён
  на `MultiFernet`: набор ключей `TOKEN_ENCRYPTION_KEYS` (PRIMARY первым), primary шифрует, остальные
  читают. Метод `rotate` + процедура `app/database/reencrypt.py::reencrypt_all` (интроспекция всех
  `EncryptedString`-полей) + CLI `scripts/reencrypt_keys.py`. Ротация без простоя: добавить новый ключ
  первым → прогнать перешифровку → убрать старый. Обратная совместимость (один ключ / derive / legacy
  plaintext) сохранена.
- [x] **Security-регрессии в тестах** — **сделано (v4.29.0).** `tests/test_security_regressions.py`
  (21 тест): заголовки безопасности на всех типах ответов (включая CSRF-403), HSTS только при включении,
  CSRF-блок на всех изменяющих методах, старт-гард. Дополнительно: `validate_production_security` теперь
  требует явный `TOKEN_ENCRYPTION_KEY(S)` в production (разделение ключей, возможность ротации).
- [x] **MFA/2FA (TOTP)** — **сделано (v4.30.0).** Второй фактор входа (pyotp). enroll → confirm
  (активация + recovery-коды) → при логине промежуточный `mfa_pending`-токен → verify (TOTP или
  recovery-код) обменивает на сессию. disable/status. Секрет зашифрован, recovery-коды хешированы
  (bcrypt). `get_current_user` отвергает purpose-токены (промежуточный/одноразовый не дают сессии).
- [ ] **Rate-limit на общий стор (Redis)** — _веха деплоя:_ сейчас in-memory (на одном инстансе
  достаточно), общий стор нужен перед мультиинстансным масштабированием.
- [ ] **CSP без `'unsafe-inline'`** — _с фронтом:_ переход на nonce-CSP, согласовать с фронтендом.
- [ ] **Cookie SameSite + CSRF-токен** — _частично с фронтом:_ double-submit CSRF-токен поверх
  Origin-проверки.

## 4. План реагирования на инцидент

При подозрении на утечку или компрометацию:

1. **Локализовать.** Отозвать/сменить скомпрометированные секреты: ключ БД, креды reverse-proxy, в будущем — банковские токены. Перевыпустить `.env`.
2. **Оценить.** Просмотреть журнал событий (`events`): отказы CSRF, превышения rate-limit, аномальные паттерны доступа.
3. **Сдержать.** При активной атаке — закрыть внешний доступ (reverse-proxy), снизить лимиты, при необходимости остановить сервис.
4. **Устранить.** Закрыть причину: обновить уязвимую зависимость (`pip-audit`), исправить код, перезапустить.
5. **Восстановить.** Поднять сервис из чистого состояния, проверить целостность данных.
6. **Уведомить.** При утечке персональных данных — действовать по 152-ФЗ (уведомление Роскомнадзора и затронутых лиц в установленные сроки).
7. **Зафиксировать.** Описать инцидент и обновить меры, чтобы он не повторился.

## 5. Тезисы для собеседования

Как в FINPILOT закрыт OWASP-минимум:

- **Инъекции** — весь доступ к БД через SQLAlchemy ORM, параметризованные запросы, сырого SQL нет.
- **XSS** — пользовательский вывод во фронте экранируется (`escapeHtml`); шаблоны Jinja2 экранируют по умолчанию.
- **CSRF** — изменяющие запросы проверяют `Origin` против белого списка; для API без cookie это корректный минимум (отсутствие `Origin` у не-браузерных клиентов CSRF-риска не несёт).
- **Секреты** — только через переменные окружения (`.env`, `python-dotenv`/`pydantic-settings`), никогда в коде; логи проходят через `mask_sensitive`.
- **Заголовки** — `X-Frame-Options`, `X-Content-Type-Options`, CSP, `Referrer-Policy`, HSTS в проде.
- **Зависимости** — `pip-audit` в CI ловит известные CVE; библиотеки обновлены.
- **Защита от перебора** — rate-limit со скользящим окном по IP.
- **Минимум привилегий** — контейнер запускается под непривилегированным пользователем (`Dockerfile`).

## 6. Домены проверки (по чему смотреть при ревью)

Любой аудит (по ГОСТ или по здравому смыслу) смотрит одни и те же домены — здесь они с нашим статусом:

| Домен | Статус в FINPILOT |
|---|---|
| Управление доступом | ✅ JWT + lockout + `_owner_filter` + admin-key |
| Защита данных (шифрование) | ✅ Fernet в покое; 🔴 TLS в транзите — на проде |
| Аутентификация | ✅ хеш паролей (bcrypt) + MFA/TOTP (v4.30.0) |
| Журналирование/мониторинг | ✅ события + PII-аудит; 🟢 алертинг — Sentry на проде |
| Защита от типовых атак | ✅ ORM / CSRF / заголовки / XSS / валидация / rate-limit |
| Управление уязвимостями | ✅ pip-audit жёстким гейтом в CI; 🟡 cadence / Dependabot |
| Реагирование на инциденты | ✅ план — §4 выше; постмортемы — `docs/reports/incidents/` |
