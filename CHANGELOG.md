## [4.17.0] — 2026-06-27 — P3.5: A/B-тестирование (прод-уровень) + рефактор доков-разборов (MINOR)

Новая возможность — фреймворк A/B-экспериментов прод-уровня (детерминированно, без ML): эксперименты
управляются через админ-API (создать → running → stopped), пользователь стабильно попадает в один
вариант с **фиксацией назначения** (смена конфигурации не перекидывает уже назначенных), результаты
считаются по конверсии на событийном логе. Версия — **MINOR**: это фича, а не патч (исправляем дрейф
линии 4.16.x, где патчем шли в т.ч. фичи; см. WATCHLOG §6 #4).

### Added
- **Ядро** `app/core/experiments.py`: `assign_variant(experiment_key, subject_id, variants)` —
  детерминированное назначение стабильным хешем `sha256(key:subject) % Σweight` по кумулятивным весам.
  Чистая функция, без БД и без ML.
- **Модели + миграция 0021:** `Experiment` (key уникальный, status draft/running/stopped, variants
  JSON `[{name, weight}]`, conversion_event) + `ExperimentAssignment` (`UNIQUE(experiment_id,
  subject_id)` — лочит назначение).
- **Сервис** `app/services/experiments.py`: `get_or_assign_variant` (резолв subject → назначение с
  фиксацией + лог exposure; гонка через UNIQUE + повторное чтение), `create/get/list/update/delete`,
  `validate_variants`.
- **Измерение** `analytics.experiment_results(db, key)` — по варианту assigned/converted/rate;
  конверсия = subject (user_id или session_id) с событием `conversion_event`.
- **Эндпоинты** `app/api/routes_experiments.py`: публичный `GET /api/experiments/{key}/variant`
  (subject = user_id или `sid`); админ под `require_admin` — `POST/GET/PATCH/DELETE
  /api/admin/experiments`, `GET /api/admin/experiments/{key}/results`.
- **Фронт** (`app.js`, `index.html`): `fpExperiments` (стабильный subject через localStorage +
  получение варианта) + демо-эксперимент на подзаголовке сводки (graceful control, если нет running).
- **E2E-спека** `tests/e2e/test_experiments_e2e.py` — полный цикл через HTTP (создать → вариант →
  результаты). Гоняется на Mac/CI.
- **Доки:** `docs/report_types.md` (роутер: какой тип разбора писать), `docs/incident_postmortem_guide.md`
  (шаблон инцидента), `docs/documentation_methodology.md` (общая теория, внесена из Claude-проекта).

### Changed
- **Рефактор доков-разборов:** `docs/incident_postmortem.md` (был смесью роутера и шаблона) расщеплён
  на `report_types.md` + `incident_postmortem_guide.md`; старый файл удалён, все ссылки обновлены.
- **Чистка внешних ссылок:** в репо-доках дангл-имена внешних разборов `FINPILOT_*_incident/investigation.md`

  `FINPILOT_UI_UX_Design_Standard.md` → `docs/ui_ux_design_standard.md`.
- **WATCHLOG §6 #4** (SemVer): заострено MINOR (фича) / PATCH (мелочь) / MAJOR (слом); линия 4.16.x
  задним числом не переписывается.

### Tests

## [4.16.34] — 2026-06-27 — P2.7: обучение категоризации на правках пользователя (MINOR-уровень, патч-линия)

Детерминированная адаптация категоризатора под пользователя (без ML/LLM): ручное переназначение
категории операции запоминается как правило и применяется к будущим импортам и ретроактивно к
существующим совпадающим операциям. По факту у транзакций не было update-пути (только
create/delete/restore) — поэтому батч включает и переназначение, и обучение. Версия — патч в линии
4.16.x по сложившейся практике P2.4–P2.6.

### Added
- **`UserCategoryRule`** (`user_category_rules`, миграция 0020): правило `match_token -> category`
  на пользователя, `UNIQUE(user_id, match_token, type)`. `match_token` хранится нормализованным.
- **Ядро** `core/categorization.py`: `normalize_match_key` (casefold + схлопывание пробелов) и
  `classify_with_rules` — приоритет пользовательских правил над дефолтным движком, матч по substring
  нормализованного токена с guard минимальной длины (3 символа). Правила передаются снаружи списком,
  поэтому ядро не зависит от БД.
- **CRUD** (`crud.py`): `upsert_category_rule`, `get_category_rules`, `get_user_category_rules`,
  `set_transaction_category` (с авто-резолвом `category_id` по имени+типу), `apply_category_rule`
  (ретроактив + счётчик изменённых), `delete_category_rule`. Изоляция по `user_id` через `_owner_filter`.
- **Эндпоинты** (`routes_transactions.py`): `POST /api/transactions/{id}/category` (переназначение +
  обучение + опц. ретроактив, возвращает операцию/правило/`updated_count`), `GET /api/category-rules`,
  `DELETE /api/category-rules/{id}` (204).
- **UI** (`transactions.html`, `app.js`): кнопка ↻ на строке операции → модалка переназначения
  (категория + редактируемый match-токен + чекбокс «применить к похожим»); секция «Выученные правила»
  со списком и удалением.
- **E2E-спека** `tests/e2e/test_category_learning_e2e.py` (переназначение через UI → обучение,
  ретроактив, удаление правила). Гоняется на Mac/CI (браузер в песочнице недоступен).

### Changed
- `create_transaction` и `bulk_create_transactions` учитывают пользовательские правила при
  авто-категоризации (явный ввод категории по-прежнему приоритетнее правила). Для bulk правила
  префетчатся одним запросом — без N+1.

### Tests
- `tests/test_category_learning.py` — 27 тестов: ядро (хит/мисс/guard/explicit-бьёт-правило),
  CRUD, set/apply + счётчик, импорт уважает правила, **изоляция per-user**, эндпоинты.
- `tests/test_api_crud.py` — 4 edge-теста на ранее непокрытые ветки эндпоинтов операций (фильтры дат
  CSV-экспорта, 404 удаления/восстановления) → `routes_transactions.py` покрыт на 100%.

### Notes
- Регрессия: **719 passed**, 0 падений (SQLite). Покрытие: **91.9%** (гейт 90%). PG-матрица: миграция
  0020 + фича зелены на PostgreSQL 16.
- Синтез с черновым планом S: взята его доказанная архитектура (таблица правил + classify_with_rules +
  эндпоинты управления правилами + ретроактив), добавлены редактируемый match-токен с guard
  (срабатывает на варьирующихся описаниях и не врёт) и обязательный тест изоляции per-user.
- `APP_VERSION` → `4.16.34`.

нет по решению от 2026-06), поэтому он должен быть надёжным. Парсеры (Тинькофф/Сбер/универсальный
CSV + Тинькофф PDF) и обработка кодировок (включая CP1251) уже были; здесь закрыт ключевой пробел.

### Дедупликация (главное)
- Повторная загрузка той же выписки больше НЕ задваивает транзакции (раньше — задваивала, что
  искажало всю аналитику). Дубль определяется по ключу (дата, сумма, тип, описание) в рамках
  владельца. Ключи существующих операций загружаются один раз, проверка в памяти — O(1) на строку,
  без N запросов на большом файле.
- Дубли внутри одного файла тоже отсекаются.
- Отчёт импорта дополнен `skipped_duplicates`; сообщение показывает, сколько пропущено.

### Лимит размера файла
- `MAX_UPLOAD_SIZE_MB` (10 МБ по умолчанию): слишком большой файл отклоняется с понятной ошибкой
  — защита воркера от чтения гигантских загрузок в память.

### Тесты
- `tests/test_import_dedup.py` (5): повторный импорт не плодит дубли, дубли внутри файла,
  новые строки после повторного импорта всё равно добавляются, отчёт содержит счётчик,
  превышение размера отклоняется. Итого 344 passed.

## [4.4.0] — 2026-06-20 — Денежная политика Decimal на денежных путях (MINOR)

P1.4 из roadmap. При реализации выяснилась важная деталь, определившая границу:
денежные величины в движке (`Rt_new`, `δP`) одновременно являются рублями и входами
безразмерной оптимизации `U(a)` (нормализация `Rt_new`/`Dt_new` в `ranking.py`).
Поэтому Decimal проведён по денежным путям ввода/вывода, а ядро выбора оставлено float.

### Граница (обоснование)
- Источник истины — БД: деньги уже `Numeric(14,2)` (Decimal).
- Денежные ПУТИ ввода/вывода — Decimal с явным `ROUND_HALF_UP` до копеек.
- Ядро ВЫБОРА (нормализация, веса, `U(a)`) — float: оно безразмерно (решает, какую долю
  выбрать, а не считает копейки), и Decimal там не добавил бы точности, но рисковал бы
  сдвинуть решение. Граница защищена golden master.

### Реализация
- `app/core/money.py` — единая политика: `to_money(value) -> Decimal` (quantize 0.01,
  ROUND_HALF_UP; преобразование через `str()` исключает двоичную неточность float) и
  `money(value) -> float` для мест, где исторически ожидается float.
- CRUD: денежные суммы при записи проходят `to_money` (создание/пополнение
  obligation/goal/transaction/budget/liquid_asset, взносы и платежи) — предсказуемое
  округление на денежном пути ввода, независимое от драйвера БД.
- Движок: отображательные денежные поля (`indicators`, `avalanche_detail`) переведены на
  единую money-политику вместо россыпи `round(float, 2)`. Величины, входящие в выбор
  (`x_obligations`, `Rt_new`, raw `x_obl_unused`), не тронуты.

### Гарантия неизменности решений
- `tests/test_engine_golden.py` (5) — характеризационный golden master: имя выбранной
  альтернативы, доли и порядок топ-3 — строгие инварианты; деньги с допуском в копейку;
  безразмерная `U(a)` — точно. Зелёный и до, и после рефактора.
- Итого 326 passed (SQLite); денежные CRUD + golden зелёные на живом PostgreSQL 16.

  тихий no-op (как остальные письма), поток проходится по dev-ссылке.

- Страница `/forgot-password` — форма ввода email.

### Тесты

## [3.8.0] — 2026-06-18 — Refined-модель v3.0.0: ортогональные критерии и качество советов (MINOR)

Крупная переработка математического ядра ради качества и корректности рекомендаций для реальных пользователей. Меняет поведение алгоритма (выбор плана), API обратно совместим.

### Изменено
- **Ликвидность переведена на stock-based: `Lt = (Bliq + резерв) / расходы` — месяцы автономии.** Раньше `Lt = Rt/(Σe+ΣP)` была линейной функцией ресурса `Rt`, то есть два из четырёх критериев свёртки несли одну и ту же информацию (корреляция ≈ 0.9998) — профили риска различались лишь номинально. Теперь `Rt` реагирует на досрочку долга, а `Lt` — на пополнение резерва: критерии ортогональны, и SAW-свёртка реально различает профили. Проверено end-to-end: консервативный профиль направляет свободные деньги в резерв, агрессивный — в цели.
- **Отсев альтернатив — только жёсткие инварианты безопасности:** `Rt' ≥ 0` (не уводим бюджет в минус) и `Dt' ≤ 0.40` (ПДН Банка России). Порог ликвидности переосмыслен как «минимум месяцев автономии» и по умолчанию выключен (`0`). Причина: прежний жёсткий порог на flow-коэффициенте отсекал почти все варианты у пользователей с тонким бюджетом и выдавал отказ вместо совета.
- **Порог Avalanche `r_bench` стал динамическим:** ключевая ставка ЦБ РФ × (1 − НДФЛ) — посленалоговая доходность безрисковой альтернативы. При ключевой 16% и НДФЛ 13% даёт ≈ 0.139, что совпадает с прежним дефолтом 0.14, но адаптируется при изменении ставки. Приоритет источника: явный из запроса → реальная ставка активов пользователя → ключевая ЦБ после налога → `prefs`. При недоступности cbr.ru — фолбэк, расчёт не блокируется.
- **Шаг дискретизации альтернатив 20% → 10%:** 66 вариантов вместо 21, более тонкие рекомендации по распределению почти без потери скорости.

### Добавлено
- Кэширование недоступности cbr.ru (окно 15 мин): сервис ключевой ставки не бьётся в сеть на каждом запросе рекомендации при заблокированном доступе.
- `cbr_rate.get_opportunity_cost_rate()` — расчёт `r_bench` из ключевой ставки.
- Тесты: stock-based ликвидность и ортогональность R/L, дифференциация профилей в ранжировании, устойчивость нормализации к вырожденным множествам (нет долга/целей — деления на ноль нет), расчёт и фолбэк `r_bench`, кэш неудачи cbr (всего 247 тестов).

### Документация
- **Математическая модель v3.0.0** (`docs/math_model_v3_0_0.md`) — полная актуальная спецификация refined-модели: ортогональные критерии, stock-based ликвидность, динамический `r_bench`, обоснование всех параметров с точки зрения качества советов.

- **График прогноза R_t** выходил за рамки: подписи оси Y печатались полными числами (−889 241) и не вмещались. Теперь компактный формат (k/M), больше места слева, доверительный коридор заметнее (полоса + чёткие границы).

### Изменено
