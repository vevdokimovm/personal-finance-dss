## [4.9.2] — 2026-06-20 — Fix: не работала кнопка удаления обязательства (PATCH)

Крестик «✕» на карточке обязательства не удалял её. Причина: карточка — это `<details>`,
и на кнопке стоял `onclick="event.stopPropagation()"` (чтобы клик по крестику не
раскрывал/сворачивал карточку). Но удаление навешано делегированием на `#obligations-list`
и срабатывает по всплытию события — `stopPropagation` это всплытие глушил, и клик не доходил
до обработчика. Крестик гасил сам себя.

- Замена на `onclick="event.preventDefault()"`: отменяет раскрытие `<details>`, но всплытие
  сохраняет — делегированный обработчик удаления снова срабатывает.
- Бэкенд DELETE обязательства был исправен (204, каскад) — баг был чисто фронтовый. Кнопка
  удаления цели работала, потому что её карточка — обычный `<div>` без toggle.

## [4.7.0] — 2026-06-20 — Надёжность импорта выписок: дедупликация (MINOR)

P2.1 из roadmap (старт P2). Импорт — основной путь поступления данных (прямой банк-интеграции
нет по решению от 2026-06), поэтому он должен быть надёжным. Парсеры (Тинькофф/Сбер/универсальный
CSV + Тинькофф PDF) и обработка кодировок (включая CP1251) уже были; здесь закрыт ключевой пробел.

### Дедупликация (главное)
- Повторная загрузка той же выписки больше НЕ задваивает транзакции (раньше — задваивала, что
  искажало всю аналитику). Дубль определяется по ключу (дата, сумма, тип, описание) в рамках
  владельца. Ключи существующих операций загружаются один раз, проверка в памяти — O(1) на строку,
  без N запросов на большом файле.
- Дубли внутри одного файла тоже отсекаются.
- Отчёт импорта дополнен `skipped_duplicates`; сообщение показывает, сколько пропущено.

### Лимит размера файла
- `MAX_UPLOAD_SIZE_MB` (10 МБ по умолчанию): слишком большой файл отклоняется с понятной ошибкой
  — защита воркера от чтения гигантских загрузок в память.

### Тесты
- `tests/test_import_dedup.py` (5): повторный импорт не плодит дубли, дубли внутри файла,
  новые строки после повторного импорта всё равно добавляются, отчёт содержит счётчик,
  превышение размера отклоняется. Итого 344 passed.

## [4.4.1] — 2026-06-20 — Fix: docker-compose поднимается из коробки (PATCH)

`docker compose up --build` падал на интерполяции `JWT_SECRET: ${JWT_SECRET:?...}` —
обязательная переменная без `.env` валила запуск ещё до старта контейнера. Это была
лишняя строгость на уровне compose для локального запуска.

- docker-compose: дев-дружественные дефолты — `ENVIRONMENT=development`, `COOKIE_SECURE=false`,
  дев-`JWT_SECRET`. Теперь `docker compose up` поднимается локально по HTTP без `.env`.
- Безопасность прода — на правильном слое (приложение): дефолтный/короткий `JWT_SECRET` или
  `COOKIE_SECURE=false` при `ENVIRONMENT=production` по-прежнему валят старт (fail-loud).
  Для прода переопредели в `.env` (см. `.env.example`).

## [4.4.0] — 2026-06-20 — Денежная политика Decimal на денежных путях (MINOR)

P1.4 из roadmap. При реализации выяснилась важная деталь, определившая границу:
денежные величины в движке (`Rt_new`, `δP`) одновременно являются рублями и входами
безразмерной оптимизации `U(a)` (нормализация `Rt_new`/`Dt_new` в `ranking.py`).
Поэтому Decimal проведён по денежным путям ввода/вывода, а ядро выбора оставлено float.

### Граница (обоснование)
- Источник истины — БД: деньги уже `Numeric(14,2)` (Decimal).
- Денежные ПУТИ ввода/вывода — Decimal с явным `ROUND_HALF_UP` до копеек.
- Ядро ВЫБОРА (нормализация, веса, `U(a)`) — float: оно безразмерно (решает, какую долю
  выбрать, а не считает копейки), и Decimal там не добавил бы точности, но рисковал бы
  сдвинуть решение. Граница защищена golden master.

### Реализация
- `app/core/money.py` — единая политика: `to_money(value) -> Decimal` (quantize 0.01,
  ROUND_HALF_UP; преобразование через `str()` исключает двоичную неточность float) и
  `money(value) -> float` для мест, где исторически ожидается float.
- CRUD: денежные суммы при записи проходят `to_money` (создание/пополнение
  obligation/goal/transaction/budget/liquid_asset, взносы и платежи) — предсказуемое
  округление на денежном пути ввода, независимое от драйвера БД.
- Движок: отображательные денежные поля (`indicators`, `avalanche_detail`) переведены на
  единую money-политику вместо россыпи `round(float, 2)`. Величины, входящие в выбор
  (`x_obligations`, `Rt_new`, raw `x_obl_unused`), не тронуты.

### Гарантия неизменности решений
- `tests/test_engine_golden.py` (5) — характеризационный golden master: имя выбранной
  альтернативы, доли и порядок топ-3 — строгие инварианты; деньги с допуском в копейку;
  безразмерная `U(a)` — точно. Зелёный и до, и после рефактора.
- Итого 326 passed (SQLite); денежные CRUD + golden зелёные на живом PostgreSQL 16.

## [4.3.0] — 2026-06-20 — Восстановление пароля + SMTP (MINOR)

### Поток восстановления (API)
- `POST /api/auth/forgot-password` ({email}) — запрос сброса. Ответ одинаков независимо от

  Принимает только токен с назначением `password_reset` (короткий TTL), иначе 400.
  Успешный сброс снимает блокировку аккаунта (`failed_login_attempts` / `locked_until`).

  как у подтверждения email). TTL — `PASSWORD_RESET_TTL_HOURS` (по умолчанию 1 час).

### Письмо
- `EmailService.send_password_reset` — письмо со ссылкой (text + HTML). Без SMTP-конфига —
  тихий no-op (как остальные письма), поток проходится по dev-ссылке.

### UI (полная петля)

- Страница `/reset-password?token=...` — форма нового пароля с проверкой совпадения и длины,

- Ссылка «Забыли пароль?» в модалке входа.

### Тесты

### Вне зоны кода

## [4.2.0] — 2026-06-20 — Безопасность входа, fail-loud конфиг, PostgreSQL-готовность (MINOR)

P1.2 из roadmap: защита логина уровня реального финтеха + честная верификация на боевой
СУБД. Весь батч прогнан и на SQLite (315 passed), и на живом PostgreSQL 16.

### Account lockout — защита от перебора пароля (P1.2, NFR-05)
- Поля `users.failed_login_attempts` и `users.locked_until` (миграция `0014`).
- После `LOGIN_MAX_ATTEMPTS` (5) неудачных входов аккаунт блокируется на
  `LOGIN_LOCKOUT_MINUTES` (15). Блокировка действует даже при верном пароле (HTTP 429),
  успешный вход сбрасывает счётчик. CRUD-хелперы `register_failed_login` / `reset_failed_logins`.
- Хранение в БД (а не in-memory): переживает рестарт и консистентно между воркерами gunicorn.

### Fail-loud конфигурация production (P1.2)
- `validate_production_security()` в `app/config.py`: при `ENVIRONMENT=production` приложение
  НЕ стартует с дефолтным/коротким (<32 симв.) `JWT_SECRET` или с `COOKIE_SECURE=false`.
- Проверка встроена в lifespan `app/main.py` — падает на старте с понятной ошибкой.

### Валидация сумм — закрыт BUG-018
- Отрицательные денежные значения отклоняются (HTTP 422) в схемах `GoalCreate`,
  `ObligationCreate`, `LiquidAssetCreate/Update` (`Field(ge=0)` на amount/target/rate/payment).

### PostgreSQL: переход на боевую СУБД (P1.5, частично)
- Весь БД-критичный набор прогнан на живом PostgreSQL 16 — вскрыты и исправлены два расхождения,
  которые SQLite прятал (FK по умолчанию не форсятся):
  - **Реальный баг**: удаление цели/обязательства с историей (`goal_contributions` /
    `obligation_payments`) падало с FK violation. `delete_goal` / `delete_obligation` теперь
    удаляют дочерние строки явно (кросс-СУБД, без хрупкой миграции имён constraint).
  - **Полное удаление аккаунта** (152-ФЗ): `delete_user` дополнительно вычищает историю,

## [3.8.0] — 2026-06-18 — Refined-модель v3.0.0: ортогональные критерии и качество советов (MINOR)

Крупная переработка математического ядра ради качества и корректности рекомендаций для реальных пользователей. Меняет поведение алгоритма (выбор плана), API обратно совместим.

### Изменено
- **Ликвидность переведена на stock-based: `Lt = (Bliq + резерв) / расходы` — месяцы автономии.** Раньше `Lt = Rt/(Σe+ΣP)` была линейной функцией ресурса `Rt`, то есть два из четырёх критериев свёртки несли одну и ту же информацию (корреляция ≈ 0.9998) — профили риска различались лишь номинально. Теперь `Rt` реагирует на досрочку долга, а `Lt` — на пополнение резерва: критерии ортогональны, и SAW-свёртка реально различает профили. Проверено end-to-end: консервативный профиль направляет свободные деньги в резерв, агрессивный — в цели.
- **Отсев альтернатив — только жёсткие инварианты безопасности:** `Rt' ≥ 0` (не уводим бюджет в минус) и `Dt' ≤ 0.40` (ПДН Банка России). Порог ликвидности переосмыслен как «минимум месяцев автономии» и по умолчанию выключен (`0`). Причина: прежний жёсткий порог на flow-коэффициенте отсекал почти все варианты у пользователей с тонким бюджетом и выдавал отказ вместо совета.
- **Порог Avalanche `r_bench` стал динамическим:** ключевая ставка ЦБ РФ × (1 − НДФЛ) — посленалоговая доходность безрисковой альтернативы. При ключевой 16% и НДФЛ 13% даёт ≈ 0.139, что совпадает с прежним дефолтом 0.14, но адаптируется при изменении ставки. Приоритет источника: явный из запроса → реальная ставка активов пользователя → ключевая ЦБ после налога → `prefs`. При недоступности cbr.ru — фолбэк, расчёт не блокируется.
- **Шаг дискретизации альтернатив 20% → 10%:** 66 вариантов вместо 21, более тонкие рекомендации по распределению почти без потери скорости.

### Добавлено

## [3.0.0] — 2026-06-12 — Международный / многопользовательский режим (MAJOR)

Ломающий релиз: схема БД расширена (`users`, FK `user_id`, денежные поля → `numeric`),
появилась аутентификация и изоляция данных по пользователю. Логика ядра СППР
(SAW + Avalanche + Monte-Carlo) не изменена — KEEP-01..05 сохранены.

### Added
- **Аутентификация (INFRA-06, NFR-05, DATA-03):** модель `User` (uuid), bcrypt-хеши,
  JWT (httpOnly-cookie + Bearer), маршруты `/api/auth/{register,login,logout,me}`.
- **Изоляция данных:** все пользовательские сущности получили FK `user_id`; роуты
  фильтруют по текущему пользователю. Анонимный режим (без токена) видит только
  данные без владельца.
- **Single→multi миграция без потери:** первый зарегистрированный пользователь
  усыновляет данные анонимного режима (`adopt_orphan_rows`).
- **Мультивалюта (FR-19, DATA-08):** поле `currency` на сущностях, `base_currency`
  у пользователя, таблица `fx_rates` (USD-пивот), `CurrencyConverter`, маршруты
  `/api/fx/{rates,convert}`.
- **Ingestion-слой (DATA-07, KEEP-07):** каноническая модель `FinancialSnapshot`,
  Protocol-контракты (`FinanceEngine`, `FinancialDataProvider`, `TokenStore`,
  `PlaidClient`), адаптер реального ядра `CoreFinanceEngine` (INFRA-15, REFACTOR-04).
- **Provider: Manual с персистентностью (INFRA-18):** `manual_snapshots`, репозиторий
  ручного ввода переживает рестарт.
- **Provider: Plaid (FR-18, INFRA-16/17):** `EncryptedTokenStore` (Fernet-шифрование
  токенов «в покое»), `PlaidProvider`, ленивый `RealPlaidClient`, маршруты
  `/api/plaid/{exchange,sync}`. Опционально — активируется `PLAID_*`.
- **B2B API (FR-23):** `/v1/analyze` принимает канонический снимок партнёра и
  возвращает рекомендацию; авторизация по `X-API-Key`. Активируется `B2B_API_KEYS`.
- **UI:** виджет входа/профиля в шапке, модалка вход/регистрация, `auth.js`
  (Bearer-перехват fetch).
- **Миграция 0009:** users, FK, numeric-типы, fx_rates, ingestion-таблицы; upgrade/downgrade.
- **Тесты:** `tests/test_auth.py` (изоляция, усыновление, хеши), `tests/test_international.py`
  (валюта, ingestion-движок, шифрование, B2B).

### Changed
- Денежные поля: `Float` → `Numeric(14,2)`, ставки → `Numeric(6,4)` во всех таблицах
  (точность денег, устранение ошибок округления).
- `UserPrefs`: одна строка на пользователя (`user_id` unique FK) + `base_currency`.
- `events`/`recommendations`: `user_id` `int` → `uuid`-строка.
- CSRF-middleware пропускает запросы по Bearer/API-Key (нет амбиентных cookie → нет CSRF-риска).
- Rate-limit распространён на `/api/auth/login`, `/api/auth/register`, `/v1/analyze`.
- `APP_VERSION` → `3.0.0`.

### Security
- Пароли — только bcrypt-хеш с солью (NFR-05).
- Plaid-токены шифруются Fernet «в покое» и не утекают в логи/URL (NFR-06, INFRA-17).
- Новые секреты (`JWT_SECRET`, `PLAID_*`, `TOKEN_ENCRYPTION_KEY`) — только через `.env`.
