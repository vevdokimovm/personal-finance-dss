# Комплексный регламент версионирования, проведения исследований и инженерного тестирования (UX/QA)

Настоящая инструкция является официальным внутренним стандартом управления жизненным циклом программного комплекса **FINPILOT** (Интерпретируемая СППР в сфере управления персональными финансами). Регламент определяет правила семантического версионирования, порядок проведения и документирования юзабилити-исследований (UX), а также стандарты обеспечения качества (QA) программного кода бэкенда на FastAPI и структуры базы данных.

---

## 1. Система семантического версионирования (Semantic Versioning 2.0.0)

Контроль версий программного обеспечения и всех сопутствующих аналитических артефактов строится на строгом соблюдении спецификации **SemVer 2.0.0**. Идентификатор имеет структуру `vMAJOR.MINOR.PATCH` (например, `v1.2.0`).

### 1.1. Триггеры изменения компонентов версии

#### MAJOR (Мажорная версия)

Инкремент мажорного номера версии происходит исключительно при внесении **обратнонесовместимых изменений** в архитектуру данных, API-слой или математическое ядро принятия решений.

**Критерии и примеры:**

- Изменение фундаментального математического аппарата оптимизации (например, переход со статического многокритериального метода ранжирования SAW на стохастические модели, динамическое программирование или нейросетевые модели).
- Изменение реляционной схемы базы данных (СУБД SQLite via SQLAlchemy ORM), нарушающее целостность старых пользовательских сохранений и требующее ручного написания деструктивных миграций.
- Кардинальный рефакторинг архитектуры приложения (например, миграция с монолитного FastAPI бэкенда на распределённую микросервисную архитектуру с использованием шины данных).

**Правило сброса:** при увеличении `MAJOR` компоненты `MINOR` и `PATCH` жёстко сбрасываются в `0` (например: `v1.8.4` → `v2.0.0`).

#### MINOR (Минорная версия)

Инкремент минорного номера происходит при добавлении **новой функциональности**, которая полностью сохраняет обратную совместимость с предыдущим программным кодом текущего мажорного релиза.

**Критерии и примеры:**

- Интеграция нового независимого аналитического модуля (например, разработка блока прогнозирования кэш-флоу методом Монте-Карло или экспоненциального сглаживания SES).
- Расширение RESTful API (добавление новых роутеров и эндпоинтов в FastAPI для работы с финансовыми целями или транзакциями).
- Разработка подсистемы экспорта данных и сгенерированных СППР финансовых рекомендаций в форматы PDF, XLSX или CSV.
- Обновление фронтенд-слоя (внедрение новых интерактивных дашбордов и графиков в Jinja2-шаблоны).

**Правило сброса:** при увеличении `MINOR` компонент `PATCH` сбрасывается в `0` (например: `v1.2.5` → `v1.3.0`).

#### PATCH (Патч-версия)

Инкремент патч-версии происходит при исправлении **обнаруженных программных дефектов** (bug fixes), оптимизации производительности модулей или внесении косметических изменений в интерфейс, не затрагивающих бизнес-логику системы.

**Критерии и примеры:**

- Исправление логических ошибок в формулах нормализации критериев или расчёте функции полезности $U(a)$ в алгоритме SAW.
- Устранение исключений времени выполнения (например, перехват и обработка `ZeroDivisionError` при расчёте коэффициента долговой нагрузки DTI или коэффициента ликвидности для пустых профилей).
- Косметические исправления UI: корректировка стилей CSS, изменение шрифтовых сеток, исправление опечаток в текстах подсказок и объяснений (XAI).
- Оптимизация работы с БД: добавление индексов, исправление сырых SQL-запросов или оптимизация подгрузки связанных сущностей в SQLAlchemy.

**Правило изменения:** изменяется строго последовательно инкрементом на единицу (`v1.3.0` → `v1.3.1` → `v1.3.2`).

### 1.2. Инженерная интеграция версионирования в проект

1. **Конфигурационный файл кода.** Текущая рабочая версия фиксируется в коде бэкенда единой строковой константой в файле `src/app/config.py` (или `main.py`):

   ```python
   # src/app/config.py
   APP_VERSION = "1.0.3"
   ```

2. **Отображение в интерфейсе пользователя (UI).** Значение константы `APP_VERSION` в обязательном порядке прокидывается через контекст рендеринга Jinja2 во все базовые HTML-шаблоны фронтенда и выводится в правом нижнем углу футера: `FINPILOT v1.0.3`. Это необходимо для верификации сборки при анализе скриншотов юзабилити-тестов и логов ошибок.

3. **Фиксация в Git (тегирование).** Любое перемещение кода на этап тестирования или релиз в ветку `main` сопровождается созданием аннотированного тега:

   ```bash
   git tag -a v1.0.3 -m "Release version 1.0.3: Patch DTI calculation overflow and optimize SQLite session pooling"
   git push origin v1.0.3
   ```

## 2. Юзабилити-тестирование (UX Testing & Research)

Юзабилити-тестирование направлено на эмпирическую проверку доступности интерфейса, удобства ввода финансовых данных и интерпретируемости объяснений (Explainable AI), генерируемых алгоритмом ранжирования стратегий.

### 2.1. Регламент деперсонализации и шифрования респондентов

В целях обеспечения конфиденциальности и исключения субъективных когнитивных искажений использование реальных персональных данных (ФИО, контакты) респондентов в протоколах тестирования **категорически запрещено**.

- **Шаблон кодирования ID:** `R[index]` (двузначный индекс с ведущим нулём). Пример: `R01`, `R02`, `R12`.
- **Паспорт респондента:** метаданные пользователей ведутся в изолированной мастер-таблице исследователя, а в протокол выносятся только агрегированные атрибуты — возраст, уровень финансовой грамотности, текущая долговая нагрузка (коэффициент DTI).

### 2.2. Именование файлов протоколов

Каждая тестовая сессия документируется отдельным файлом формата Markdown (`.md`), который сохраняется в директорию репозитория `/research/usability/protocols/`.

- **Шаблон имени файла:** `UT-[Фокус]-[Версия]-R[ID].md`
- **Допустимые маркеры фокуса теста:**
  - `A` (Data Acquisition) — тестирование интерфейсов ввода доходов, расходов, долгов и целей.
  - `R` (Recommendations) — тестирование экрана выдачи рекомендаций.
  - `F` (Full Flow) — сквозной тест системы от регистрации до выполнения финансового плана.
- **Пример корректного имени файла:** `UT-F-v1.0.3-R02.md`

### 2.3. Унифицированный шаблон индивидуального протокола


## 3. Обеспечение качества программного обеспечения (QA Testing)

Процесс QA направлен на обеспечение детерминированности расчётов математического ядра, стабильности FastAPI эндпоинтов и предотвращение регрессионных ошибок при обновлении версий.

### 3.1. Структура и правила ведения тестовой документации

Все артефакты тестирования хранятся в каталоге `/tests/qa-artifacts/`.

- **Тест-планы (Test Plans):** `TP-v[Версия]-[Компонент].md` → `TP-v1.0.3-core_math.md`
- **Тест-кейсы (Test Cases):** `TC-[Модуль]-[ID]-[Описание].md` → `TC-API-04-jwt_token_expiration.md`

### 3.2. Регламент оформления баг-репортов (Bug Reports)

- **Шаблон имени файла:** `BUG-[Версия]-[ID]-[Краткое_описание].md`
- **Пример имени файла:** `BUG-v1.0.3-04-dti_calculation_zero_division.md`

#### Унифицированный шаблон баг-репорта

````markdown
# [BUG-04] Исключение ZeroDivisionError при нулевом доходе

- **Обнаружено в версии ПО:** v1.0.3
- **Компонент системы:** Backend Core (`/src/app/core/math.py`)
- **Окружение:** Windows 11 Pro, Python 3.14.0, FastAPI 0.111.0
- **Уровень серьёзности (Severity):** Critical
- **Приоритет исправления (Priority):** High

## 1. Шаги для воспроизведения (STR)

1. Авторизоваться в системе.
2. Перейти в модуль «Доходы» и установить суммарный доход равным 0.
3. Перейти в модуль «Обязательства» и добавить кредит (платёж 20 000 руб).
4. Нажать на кнопку «Сгенерировать финансовую рекомендацию».

## 2. Ожидаемый результат

Система должна выдать валидационную ошибку в UI: *«Ошибка расчёта: доход должен быть больше нуля»* и вернуть статус `422`.

## 3. Фактический результат

Сервер возвращает `500 Internal Server Error`.
**Лог ошибки:** `ZeroDivisionError: division by zero` в `math.py`, строка 84.
````

## 4. Сводный регламент контроля изменений и релизный воркфлоу

Все изменения в кодовой базе проходят строго детерминированный цикл:

1. Оформление артефакта (`BUG-репорт` или `UT-протокол`).
2. Развёртывание локальной Git-ветки (`fix/` или `feat/`).
3. Внесение правок + прогон PyTest.
4. Инкремент версии в `config.py`.
5. Слияние в `main` + создание тега (`git tag vX.X.X`).

### 4.1. Стандарты оформления коммитов (Conventional Commits)

- **Шаблон сообщения:** `<type>(<scope>): <short description>`
- **Примеры:**
  - `feat(core): add Monte-Carlo cash flow simulation engine`
  - `fix(math): implement zero-income validation in DTI calculation`
  - `docs(ux): add individual usability protocol UT-F-v1.0.3-R02`
