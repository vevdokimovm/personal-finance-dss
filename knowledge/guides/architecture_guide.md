# Архитектурирование ПО — конспект-инструкция

> Сводный конспект по курсу: технологии, методики и — главное — **подходы и принципы, на которые опирается архитектор** при принятии решений. Материал собран из пяти практических блоков (Docker → кэширование → EDA → observability → CQRS/ES). Код-примеры даны в Python/FastAPI-стеке, а не в Java/Spring как в исходных лабах.

## Часть 0. Как мыслит архитектор (мета-уровень)

Прежде чем нырять в конкретные технологии — важно понять оптику, через которую архитектор смотрит на любую из них. Все пять блоков курса — это инструменты, а инструмент выбирается не «потому что модно», а под конкретную проблему.

### 0.1. Главный навык — это не знание паттернов, а trade-off анализ

Архитектура — это серия **компромиссов**. У любого решения есть цена. Распределённый кэш даёт масштабируемость, но добавляет сетевую задержку и проблему консистентности. Event-driven даёт слабую связанность, но усложняет отладку. CQRS разделяет нагрузку, но удваивает количество кода. Зрелость архитектора измеряется не тем, сколько паттернов он знает, а тем, насколько точно он понимает, **что он отдаёт взамен** за каждое преимущество.

Сквозной рефрен всего курса — *«не применяйте сложное там, где хватает простого»*. Это прямо написано про Event Sourcing («не стоит применять в простых доменах, где достаточно CRUD»), но это универсальный закон: сложность оправдана только тогда, когда боль от её отсутствия превышает боль от её введения.

### 0.2. Связанность (coupling) и связность (cohesion)

Два фундаментальных понятия, к которым сводится половина архитектурных решений:

1. **Coupling (связанность)** — насколько сильно компоненты зависят друг от друга. Цель — **слабая связанность (loose coupling)**: меняя один сервис, не должен ломаться другой.
2. **Cohesion (связность)** — насколько логически едины элементы внутри одного компонента. Цель — **высокая связность**: всё, что меняется вместе, лежит вместе.

Почти каждый паттерн курса работает на снижение связанности: брокер сообщений развязывает сервисы во времени, CQRS развязывает чтение и запись, контейнеры развязывают приложение от окружения.

### 0.3. Нефункциональные требования (NFR) — то, ради чего всё это

Функциональные требования отвечают на вопрос «что система делает». Архитектура же в основном про **нефункциональные** — «как хорошо она это делает»:

| NFR | За что отвечает | Какой блок курса закрывает |
|---|---|---|
| **Scalability** (масштабируемость) | держать рост нагрузки | контейнеры + оркестрация, EDA, кэш |
| **Availability / Reliability** (доступность/надёжность) | работать, несмотря на сбои | EDA (очереди не теряют сообщения), оркестрация |
| **Performance** (производительность) | низкая задержка отклика | кэширование, CQRS (read-модели) |
| **Observability** (наблюдаемость) | понимать, что внутри происходит | метрики/логи/трейсы |
| **Maintainability** (поддерживаемость) | дёшево менять и развивать | loose coupling, разделение ответственности |
| **Auditability** (аудируемость) | знать историю изменений | Event Sourcing |

### 0.4. Консистентность: CAP и eventual consistency

В распределённых системах нельзя одновременно иметь идеальную согласованность данных, доступность и устойчивость к разрывам сети (теорема CAP). На практике это значит: как только данные «размазываются» по нескольким местам (кэш + БД, write-модель + read-модель, несколько сервисов через очередь), приходится выбирать между:

- **Strong consistency** — данные везде одинаковы в любой момент (дорого, медленно).
- **Eventual consistency** — данные «сойдутся» через какое-то время (быстро, дёшево, но нужно проектировать с учётом окна рассинхрона).

Кэш с TTL, read-модели в CQRS, асинхронные консьюмеры — всё это **eventual consistency**. Архитектор обязан явно понимать, где у него окно несогласованности и допустимо ли это для бизнеса.

### 0.5. Идемпотентность

В распределённом мире сообщения дублируются, запросы повторяются (retry), сети рвутся. **Идемпотентная операция** — это операция, которую можно безопасно выполнить несколько раз с тем же результатом. Это не паттерн, а образ мышления: проектируя консьюмера очереди или обработчик команды, всегда спрашивай — «что будет, если это выполнится дважды?».

## Часть 1. Контейнеризация (Docker)

### 1.1. Зачем — эволюция развёртывания

Три поколения:

1. **Bare metal** — приложения крутятся прямо на ОС физического сервера. Конфликты зависимостей, «у меня на машине работало», тяжёлое масштабирование.
2. **Виртуализация (VM)** — несколько изолированных виртуальных машин на одном железе, у каждой своя полная ОС. Изоляция хорошая, но тяжёлая: каждая VM тащит целую операционку.
3. **Контейнеризация** — лёгкая альтернатива VM: изолированные окружения в рамках **одной** ОС-ядра. Стартуют за секунды, занимают мало, легко тиражируются. Массовое распространение пошло с выходом Docker в 2013 году.

Ключевая разница: VM виртуализирует **железо**, контейнер виртуализирует **ОС** (точнее — изолирует процессы через namespaces и cgroups ядра Linux).

### 1.2. Основные концепты

1. **Образ (image)** — неизменяемый шаблон, содержащий всё для работы приложения: код, зависимости, системные библиотеки, настройки окружения. Это «слепок».
2. **Контейнер (container)** — запущенный экземпляр образа. Из одного образа можно поднять сколько угодно контейнеров. Легковесны, быстро стартуют.
3. **Dockerfile** — сценарий сборки образа: на основе какого базового образа собираем, что копируем внутрь, с какой команды стартуем.
4. **Реестр (registry)** — хранилище образов и общая точка доступа (Docker Hub — официальный публичный реестр). Запушил образ → любой может скачать.
5. **Docker network** — виртуальная сеть, через которую контейнеры общаются друг с другом по именам.
6. **Оркестрация (orchestration)** — управление множеством контейнеров в проде (Kubernetes, Docker Swarm): автоперезапуск, масштабирование, балансировка.

### 1.3. Принципы, на которые опираются при контейнеризации

1. **Иммутабельность образа.** Образ не меняют после сборки — пересобирают. Это даёт воспроизводимость: один и тот же образ ведёт себя одинаково на ноутбуке и в проде.
2. **Конфигурация через переменные окружения.** Параметры (подключение к БД, порт) НЕ зашиваются в образ, а берутся из env. Это позволяет один образ гонять в dev/stage/prod без пересборки. (В твоём случае — `.env` + `python-dotenv`, а в контейнере — через `environment` в compose.)
3. **Один процесс — один контейнер.** Контейнер на приложение, контейнер на БД, контейнер на кэш. Не пихать всё в один.
4. **Отделение данных от контейнера (volumes).** Контейнер эфемерен — удалил и поднял заново. Данные БД должны переживать перезапуск, поэтому их выносят в **volume**.

### 1.4. Dockerfile для FastAPI (твой стек)

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
ENV APP_PORT=8080
EXPOSE 8080
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8080"]
```

Директивы: `FROM` — базовый образ; `WORKDIR` — рабочая папка внутри образа; `COPY` — копирование файлов внутрь; `RUN` — выполнить команду при сборке; `ENV` — задать переменную окружения; `EXPOSE` — задекларировать порт; `CMD`/`ENTRYPOINT` — команда старта контейнера.

> Тонкость: `requirements.txt` копируется и устанавливается **до** копирования кода. Это слой кэширования — пока зависимости не менялись, Docker не переустанавливает их при каждой пересборке.

### 1.5. Docker Compose

Compose — инструмент управления конфигурацией **нескольких** контейнеров через декларативный YAML-файл вместо длинных `docker run`. Автоматически поднимает общую сеть между сервисами (они видят друг друга по имени сервиса).

```yaml
services:
  api:
    build: .
    environment:
      DB_HOST: "postgres"
      DB_PORT: "5432"
      APP_PORT: "8080"
    ports:
      - "8080:8080"
    depends_on:
      - postgres

  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: "app"
      POSTGRES_USER: "app"
      POSTGRES_PASSWORD: "app"
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

Ключевые поля: `image`/`build` — откуда брать образ; `environment` — переменные конфигурации; `ports` — проброс портов (`хост:контейнер`); `volumes` — постоянное хранилище; `depends_on` — порядок старта; `restart` — политика перезапуска.

**Проброс портов** (`"8080:8080"`): запрос на `localhost:8080` хоста уходит в порт 8080 контейнера. Формат всегда `внешний_порт:внутренний_порт`.

### 1.6. ❓ Защита: суть в трёх ответах

1. **Образ vs контейнер**: образ — статичный шаблон (рецепт), контейнер — запущенный экземпляр (готовое блюдо). Из одного образа → много контейнеров.
2. **Зачем реестр**: централизованно хранить и распространять образы между машинами/командами.
3. **Зачем compose**: декларативно управлять многоконтейнерным развёртыванием и сетью между сервисами вместо ручных `docker run`.

## Часть 2. Кэширование (Redis)

### 2.1. Что это и когда применять

**Кэширование** — механизм временного хранения данных в быстродоступной памяти ради снижения времени отклика и нагрузки на основное хранилище. Главное правило применимости: **кэшируем то, что читают чаще, чем изменяют.**

Типичные кандидаты: часто запрашиваемые страницы и API-ответы, результаты тяжёлых SQL-запросов, пользовательские сессии, промежуточные вычисления.

### 2.2. Виды кэширования

1. **Локальное (In-Process)** — кэш в памяти самого приложения (например `functools.lru_cache` в Python). Плюс: максимальная скорость, ноль сетевых задержек. Минус: ограничен памятью процесса, не шарится между инстансами приложения (нет распределённости).
2. **Распределённое (Distributed)** — кэш в отдельном сервере (Redis, Memcached). Плюс: масштабируемость, единый кэш на все инстансы приложения. Минус: сетевые задержки, нужна репликация для надёжности.

Выбор — это снова trade-off: локальный быстрее, но не масштабируется; распределённый масштабируется, но платит сетью.

### 2.3. Стратегии работы с кэшем

| Стратегия | Как работает | Когда |
|---|---|---|
| **Cache-Aside** (Lazy Loading) | приложение само проверяет кэш; нет данных → читает из БД и кладёт в кэш | самая частая, для read-heavy |
| **Write-Through** | запись идёт в кэш и сразу в БД синхронно | когда важна свежесть кэша |
| **Write-Behind** (Write-Back) | запись в кэш, а в БД — асинхронно позже | высокая нагрузка на запись, допустим риск потери |

**Cache-aside** — основная рабочая лошадка. Логика:

1. Пришёл запрос → смотрим в кэш.
2. **Cache hit** (есть) → отдаём из кэша.
3. **Cache miss** (нет) → идём в БД, кладём результат в кэш, отдаём.
4. При **изменении/удалении** записи → принудительно чистим (инвалидируем) её в кэше.

### 2.4. Политики вытеснения (cache eviction)

Память кэша конечна — когда она заполнена, что-то надо удалять:

1. **LRU** (Least Recently Used) — выкидывает то, к чему дольше всего не обращались.
2. **LFU** (Least Frequently Used) — выкидывает то, что запрашивают реже всего.
3. **FIFO** (First In, First Out) — выкидывает самое старое по времени попадания.
4. **TTL-based** — данные удаляются по истечении заданного времени жизни.

### 2.5. Redis

**Redis** (Remote Dictionary Server) — in-memory key-value хранилище со сложными структурами данных. Почему его любят: высокая скорость (данные в RAM), гибкое вытеснение (TTL/LRU/LFU), богатые структуры данных, клиенты под все языки, сохранение состояния при перезапуске (RDB-снапшоты и AOF-лог), репликация и кластеризация.

> Контекст РФ: официальный Redis — *source-available*, не open-source, сайт без VPN недоступен. Рабочая альтернатива — **KeyDB** (open-source форк, полностью совместим с API Redis) или Valkey.

**Структуры данных Redis:**

| Тип | Что это | Применение |
|---|---|---|
| **String** | строка до 512 МБ (числа, JSON, сериализованные объекты) | базовое кэширование значений |
| **List** | упорядоченный список строк | очередь (FIFO) или стек (LIFO) |
| **Set** | неупорядоченное множество уникальных строк | теги, категории, уникальные ID |
| **Sorted Set (ZSet)** | как Set, но с весом (score), сортировка по нему | лидерборды, рейтинги |
| **Hash** | словарь ключ→поле→значение | хранение объектов целиком |
| **Bitmap** | операции над битами | флаги, отслеживание активности |

**TTL** (Time-To-Live) — время жизни ключа, после которого он автоматически удаляется. Команды: `EXPIRE key sec` (задать), `TTL key` (посмотреть остаток), `PERSIST key` (снять TTL, сделать вечным).

Полезно про асимптотику (любят на защите): операции с концами списка (`LPUSH`/`RPUSH`/`LPOP`/`RPOP`) — O(1); чтение диапазона `LRANGE` — O(N); работа с Set (`SADD`/`SREM`/`SISMEMBER`) — O(1); `GET`/`SET` строки — O(1). Удаление ключа: `DEL` — синхронно, `UNLINK` — асинхронно (лучше для больших значений).

### 2.6. Cache-aside на Python (redis-py)

```python
import json
from datetime import timedelta
from uuid import UUID

import redis.asyncio as redis


class UserCacheService:
    def __init__(self, client: redis.Redis) -> None:
        self._client = client
        self._prefix = "user:v1:"
        self._ttl = timedelta(seconds=60)

    def _key(self, user_id: UUID) -> str:
        return f"{self._prefix}{user_id}"

    async def get(self, user_id: UUID) -> dict | None:
        raw = await self._client.get(self._key(user_id))
        return json.loads(raw) if raw else None

    async def set(self, user_id: UUID, data: dict) -> None:
        await self._client.set(self._key(user_id), json.dumps(data), ex=self._ttl)

    async def invalidate(self, user_id: UUID) -> None:
        await self._client.delete(self._key(user_id))
```

В сервисном слое логика cache-aside: `get_user` сначала зовёт `cache.get`, при miss идёт в репозиторий и делает `cache.set`; `update_user`/`delete_user` в конце зовут `cache.invalidate`.

### 2.7. Принцип, который стоит за всем разделом

> «There are only two hard things in Computer Science: cache invalidation and naming things.» — Phil Karlton

Самое сложное в кэше — не положить данные, а вовремя их **инвалидировать**. Версионирование ключей (`user:v1:`) и TTL — две главные техники защиты от «протухших» данных. TTL — страховка: даже если ты забыл инвалидировать руками, данные сами протухнут.

### 2.8. ❓ Защита: суть

1. **Зачем кэш**: снизить задержку и нагрузку на БД, кэшируя часто читаемые / редко меняемые данные.
2. **Cache-aside, когда обоснован**: read-heavy сценарии; инвалидация при записи обеспечивает консистентность.
3. **Стратегии вытеснения**: LRU / LFU / FIFO / TTL — выбор зависит от паттерна доступа.

## Часть 3. Event-Driven Architecture (EDA, RabbitMQ)

### 3.1. Что это

**EDA** — архитектурный стиль, где компоненты обмениваются **событиями**, а не вызывают друг друга напрямую. **Событие (event)** — описание свершившегося факта: «пользователь зарегистрировался», «заказ создан».

Компоненты:

1. **Producer** — публикует событие.
2. **Consumer** — реагирует на событие.
3. **Event broker** (RabbitMQ, Kafka) — передаёт события от продюсера к консьюмеру.

### 3.2. Почему async лучше прямого вызова (центральная идея)

Возьмём пример: при создании пользователя нужно отправить уведомление. Два пути:

**Синхронно (плохо):** `user-service` через HTTP дёргает `notification-service` и ждёт ответа. Проблемы:
- жёсткая связанность — сервисы знают друг о друге напрямую;
- замедление — `user-service` ждёт, пока отправится уведомление, прежде чем ответить клиенту;
- хрупкость — упал `notification-service`, сломался и `user-service`.

**Асинхронно через брокер (хорошо):** `user-service` кидает событие в брокер и сразу отвечает клиенту. `notification-service` заберёт событие из очереди, когда сможет. Сервисы развязаны.

### 3.3. Плюсы и минусы

**Плюсы:**
1. **Loose coupling** — сервисы не знают друг о друге; подписчиков можно добавлять/удалять/менять без правок продюсера.
2. **Масштабируемость** — подписчики масштабируются независимо; нагрузку можно жевать параллельно (паттерн **competing consumers** — несколько консьюмеров на одной очереди делят работу).
3. **Асинхронность** — долгие операции уходят «в фон», основной сервис быстро отвечает.
4. **Отказоустойчивость** — упал консьюмер → сообщения копятся в очереди, а не теряются. Есть откат/повтор.

**Минусы (та самая цена):**
1. Сложность отладки и трассировки — поток управления неочевиден.
2. Сложность тестирования.
3. Проблемы с порядком и консистентностью сообщений.
4. Управление схемами и версионированием событий.

### 3.4. RabbitMQ

Брокер сообщений, реализующий протокол **AMQP**. Схема маршрутизации:

> **Producer** отправляет сообщение (с `routing key` и заголовками) в **Exchange** → по типу обменника и правилам **Binding** сообщение попадает в одну/несколько **Queue** → **Consumer** забирает из очереди.

Producer публикует НЕ в очередь напрямую, а в **exchange**. Именно exchange решает, в какие очереди разложить сообщение.

**Типы Exchange:**

| Тип | Логика маршрутизации |
|---|---|
| **Fanout** | шлёт во все связанные очереди, игнорируя routing key (broadcast) |
| **Direct** | точное совпадение routing key сообщения с key биндинга |
| **Topic** | как direct, но с шаблонами: `*` = ровно одно слово, `#` = ноль или более слов |
| **Headers** | маршрутизация по заголовкам сообщения, а не по routing key; `x-match`=`all` (все заголовки) или `any` (хотя бы один) |

**Обработка ошибок — Dead Letter Exchange (DLX):** спецобменник, куда RabbitMQ отправляет недоставленные/отвергнутые сообщения. У очереди задаются аргументы `x-dead-letter-exchange` (куда слать «мёртвые» сообщения) и `x-dead-letter-routing-key`. Это основа для retry-механик и разбора «ядовитых» сообщений (poison messages).

### 3.5. Producer/Consumer на Python (aio-pika)

```python
import json

import aio_pika


class UserEventPublisher:
    def __init__(self, channel: aio_pika.abc.AbstractChannel) -> None:
        self._channel = channel
        self._exchange_name = "user.exchange"

    async def publish_user_created(self, payload: dict) -> None:
        exchange = await self._channel.declare_exchange(
            self._exchange_name, aio_pika.ExchangeType.TOPIC, durable=True
        )
        message = aio_pika.Message(
            body=json.dumps(payload).encode(),
            delivery_mode=aio_pika.DeliveryMode.PERSISTENT,
        )
        await exchange.publish(message, routing_key="user.created")
```

```python
async def consume(channel: aio_pika.abc.AbstractChannel) -> None:
    queue = await channel.declare_queue("user.events", durable=True)
    exchange = await channel.declare_exchange(
        "user.exchange", aio_pika.ExchangeType.TOPIC, durable=True
    )
    await queue.bind(exchange, routing_key="user.#")

    async with queue.iterator() as messages:
        async for message in messages:
            async with message.process():
                event = json.loads(message.body)
                # обработка события (идемпотентно!)
```

> `durable=True` у очереди/обменника и `PERSISTENT` у сообщения — чтобы данные пережили перезапуск брокера. `message.process()` подтверждает (ack) сообщение только при успешной обработке — при падении оно вернётся в очередь.

### 3.6. Принципы, на которые опираются в EDA

1. **Идемпотентность консьюмеров** — RabbitMQ гарантирует доставку *at-least-once* (как минимум один раз), то есть дубли возможны. Консьюмер обязан корректно переваривать повтор.
2. **Контракт событий** — продюсер и консьюмер договариваются о схеме события. Версионирование (`v1`, `v2`) обязательно, иначе при эволюции схемы всё развалится.
3. **Durability** — критичные сообщения и очереди делают устойчивыми, иначе перезапуск брокера = потеря данных.

### 3.7. ❓ Защита: суть

1. **Чем direct отличается от topic**: direct — точное совпадение routing key; topic — совпадение по шаблону с `*` и `#`.
2. **Плюс/минус EDA**: главный плюс — слабая связанность и асинхронность; главный минус — сложность отладки и консистентности.
3. **Роль exchange**: producer публикует в exchange, а тот по биндингам раскладывает сообщение по очередям — продюсер не знает о консьюмерах.

## Часть 4. Observability (наблюдаемость)

### 4.1. Observability vs Monitoring

- **Monitoring (мониторинг)** — сбор данных и отчёты по заранее известным метрикам: «следим за тем, что ожидаем увидеть» (CPU, память, частота ошибок). Отвечает на *известные* вопросы.
- **Observability (наблюдаемость)** — насколько легко понять **внутреннее** состояние системы по её внешним проявлениям. Позволяет отвечать на *неизвестные заранее* вопросы — «почему именно этот запрос тормозит?».

Мониторинг — подмножество наблюдаемости. Observability шире: она про способность задать системе вопрос, который ты не предусмотрел заранее.

### 4.2. Три столпа наблюдаемости

1. **Метрики (metrics)** — числовые данные о состоянии (загрузка CPU, latency, число запросов). Дёшевы, агрегируемы, хороши для дашбордов и алертов.
2. **Логи (logs)** — записи о происходящих событиях. Детальны, хороши для разбора конкретного инцидента.
3. **Трассировки (traces)** — путь запроса через все сервисы системы: от клиента через API, очередь, БД и обратно. Отвечают на вопрос «где именно потратилось время».

### 4.3. SLI / SLO / SLA — язык договорённостей о качестве

Цепочка: **SLI → измеряет → SLO → определяет цель → SLA → юридически фиксирует**

1. **SLI** (Service Level Indicator) — конкретный измеримый показатель (например, доля запросов с откликом < 200 мс).
2. **SLO** (Service Level Objective) — внутренняя цель по SLI (например, «99.9% запросов < 200 мс»).
3. **SLA** (Service Level Agreement) — юридическое обязательство перед клиентом с компенсацией за нарушение.

Метрики дают SLI; логи и трассировки помогают понять, почему SLO не выполняется, и вернуть систему в норму.

### 4.4. Золотые сигналы (golden signals)

Что мониторить в первую очередь у любого сервиса (методология Google SRE):

1. **Latency** — время отклика (отдельно для успешных и ошибочных запросов).
2. **Traffic** — объём нагрузки (RPS).
3. **Errors** — частота ошибок.
4. **Saturation** — насыщенность ресурсов (насколько забиты CPU/память/диск).

Родственные методики: **RED** (Rate, Errors, Duration — для сервисов) и **USE** (Utilization, Saturation, Errors — для ресурсов).

### 4.5. Prometheus

Система сбора и хранения метрик. Работает по **pull-модели**: Prometheus сам периодически (`scrape_interval`, например 15с) **забирает** метрики с эндпоинта приложения (`/metrics`), а не приложение их пушит. (Есть и push через Pushgateway для коротких задач.)

**Типы метрик:**

| Тип | Что | Пример |
|---|---|---|
| **Counter** | только растёт (или сбрасывается в 0 при рестарте) | число запросов, ошибок |
| **Gauge** | растёт и падает | текущая температура, число активных коннектов |
| **Histogram** | распределение по бакетам + сумма + count | длительность запросов, размеры ответов |
| **Summary** | как histogram, но считает квантили на стороне клиента | перцентили latency |

**PromQL** — язык запросов Prometheus. Примеры паттернов: `rate(http_requests_total[5m])` — скорость роста счётчика за 5 минут; деление `rate(..._sum) / rate(..._count)` — средняя длительность.

**Экосистема:**
- **Grafana** — визуализация метрик (дашборды поверх Prometheus как datasource).
- **Alertmanager** — Prometheus сам алерты НЕ шлёт; он вычисляет правила (alerting rules) и передаёт сработавшие в Alertmanager, который группирует, давит дубли и маршрутизирует уведомления (email/Telegram/Slack). Алерт проходит стадии `Pending` → `Firing`.

### 4.6. Метрики FastAPI на Python (prometheus_client)

```python
from prometheus_client import Counter, Histogram

users_created_total = Counter(
    "users_created_total", "Total number of created users"
)
request_duration = Histogram(
    "api_request_duration_seconds",
    "Request duration",
    labelnames=["method"],
)


# в обработчике:
with request_duration.labels(method="post").time():
    user = await service.create_user(payload)
    users_created_total.inc()
```

Эндпоинт `/metrics` отдаётся библиотекой `prometheus_client` (или готовым `prometheus-fastapi-instrumentator`), Prometheus его скрейпит.

### 4.7. Логи и трассировки

**Логи — best practices:** структурные логи (JSON, а не свободный текст — машиночитаемо), уровни (DEBUG/INFO/WARN/ERROR), **correlation/trace ID** в каждой записи (чтобы склеить логи одного запроса через все сервисы), централизованный сбор (ELK/Loki). В Python — `structlog` или стандартный `logging` с JSON-форматтером.

**Трассировки — понятия:**
- **Trace ID** — единый идентификатор всего пути запроса через систему.
- **Span ID** — идентификатор отдельного отрезка работы внутри трейса (один вызов БД, один HTTP-запрос). Спаны вкладываются друг в друга, образуя дерево.

**OpenTelemetry (OTel)** — открытый стандарт сбора трейсов, метрик и логов. Даёт единый API/SDK: инструментируешь приложение один раз, а отправлять данные можешь в любую систему (Jaeger, Tempo, Prometheus). Главная ценность — вендоронезависимость. В Python — `opentelemetry-sdk` + автоинструментация для FastAPI.

### 4.8. ❓ Защита: суть

1. **Observability vs monitoring**: мониторинг следит за известными метриками; наблюдаемость позволяет понять внутреннее состояние и отвечать на незаданные заранее вопросы.
2. **SLI/SLO/SLA**: индикатор → цель → юридический контракт.
3. **Trace ID vs Span ID**: trace ID — весь путь запроса; span ID — отдельный отрезок внутри него.
4. **Модели сбора Prometheus**: pull (основная, скрейп с `/metrics`) и push (через Pushgateway).

## Часть 5. CQRS и Event Sourcing

### 5.1. CQRS

**CQRS** (Command Query Responsibility Segregation) — паттерн, разделяющий модели для **чтения** (Query) и для **изменения** (Command):

- **Command** изменяет состояние, но ничего не возвращает кроме подтверждения.
- **Query** читает состояние, но ничего не изменяет.

Это разделение на уровне моделей, DTO, контроллеров и репозиториев: отдельный путь записи (write-side) и отдельный путь чтения (read-side).

**Зачем:**
1. Разные требования к производительности чтения и записи (обычно читают в разы чаще).
2. Независимая оптимизация моделей чтения от моделей записи.
3. Упрощение масштабирования (read-side масштабируется отдельно).

**Когда применять:** высокая нагрузка на чтение/запись, зрелый проект, сложная бизнес-логика. **Когда НЕ применять:** простой CRUD — CQRS только добавит лишний код без выгоды.

### 5.2. Event Sourcing

**Event Sourcing** (ES) — паттерн, где состояние объекта НЕ хранится напрямую, а **восстанавливается из последовательности событий**, описывающих все произошедшие изменения.

Классический пример — баланс. Вместо хранения числа `balance = 300` хранится журнал:
```
BalanceCreated(user=X)
BalanceCredited(user=X, amount=500)
BalanceDebited(user=X, amount=200)
```
Текущий баланс (300) получается «проигрыванием» этих событий по порядку.

**Традиционный подход хранит результат; Event Sourcing хранит историю, из которой результат выводится.**

**Когда применять:**
1. Нужно знать **причину** каждого изменения данных.
2. Нужен **журнал аудита**, возможность воспроизвести события, откатиться к точке во времени, отменить операции.
3. Нужна гибкость с read-моделями — создавать новые представления под меняющиеся требования.

**Когда НЕ применять:** простые домены с маленькой бизнес-логикой, где хватает CRUD и не нужен аудит.

### 5.3. Компоненты ES

1. **Command** — намерение изменить состояние («снять 200 рублей»).
2. **Aggregate (агрегат)** — объект, инкапсулирующий бизнес-правила и **инварианты**; контролирует, какие изменения допустимы, и гарантирует целостность. Например, агрегат баланса не даст снять больше, чем есть (проверка `Insufficient funds`).
3. **Event (событие)** — факт состоявшегося изменения. **Иммутабелен** — после записи не меняется никогда.
4. **Event Store** — хранилище событий, неизменяемая упорядоченная последовательность. Источник истины системы.
5. **Projection / Projector (проекция)** — строит read-модели, обрабатывая поток событий.
6. **Read-model (проекция данных)** — денормализованное представление под конкретный запрос (текущий баланс, история транзакций).

**Цикл записи:** команда → агрегат проверяет инвариант → генерирует событие → событие пишется в Event Store → проекция обновляет read-модели.

### 5.4. Версионирование, replay, snapshot

1. **Версионирование событий** — схема события со временем меняется; каждому событию проставляется ревизия (`v1`, `v2`), чтобы старые события корректно читались новым кодом.
2. **Replay (реплей)** — если read-модели потерялись (БД стёрлась), их состояние **восстанавливается автоматически**: события заново прогоняются через проекцию. Это суперсила ES — read-модель всегда производна и пересоздаваема.
3. **Snapshot (снапшот)** — если событий у агрегата накопилось слишком много, восстановление состояния (проигрывание всех событий) тормозит. Снапшот — зафиксированное состояние агрегата на определённой точке: дальше не нужно бежать по всей истории, достаточно взять снапшот + события после него. Триггерится по числу событий (например, каждые 10).

### 5.5. CQRS + ES вместе

Связка естественна: команды генерируют события (ES), события питают отдельные read-модели (CQRS). Write-side хранит события, read-side — проекции. Между ними **eventual consistency**: read-модель обновляется чуть позже записи события. Это нужно держать в голове — сразу после команды чтение может вернуть «старое» значение в течение окна синхронизации.

### 5.6. Принципы раздела

1. **События иммутабельны** — потому что они описывают свершившийся факт. Факт нельзя изменить, можно только добавить новый компенсирующий факт. Это и даёт надёжный аудит.
2. **Read-модель производна и одноразова** — она всегда пересоздаётся из событий, поэтому её можно менять, добавлять новые представления, чинить реплеем.
3. **Агрегат — граница консистентности** — все инварианты проверяются внутри агрегата; именно он решает, валидна ли команда.

### 5.7. ❓ Защита: суть

1. **CQRS — зачем**: разделить чтение и запись для независимой оптимизации и масштабирования.
2. **Почему события иммутабельны**: они фиксируют свершившийся факт; неизменяемость даёт честный аудит и возможность реплея.
3. **Как восстановить агрегат**: проиграть его события по порядку (при наличии снапшота — от снапшота + события после).
4. **Зачем snapshot**: ускорить восстановление, не пробегая всю историю событий.

## Часть 6. Как всё связывается в продакшн-систему

Пять блоков курса — это не пять отдельных тем, а **слои зрелости** одной системы. Логика наращивания:

1. **Контейнеры** — фундамент: упаковали приложение и зависимости, сделали развёртывание воспроизводимым.
2. **Кэш** — первый шаг к производительности: сняли нагрузку с БД на горячих чтениях.
3. **EDA** — развязали сервисы: вынесли долгие/побочные операции в асинхрон, повысили отказоустойчивость.
4. **Observability** — обрели зрение: как только система стала распределённой (несколько контейнеров, очереди), без метрик/логов/трейсов отладка становится невозможной. Наблюдаемость — **обязательное условие** работы с распределёнкой, а не опция.
5. **CQRS/ES** — высшая сложность: разделили модели и перешли к событийному хранению там, где нужны аудит, гибкие представления и раздельное масштабирование.

### Шпаргалка «проблема → инструмент»

| Болит | Инструмент из курса |
|---|---|
| «у меня на машине работало» | Docker (иммутабельный образ) |
| поднять то же самое в проде одной командой | Docker Compose / оркестрация |
| БД захлёбывается на одинаковых запросах | кэш (cache-aside + TTL) |
| сервис тормозит из-за побочных операций | EDA (асинхронные события) |
| упал один сервис — упало всё | очереди (буфер + развязка) |
| «почему всё медленно, где именно?» | трассировки + метрики |
| «нас разбудили в 3 ночи, что случилось?» | алерты (Prometheus + Alertmanager) |
| «кто и когда изменил эти данные?» | Event Sourcing (журнал событий) |
| читают в 100 раз чаще, чем пишут | CQRS (раздельные read-модели) |

### Сквозные принципы (то, на что опирается архитектор всегда)

1. **Trade-off над модой** — каждое преимущество имеет цену; решение принимается по балансу, а не по хайпу.
2. **Простота по умолчанию** — сложный паттерн вводится только когда боль от его отсутствия реальна и измерима.
3. **Loose coupling, high cohesion** — слабая связь между компонентами, сильная связность внутри.
4. **Проектируй под отказ** — сети рвутся, сервисы падают, сообщения дублируются; идемпотентность и буферизация — норма.
5. **Делай систему наблюдаемой с первого дня** — то, что нельзя измерить, нельзя ни отладить, ни улучшить.
6. **Явно выбирай модель консистентности** — знай, где у тебя strong, а где eventual, и допустимо ли окно рассинхрона для бизнеса.

## Часть 7. API-документация: Swagger / OpenAPI

Сквозь все практические блоки красной нитью идёт одна штука: каждый сервис тестируется и осматривается через **Swagger UI** (`/swagger-ui/...` в Java-лабах). Это не случайность — документация API сегодня считается частью самой архитектуры, а не довеском.

### 7.1. OpenAPI vs Swagger — не путать

1. **OpenAPI Specification (OAS)** — это **стандарт**: формальное описание REST API в машиночитаемом виде (JSON/YAML). Какие есть эндпоинты, какие методы, какие параметры, какие тела запросов/ответов, какие коды. Раньше назывался «Swagger Specification», с версии 3.0 переименован в OpenAPI.
2. **Swagger** — это **набор инструментов** вокруг спецификации: Swagger UI (интерактивная веб-страница), Swagger Editor, Swagger Codegen (генерация клиентов/серверов).

Коротко: OpenAPI — это язык контракта, Swagger — тулинг, который его рендерит и обслуживает.

### 7.2. Зачем это архитектору

1. **Контракт как единый источник правды.** Спецификация — это формальный договор между бэкендом и потребителями (фронт, мобилки, другие сервисы). Все смотрят в один документ, а не в чьи-то слова.
2. **Документация, которая не врёт.** Если она генерируется из кода — она не устаревает. Рукописные доки расходятся с реальностью на второй неделе.
3. **Интерактивное тестирование.** Swagger UI позволяет дёргать эндпоинты прямо из браузера — без Postman и curl. Именно поэтому в лабах всё проверяется через `/swagger-ui`.
4. **Автогенерация клиентов.** Из OpenAPI-схемы генерируются типизированные клиенты на любом языке — фронт не пишет HTTP-обвязку руками.
5. **Два подхода к разработке:**
   - **Code-first** — пишешь код, схема генерируется из него (путь FastAPI и Spring).
   - **Contract-first** — сначала проектируешь OpenAPI-контракт, потом по нему генерируешь заготовки кода (удобно, когда команды бэка и фронта работают параллельно).

### 7.3. В FastAPI это из коробки (твоё преимущество)

Здесь у тебя приятный сюрприз по сравнению с Java-лабами: в Spring для Swagger UI подключали отдельную зависимость (`springdoc-openapi`) и ходили на `/swagger-ui/index.html`. **В FastAPI документация генерируется автоматически** из твоих type hints и Pydantic-моделей — ничего дополнительно ставить не нужно.

Три эндпоинта появляются сами:

| URL | Что отдаёт |
|---|---|
| `/docs` | **Swagger UI** — интерактивная страница с тестированием |
| `/redoc` | **ReDoc** — альтернативный, более «читательский» рендер той же схемы |
| `/openapi.json` | сырая OpenAPI-спецификация (тот самый машиночитаемый контракт) |

Pydantic-модель → автоматически попадает в схему как описание тела запроса/ответа со всеми типами и валидацией. Это и есть code-first во всей красе.

```python
from uuid import UUID

from fastapi import APIRouter, status
from pydantic import BaseModel, EmailStr

router = APIRouter(prefix="/users", tags=["users"])


class UserCreate(BaseModel):
    name: str
    email: EmailStr


class UserResponse(BaseModel):
    id: UUID
    name: str
    email: EmailStr


@router.post(
    "",
    response_model=UserResponse,
    status_code=status.HTTP_201_CREATED,
    summary="Создать пользователя",
)
async def create_user(payload: UserCreate) -> UserResponse:
    ...
```

Что отсюда подтянется в Swagger UI автоматически: тег `users` (группировка эндпоинтов), `summary` (заголовок операции), схемы `UserCreate`/`UserResponse` с типами полей, код ответа `201`, валидация `EmailStr`. Поля для кастомизации: `tags`, `summary`, `description`, `response_model`, `responses` (описать разные коды), `Field(..., examples=[...])` (примеры значений).

### 7.4. ❓ Защита: суть

1. **OpenAPI vs Swagger**: OpenAPI — стандарт описания API (контракт); Swagger — инструменты вокруг него (UI, кодген).
2. **Зачем Swagger UI**: интерактивная, всегда актуальная документация + тестирование эндпоинтов из браузера.
3. **Code-first vs contract-first**: генерировать схему из кода либо генерировать код из заранее спроектированной схемы.

## Часть 8. Операционная шпаргалка: UI, порты, команды

Практический слой, который реально спасает на защите и при локальном запуске стека. Здесь собрано всё, что в лабах раскидано по скриншотам.

### 8.1. Кто на каком порту и где его веб-морда

| Сервис | Порт(ы) | Веб-интерфейс / доступ | Дефолтный логин |
|---|---|---|---|
| **Приложение (FastAPI)** | `8080` | `/docs` (Swagger), `/redoc`, `/openapi.json`, `/metrics`, `/health` | — |
| **PostgreSQL** | `5432` | DataGrip / pgAdmin / DBeaver | задаётся в env |
| **Redis / KeyDB** | `6379` | `redis-cli` / `keydb-cli`, RedisInsight, DataGrip | — |
| **RabbitMQ** | `5672` (AMQP), `15672` (Management UI), `15692` (метрики Prometheus) | Management UI в браузере | `guest` / `guest` |
| **Prometheus** | `9090` | web UI + встроенная консоль PromQL + `Status → Targets` | — |
| **Grafana** | `3000` | дашборды; datasource = `http://prometheus:9090` | `admin` / `admin` |
| **Alertmanager** | `9093` | web UI (просмотр активных алертов) | — |
| **Mailpit** (тестовый SMTP) | `1025` (SMTP), `8025` (web UI) | web UI для просмотра «отправленных» писем-алертов | — |

> RabbitMQ Management UI (`:15672`) — это где руками создают **Exchange / Queue / Binding** и шлют тестовые сообщения. Plugin `rabbitmq_prometheus` отдаёт метрики брокера на `:15692`, откуда их забирает Prometheus.

### 8.2. Ловушка с сетью в Docker Compose (частый вопрос на защите)

Внутри одной compose-сети контейнеры видят друг друга **по имени сервиса**, а не по `localhost`:

1. Приложение ходит в БД по хосту `postgres:5432`, а не `localhost:5432`.
2. Prometheus (в контейнере) скрейпит приложение по `http://user-service:8080`, а не `localhost`.
3. Grafana подключает datasource как `http://prometheus:9090`.

Именно поэтому в лабе по observability приложения **тоже** перенесли в контейнеры — иначе Prometheus из своего контейнера не достучался бы до них.

Различай два порта в маппинге `"8001:5432"`: **внутри** сети сервис слушает свой порт контейнера (`5432`), а **снаружи** (с хоста) он доступен по проброшенному (`8001`). Изнутри сети всегда обращаешься к внутреннему порту.

### 8.3. Docker / Compose — команды-шпаргалка

```bash
# Образы
docker build -t myapp .            # собрать образ из Dockerfile
docker images                      # список образов
docker tag <id> user/repo:latest   # переименовать (для пуша в реестр)
docker login                       # авторизоваться в реестре (пароль не отображается)
docker push user/repo:latest       # запушить образ в реестр

# Контейнеры
docker run -p 8080:8080 <id>       # запустить с пробросом порта
docker ps                          # запущенные контейнеры
docker exec -it <id/имя> bash      # зайти внутрь контейнера (оболочка)
docker logs -f <id/имя>            # смотреть логи в реальном времени

# Compose
docker compose up -d               # поднять весь стек в фоне
docker compose up --build -d       # пересобрать образы и поднять
docker compose build               # только собрать
docker compose down                # остановить и удалить контейнеры
docker compose ps                  # статус сервисов стека
docker compose logs -f <сервис>    # логи конкретного сервиса
```

> Цикл при изменении кода: `docker compose down` → пересобрать → `docker compose up --build -d`. Контейнер крутит старый образ, пока ты его не пересоберёшь.

### 8.4. Healthcheck — зачем

Эндпоинт здоровья (`/health`) нужен не для людей, а для инфраструктуры:

1. **Оркестратор** (Compose/K8s) понимает, жив ли контейнер, и перезапускает мёртвый.
2. **`depends_on` с условием** `condition: service_healthy` даёт стартовать приложению только после того, как БД реально готова принимать соединения (а не просто «контейнер запущен»).
3. **Балансировщик** не льёт трафик в нездоровый инстанс.

В FastAPI это просто отдельный лёгкий эндпоинт, который проверяет критичные зависимости (БД, кэш) и возвращает 200/503.

## Что изучить дальше

1. **Kubernetes** — следующий шаг после Docker Compose: оркестрация контейнеров в проде (поды, деплойменты, сервисы, ингрессы).
2. **Kafka** — брокер для высоконагруженного стриминга и event log; полезно сравнить с RabbitMQ (Kafka — лог с перечитыванием, Rabbit — классическая очередь).
3. **Saga pattern** — управление распределёнными транзакциями поверх EDA (как держать консистентность через несколько сервисов без двухфазного коммита).
4. **Outbox pattern** — надёжная публикация событий из БД (решает проблему «записал в БД, но не успел отправить событие»).
5. **DDD (Domain-Driven Design)** — теоретическая база под агрегаты, ограниченные контексты и язык предметной области; естественный спутник CQRS/ES.
6. **Distributed tracing на практике** — связать OpenTelemetry + Jaeger/Tempo и пройти трейс через реальную цепочку из нескольких FastAPI-сервисов и очереди.
