## Часть 2. Языки программирования

Не «какой лучше», а «какой для чего». У каждого языка есть **ниша**, где он сильнее остальных.

### 2.1. Системные / низкоуровневые

| Язык | Зачем нужен | Где живёт |
|---|---|---|
| **C** | максимальная скорость и контроль над памятью, основа всего | 💡 ядро Linux, embedded, драйверы |
| **C++** | C + ООП + мощь, для тяжёлых высокопроизводительных систем | 💡 движки игр (Unreal), браузеры, БД, торговые системы |
| **Rust** 🔥 | безопасность памяти **без сборщика мусора** + скорость C++ | системный софт, WebAssembly, где нужны и скорость, и надёжность |
| **Go (Golang)** 🔥 | простой, быстрый, отлично в многопоточке, быстро компилируется | 💡 Docker, Kubernetes — оба написаны на Go; cloud-native, микросервисы |

Суть: тут платишь сложностью за производительность и контроль. C/C++ — старая гвардия, Rust — современная замена с упором на безопасность, Go — «простой бэкенд для облака».

```rust
// Rust: владение памятью контролируется компилятором, без GC и без утечек
fn main() {
    let debts = vec![14.0, 9.5, 22.0];
    let expensive: Vec<&f64> = debts.iter().filter(|&&r| r > 14.0).collect();
    println!("{:?}", expensive);
}
```

### 2.2. Универсальные / backend

| Язык | Зачем нужен | Где живёт |
|---|---|---|
| **Python** ⭐🔥 | читаемый, быстрый в разработке, гигантская экосистема | 💡 бэкенд, скрипты, data/ML, автоматизация (Instagram, Dropbox) |

| **PHP** | классика веба, быстро поднять сайт | 💡 WordPress, бóльшая часть старого веба |

```python

expensive = [r for r in debts if r > 14.0]

```

## Часть 3. Парадигмы программирования

**Парадигма — это *стиль мышления* о коде.** Многие языки поддерживают несколько (Python — императивный + ООП + немного функциональный).

| Парадигма | Идея | Языки-представители |
|---|---|---|
| **Императивная** | пошагово описываешь *как* делать (меняешь состояние) | C, ассемблер |
| **ООП** ⭐🔥 | мир из объектов с состоянием и поведением | Java, C#, Python, C++ |
| **Функциональная** 🔥 | программа = композиция функций, без побочных эффектов | Haskell, Elixir, F#, Clojure |
| **Декларативная** | описываешь *что* хочешь, не *как* | SQL, HTML, конфиги |
| **Логическая** | факты + правила, система сама выводит ответ | Prolog |

### 3.1. Что такое «функциональный язык» 🔥

Ключевые идеи функционального программирования (ФП):

1. **Чистые функции** (pure functions) — результат зависит только от входа, нет побочных эффектов. Один и тот же вход → всегда один выход.
2. **Иммутабельность** (immutability) — данные не меняются, создаются новые. Меньше багов от случайных мутаций.
3. **Функции — объекты первого класса** — функцию можно передать в другую функцию, вернуть из функции (higher-order functions).
4. **Декларативность** — `map/filter/reduce` вместо ручных циклов.

```python
# Python умеет в функциональный стиль (частично)
from functools import reduce

debts = [14.0, 9.5, 22.0]

# чистая функция, без мутаций, через higher-order функции
total_expensive = reduce(
    lambda acc, r: acc + r,
    filter(lambda r: r > 14.0, debts),
    0.0,
)
print(total_expensive)  # 22.0
```

**Чисто функциональные языки** (где это не стиль, а закон):

| Язык | Ниша |
|---|---|
| **Haskell** | эталон ФП, академия, финтех, компиляторы |
| **Elixir** | веб реального времени, высоконагруженные системы (на Erlang VM) 💡 Discord, WhatsApp-инфра |
| **Erlang** | телеком, миллионы параллельных соединений, отказоустойчивость |
| **Clojure / F# / Scala** | ФП на популярных платформах (JVM / .NET) |

Зачем тебе знать: ФП-идеи (чистые функции, иммутабельность) делают код предсказуемее и легче тестируемым — их полезно тащить даже в Python. На собесе про разницу «функциональное vs ООП» спросить могут.

## Часть 5. Backend-стек ⭐

Твоя территория. Логика, данные, API на сервере.

### 5.1. Веб-фреймворки по языкам 🔥

| Язык | Фреймворк | Когда |
|---|---|---|
| **Python** ⭐ | **FastAPI** | современный, async, авто-документация API, типизация — твой выбор для FINPILOT |
| **Python** ⭐ | **Django** | «батарейки в комплекте»: ORM, админка, auth — для крупных приложений |
| **Python** ⭐ | **Flask** | минималистичный, гибкий, для маленьких сервисов |
| **JavaScript** | **Express** | минималистичный Node-фреймворк |
| **JavaScript** | **NestJS** | структурный Node-фреймворк (как Spring для JS) |
| **Java** | **Spring Boot** | корпоративный стандарт |
| **C#** | **ASP.NET Core** | мощный, быстрый, кроссплатформенный |
| **Go** | **Gin / Echo** | лёгкие быстрые фреймворки |
| **Ruby** | **Rails** | «convention over configuration», быстрый старт |
| **PHP** | **Laravel** | удобный современный PHP-фреймворк |

### 5.2. Почему FastAPI хорош (для защиты/собеса)

- **Async из коробки** — держит много одновременных запросов.
- **Pydantic** — валидация данных через type hints.
- **Авто-документация** — Swagger/OpenAPI генерируется сам.
- **Скорость** — один из самых быстрых Python-фреймворков (на уровне Node/Go для I/O-задач).

### 5.3. Что обычно стоит рядом с бэкендом

| Компонент | Зачем | Примеры |
|---|---|---|
| **ORM** ⭐ | мостик класс ↔ таблица | **SQLAlchemy** (твой), Django ORM, Prisma |
| **Миграции** | версионирование схемы БД | **Alembic** (для SQLAlchemy) |
| **Валидация** | проверка входных данных | **Pydantic** |
| **Аутентификация** | JWT, OAuth2, сессии | python-jose, Authlib |
| **Кэш** | ускорить повторные запросы | **Redis** |
| **Очереди задач** | фоновые задачи | **Celery** + Redis/RabbitMQ |
| **Веб-сервер** | принимает запросы | **Uvicorn** (ASGI), Gunicorn |
| **Реверс-прокси** | балансировка, SSL | **Nginx** |

## Часть 6. Базы данных и хранилища 🔥

### 6.1. Реляционные (SQL) ⭐

| СУБД | Когда |
|---|---|
| **PostgreSQL** | самая мощная open-source, стандарт индустрии, JSON + расширения |
| **MySQL / MariaDB** | популярная, простая, веб |
| **SQLite** ⭐ | встроенная, в одном файле, без сервера — отлично для прототипов/FINPILOT |
| **MS SQL Server** | корпоративный, Microsoft |
| **Oracle** | тяжёлый enterprise, банки |

### 6.2. NoSQL

| Тип | СУБД | Когда |
|---|---|---|
| **Документная** | **MongoDB** | гибкая схема, JSON-документы |
| **Key-Value** | **Redis** | кэш, сессии, очереди — очень быстрая (в памяти) |
| **Колоночная** | **Cassandra**, **ClickHouse** | огромные объёмы, аналитика (ClickHouse — топ для логов/метрик) |
| **Графовая** | **Neo4j** | соцсети, связи (кто кого знает) |
| **Поисковая** | **Elasticsearch** | полнотекстовый поиск, логи |
| **Time-series** | **InfluxDB**, **TimescaleDB** | метрики, IoT, данные по времени |
| **Векторная** | **pgvector**, **Pinecone**, **Qdrant** | поиск по эмбеддингам для AI/LLM (актуально сейчас) |

### 6.3. Аналитические хранилища (DWH)

**Snowflake**, **BigQuery** (Google), **Redshift** (AWS) — для аналитики на терабайтах, отделены от продакшн-БД.

> Правило: транзакционная нагрузка (OLTP) → PostgreSQL/MySQL; аналитика на больших данных (OLAP) → ClickHouse/Snowflake/BigQuery.

## Часть 7. DevOps и инфраструктура 🔥

Как код доезжает до прода и там живёт.

| Категория | Инструмент | Зачем |
|---|---|---|
| **Контейнеризация** ⭐ | **Docker** | упаковать приложение со всем окружением — «у меня работает» больше не отмазка |
| **Оркестрация** | **Kubernetes (k8s)** | управление сотнями контейнеров: масштабирование, самовосстановление |
| **CI/CD** ⭐🔥 | **GitHub Actions**, GitLab CI, Jenkins | автосборка, тесты, деплой |
| **IaC** (инфра как код) | **Terraform**, **Ansible** | описываешь инфраструктуру кодом, не кликаешь мышкой |
| **Облака** | **AWS**, **GCP**, **Azure** | арендуешь серверы/сервисы |
| **Мониторинг** | **Prometheus** + **Grafana** | собирать метрики и рисовать дашборды |
| **Логи** | **ELK** (Elasticsearch + Logstash + Kibana), Loki | централизованный сбор логов |
| **Очереди сообщений** | **Kafka**, **RabbitMQ** | асинхронная связь между сервисами |

> Для тебя ближайший must-have — **Docker + GitHub Actions**. Это закрывает «12-factor parity» и делает портфолио серьёзнее. Kubernetes — потом, когда будет много сервисов.

### 8.3. Data Engineering

| Инструмент | Зачем |
|---|---|
| **Apache Spark** | обработка больших данных распределённо (на Scala, есть PySpark) |

> Твой ETL-опыт + Airflow (он на Python) — естественное расширение в data engineering, если когда-нибудь захочешь свернуть туда.

<a name="часть-11"></a>
## Часть 11. Как системы общаются (API и протоколы) 🔥

Очень важная тема для backend-собеса.

| Подход | Суть | Когда |
|---|---|---|
| **REST** ⭐🔥 | ресурсы + HTTP-методы (GET/POST/PUT/DELETE), JSON | стандарт для веб-API, твой FastAPI |
| **GraphQL** | один эндпоинт, клиент сам запрашивает нужные поля | когда клиентам нужны разные срезы данных, экономия запросов |
| **gRPC** | бинарный протокол на HTTP/2 + Protobuf, очень быстрый | связь между микросервисами |
| **WebSocket** | двусторонний постоянный канал | чаты, realtime, уведомления |
| **Webhooks** | сервер сам стучится к тебе при событии | интеграции (оплата прошла → колбэк) |
| **Очереди (Kafka/RabbitMQ)** | асинхронная связь через сообщения | развязать сервисы, фоновые задачи |

### REST — что знать на собес 🔥

- **Методы:** GET (читать), POST (создать), PUT (заменить целиком), PATCH (частично обновить), DELETE (удалить).
- **Коды:** 2xx (успех: 200 OK, 201 Created), 3xx (редирект), 4xx (ошибка клиента: 400, 401, 403, 404), 5xx (ошибка сервера: 500, 503).
- **Идемпотентность:** GET/PUT/DELETE можно повторять без последствий, POST — нет.
- **Stateless:** сервер не хранит состояние клиента между запросами.

<a name="часть-12"></a>
## Часть 12. Архитектурные стили 🔥

| Стиль | Суть | Плюс / минус |
|---|---|---|
| **Монолит** ⭐ | всё приложение — единое целое | + просто разрабатывать и деплоить / − тяжело масштабировать по частям |
| **Микросервисы** 🔥 | приложение = набор мелких независимых сервисов | + независимое масштабирование и деплой / − сложность, сеть, наблюдаемость |
| **Serverless** | пишешь функции, инфру держит облако (AWS Lambda) | + не думаешь о серверах / − vendor lock-in, холодный старт |
| **Event-driven** | сервисы общаются через события/очереди | + развязка, масштаб / − сложнее отлаживать |

> Совет, который дают все senior: **начинай с монолита**. Микросервисы — решение проблемы масштаба, которой у соло-проекта/стартапа ещё нет. FINPILOT как монолит — абсолютно правильно.

<a name="часть-14"></a>
## Часть 14. Как выбирать стек

Когда стартуешь проект, иди по вопросам:

1. **Что за продукт?** Веб-сервис / мобилка / data-пайплайн / системная утилита — это сразу режет варианты.
2. **Что я уже знаю?** Для соло-проекта/портфолио бери знакомый стек — скорость важнее «модности».
3. **Какая нагрузка реально ожидается?** Не проектируй микросервисы под 10 пользователей. KISS/YAGNI.
4. **Есть ли экосистема/комьюнити?** Популярный инструмент = ответы на вопросы + библиотеки + найм.
5. **Целостность данных критична?** Да → реляционная БД (PostgreSQL). Нет/гибкость → подумать про NoSQL.

**Типовой «дефолтный» fullstack-стек сегодня:**
- Frontend: React + TypeScript + Vite + Tailwind
- Backend: Python (FastAPI) ⭐ или Node (NestJS) или Go
- БД: PostgreSQL (+ Redis для кэша)
- Инфра: Docker + GitHub Actions, деплой в облако

<a name="часть-15"></a>
## Часть 15. Где ты сейчас и что учить рядом ⭐

Твоё ядро — **Python backend (FastAPI + SQLAlchemy + SQLite/PostgreSQL)**. Это сильная, востребованная и правильная база. Вокруг неё по приоритету:

1. **SQL глубже** — не только через ORM, а голый SQL: JOIN, индексы, EXPLAIN, оконные функции. Бэкендера без SQL не берут.
2. **PostgreSQL** — перерасти SQLite, это рабочая лошадка индустрии.
3. **Docker** — заверни FINPILOT в контейнер. Закрывает кучу вопросов на собесе.
4. **GitHub Actions (CI)** — автотесты на каждый push. Быстрая победа для портфолио.
5. **Redis** — базовый кэш, легко добавить, хорошо звучит.
6. **REST глубоко + чуть GraphQL/gRPC на уровне «знаю что это»** — см. Часть 11.
7. **Linux/Bash на бытовом уровне** — деплой и DevOps без этого не идут.
8. **TypeScript по верхам** (опционально) — чтобы понимать фронт и быть fullstack-дружелюбным.

**Чего НЕ распыляться** на старте: Kubernetes, микросервисы, Rust, функциональные языки — это интересно, но не для первой работы. Глубина в одном стеке > поверхностность в десяти. Твой Python + крепкие SQL/Docker/CI закроют большинство junior-вакансий.

Принцип прежний: каждый пункт — маленькая задача до результата, потом следующий. Не марафон, а Kaizen.
