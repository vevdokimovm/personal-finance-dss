# Шаблон: отчёт юзабилити/UX-исследования

> **Назначение.** Превращает сырой лист наблюдений (заметки после интервью/теста, скрин переписки,
> голосовая расшифровка) в структурированный отчёт единого формата. Один тест = один отчёт по этому
> шаблону + строка в `knowledge/survey_auditory/usability/usability_testing_registry.md`.
>
> **Как пользоваться (Claude: сырьё → отчёт).** Дай Claude сырые заметки и скажи «сделай отчёт по
> шаблону юзабилити». Правила трансформации — ниже; их держаться строго, ничего не выдумывать.

---

## Правила трансформации (сырьё → отчёт)

1. **Не выдумывать.** В отчёт попадает только то, что есть в сырье. Если данных нет (возраст, длительность) —
   пишем «н/д», а не догадку. Прямые формулировки респондента сохраняем как цитаты.
2. **Одно наблюдение = одна строка.** Разбить поток заметок на атомарные наблюдения. Дубли объединить,
   указав что повторялось (повтор = сильный сигнал).
3. **Классифицировать каждое** по типу: `навигация` · `понимание/ментальная модель` · `терминология` ·
   `баг` · `визуал/тема` · `доверие/объяснимость` · `фича-запрос` · `позитив`.
4. **Проставить severity** (влияние на пользователя, не на код):
   - `P0` — блокер: пользователь не может выполнить ключевую задачу.
   - `P1` — серьёзно: выполняет, но с ошибками/фрустрацией/недоверием к результату.
   - `P2` — заметно: мешает, но обходится.
   - `P3` — косметика: не мешает выполнению.
5. **Баги — отдельным блоком** (для трекера): шаги воспроизведения из сырья, если описаны.
6. **Связать с продуктом**: какие пункты `docs/ROADMAP.md`, требования (`SEC-`/`FR-`), ценности
   (`knowledge/business/product_philosophy.md`) затрагивает находка. Это превращает жалобу в задачу.
7. **Версия под тестом важна.** Помечать, если тестировалась устаревшая версия (напр. v2.0.2) — часть
   находок может быть уже неактуальна, но UX-уроки обычно вневременны.
