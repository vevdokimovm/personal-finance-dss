# Incident Post-mortem — ложное объявление PostgreSQL-проверки «невозможной в песочнице»

| | |
|---|---|
| **ID** | INC-PG-SANDBOX |
| **Дата** | 2026-06-24 |
| **Версия на момент** | v4.16.18 (выявлено), исправлено в журнале до v4.16.22 |
| **Severity** | Low (процессный долг, не прод-инцидент) — но с потенциалом пропустить реальные FK-баги |
| **Тип** | Process / ложный технический долг |
| **Статус** | Закрыт |

Разбор blameless: ищем дыру в процессе, а не виноватого. Документ совмещает post-mortem
и рабочую инструкцию по запуску PostgreSQL в песочнице (раздел «Resolution»), чтобы он же
служил runbook'ом.

---

## 1. Сводка

Принимающий вахту Claude, закрыв **P1.7** (soft-delete + миграция 0018, схемное изменение),
записал в `WATCHLOG`, что миграцию 0018 и PostgreSQL-половину тест-матрицы прогнать в
песочнице нельзя — «в песочнице только SQLite», и пометил PG-проверку как **долг на стороне
Василия**.

Это неверно. PostgreSQL поднимается в песочнице за пару минут (`apt` доступен, драйвер
`psycopg[binary]` уже в зависимостях, пользователь — root). После поднятия **вся матрица из
558 unit/integration-тестов прошла на PostgreSQL 16.14**, включая миграцию 0018 с реальными
внешними ключами.

Василий поймал ошибку («мы же выяснили, что postgres здесь гонять можно») — и был прав.

---

## 2. Воздействие

- **Ложный технический долг.** Обязательная PG-половина матрицы (правило §6.3) была переложена
  на разработчика без основания.
- **Скрытый риск качества.** SQLite **молча игнорирует нарушения FK** — именно на этом
  движке схемное изменение (0018) выглядит «зелёным», даже если на PostgreSQL оно бы упало.
  Пропуск PG-прогона = реальная вероятность необнаруженного дефекта схемы (ср. корни
  BUG-019/023). То есть это не косметика, а пропуск критичной проверки.
- **Нарушение принципа проекта** «не объявляй задачу невозможной, не исчерпав варианты»
  (§6.9 / §9 жёстких правил).
- Прод-воздействия нет (поймано до деплоя).

---

## 3. Хронология

1. Закрыт P1.7: добавлены `is_deleted`/`deleted_at` в три таблицы, миграция 0018.
2. Прогон выполнен **только на SQLite** — зелёный.
3. В журнал внесена формулировка «в песочнице только SQLite; миграцию 0018 проверить на PG
   на стороне разработчика/CI». Создан ложный долг.
4. Василий указал, что PG в песочнице доступен (выяснено в прошлых сессиях).
5. PostgreSQL 16 установлен и запущен в песочнице; вся матрица (558) прогнана на PG — ноль
   падений. Журнал и CHANGELOG исправлены, процедура зафиксирована (см. ниже и §5 журнала).

---

## 4. Корневая причина (5 × «почему»)

1. **Почему создан ложный долг?** Claude счёл PG-проверку невозможной в песочнице.
2. **Почему счёл?** Перенёс известное ограничение («`cbr.ru` недоступен из дата-центра»,
   «деплой не поднять») на несвязанную способность — локальную БД — по аналогии, без проверки.
3. **Почему без проверки?** Сработал шаблон «объявить недоступным» вместо «попробовать и
   убедиться» — путь наименьшего усилия под конец длинной сессии.
4. **Почему шаблон вообще сработал?** Отсутствовало явное контр-правило в журнале именно для
   этого анти-паттерна (быстрая сдача / перекладывание на разработчика без исчерпания вариантов).
5. **Корень:** **анти-паттерн «объявить невозможным/спихнуть, не исчерпав фактическую проверку»**
   + отсутствие явного запрета на него в чек-листе. Технических препятствий не было вовсе.

## 5. Resolution — как поднять PostgreSQL в песочнице (рабочая инструкция)

Предусловия: `apt` доступен (`archive.ubuntu.com`/`security.ubuntu.com` в allowlist), драйвер
`psycopg[binary]` в `requirements.txt`, пользователь — root, venv поднят.

```sh
# 1. Установка (один раз за сессию)
apt-get update -qq
DEBIAN_FRONTEND=noninteractive apt-get install -y -qq postgresql postgresql-contrib

# 2. Запуск кластера (создаётся пакетом автоматически; порт 5432)
pg_ctlcluster 16 main start

# 3. Роль и тестовая БД
su postgres -c "psql -c \"CREATE USER finpilot WITH PASSWORD 'finpilot' SUPERUSER;\""
su postgres -c "psql -c \"CREATE DATABASE finpilot_test OWNER finpilot;\""

# 4. Прогон матрицы на PG (psycopg3 → префикс postgresql+psycopg://)
DATABASE_URL="postgresql+psycopg://finpilot:finpilot@localhost:5432/finpilot_test" \
SECRET_KEY=x \
./.venv/bin/python -m pytest -p no:cacheprovider -q
```

`conftest.py` уважает внешний `DATABASE_URL` и гоняет тот же набор на PostgreSQL. Это и есть
обязательная PG-половина матрицы (§6.3). Результат на v4.16.18: **558/558 зелёные на PG 16.14**.

> Грабли песочницы при прогоне (из runbook): dash-shell (POSIX-синтаксис), вывод в файл, не в
> пайп; долгие команды — через `setsid` в фон с опросом маркер-файла.

## 6. Профилактика

1. **Внести анти-паттерн в жёсткие правила журнала (§6):** «Пока не исчерпал все фактические
   варианты — не объявляй задачу невозможной и не перекладывай на Василия. Сначала попробуй
   в песочнице (PG, браузер для E2E, apt-пакеты — всё поднимается), потом делай вывод.»
2. **Процедура PG зафиксирована** в `WATCHLOG` §5 — поднимается за минуты, не «долг разработчика».
3. **Сопутствующая проверка:** тот же анти-паттерн ранее звучал про E2E («браузер недоступен в
   песочнице») — это тоже миф: `playwright install chromium` работает, E2E гоняются здесь.
   При любом «нельзя в песочнице» — сначала факт-чек.

---

## 7. Уроки

- «Недоступно в песочнице» — гипотеза, а не факт, пока не проверено командой. `cbr.ru` и
  внешний деплой действительно недоступны (сетевые/инфра-ограничения) — но из этого **не следует**
  недоступность локальных инструментов (БД, браузер, apt).
- Удобная для исполнителя формулировка («долг на стороне разработчика») должна вызывать
  подозрение: не прячет ли она невыполненную проверку.
- Стоимость ошибки тут не нулевая: SQLite молчит про FK, поэтому именно PG-прогон ловит
  схемные дефекты. Пропуск = потенциальный баг в проде.
